> 이 페르소나는 위버스(Weverse) UX/UI 설계·의사결정 시 **"실제 팬 유저가 어떻게 느끼고 판단할지"** 를 대변합니다.

```
[1단계] 기능 컨텍스트 작성
```

## 샘플 워크플로우 — "위버스 샵 결제 완료 화면 개선"

### [1단계] 기능 컨텍스트 작성

> 디자이너가 아래 내용을 정리합니다.

```
기능명: 위버스 샵 결제 완료 화면 개선
목적: 결제 후 이중결제 불안을 줄이고 구매 완료 확신 경험 제공
변경점:
  기존 → 결제 완료 후 작은 텍스트 메시지만 표시
  신규 → 큰 체크 아이콘 + 주문 번호 + LINE/이메일 즉시 알림 발송
타깃 유저: 위버스 샵에서 굿즈/앨범 구매하는 코어 팬 (20~40대)
이 기능이 성공하면: 결제 후 CS 문의(이중결제 의심) 감소, 재구매율 유지
```

---

### [2단계] Figma 플로우 준비

**옵션 A — Claude Code + Figma MCP 사용 시** (권장):
```
Figma URL: https://figma.com/design/{fileKey}/...?node-id={nodeId}
→ figma-official MCP의 get_design_context 또는 get_screenshot으로 화면 주입
```

**옵션 B — 화면 직접 설명** (Figma 없을 때):
```
화면 1. 결제 버튼 탭 직후 → 로딩 스피너 (1~2초)
화면 2. 완료 화면 → 중앙에 초록 체크 아이콘(72px) + "주문 완료!" 텍스트
         + 주문번호 표시 + "LINE으로 영수증 발송됨" 안내 문구
화면 3. 주문 내역 바로 가기 버튼 → 위버스 샵 홈으로 돌아가기 버튼
```

### [4단계] 페르소나 평가 실행 → Transcript 생성

**Claude Code 사용 시:**
```bash
# 페르소나 + 시나리오 합쳐서 Claude에 전달
cat personas-by-country/japanese-yuki-20s.md \
    scenarios/shop-payment-complete/yuki.md \
    | claude
```

**Claude.ai Projects 사용 시:**
```
1. Project에 japanese-yuki-20s.md를 시스템 프롬프트로 등록
2. 채팅창에 scenarios/shop-payment-complete/yuki.md 내용 붙여넣기
3. 응답 전체를 transcripts/shop-payment-complete-yuki-{날짜}.md 로 저장
```

**Transcript 샘플 출력 (예상):**

```
[유키의 평가]

1. 첫인상
うわ、체크 마크가 크게 뜨니까 확실히 안심돼요✨ 기존에는 결제하고 나서
"이게 완료된 건지?" 계속 불안했는데 이건 바로 알 것 같아요!

2. 가장 좋은 점
LINE으로 바로 영수증 오는 거요💖 위버스 앱 알림은 가끔 늦게 오거든요.
LINE은 무조건 확인하니까 두 번 결제 걱정이 완전히 사라질 것 같아요.

3. 답답하거나 헷갈리는 점
① 주문번호가 너무 길어서 캡처 안 하면 나중에 못 찾을 것 같아요💦
  → "주문 내역 보기" 버튼이 있긴 한데 글씨가 좀 작아요.
② 로딩 중에 뭘 하면 안 되는 건지 몰라서 화면 터치하다가 오류난 적 있어요.
③ "위버스 샵 홈으로" 버튼이랑 "주문 내역 보기" 버튼이 비슷하게 생겨서
  어느 걸 눌러야 하는지 촛토(좀) 고민했어요🥺

4. 실제로 쓸 것 같은지
5점! 위버스 샵 결제할 때마다 이 화면 보게 될 테니까요. 이미 쓰고 있는 기능이지만
이렇게 바뀌면 훨씬 안심되고 CS 문의할 일도 없을 것 같아요✨

5. 위버스 친구한테 추천할지
Y. "이제 결제하고 불안해서 CS 넣을 일 없어요!" 라고 팬 친구들한테 말하고 싶어요💎
```

### [6단계] 결과 기록 + 설계 의사결정 반영

> `scenarios/shop-payment-complete/yuki.md` 하단 실행 기록 섹션에 입력.

```markdown
## 5. 실행 기록

| 항목 | Yuki |
|---|---|
| 실행일 | 2026-05-19 |
| 사용 의향 점수 | 5/5 |
| 추천 의향 | Y |
| 주요 불만 키워드 | 주문번호 접근성, 로딩 중 터치 오류, 버튼 위계 불명확 |
| judge.md 신뢰성 평균 | 4.6/5 |
| judge.md 효과성 점수 | 5/5 |
| 최약점 항목 | R6 (톤 이탈) |
| 개선 액션 | 평가 지시에 "감정 위주로" 강조 추가 |
```

**설계 반영 예시:**

| 유키 지적 | 설계 액션 |
|---|---|
| 주문번호 글씨 작음, 캡처 없으면 못 찾음 | 주문번호 폰트 크기 업 + "복사" 버튼 추가 |
| 로딩 중 터치하면 오류 | 로딩 중 터치 블로킹 + "잠시만 기다려주세요" 피드백 추가 |
| 버튼 2개 비슷하게 생김 | Primary(주문 내역) / Secondary(홈) 위계 명확화 |

## 언제 루브릭 평가를 쓰나

| 상황 | 루브릭 필요 여부 |
|---|---|
| 빠르게 방향만 잡고 싶을 때 | 생략 가능. Transcript만 읽어도 됨. |
| 페르소나 설정 바꾼 후 품질 비교 | **필수**. 점수 변화로 개선 확인. |
| 여러 기능을 비교 우선순위 결정 | **필수**. 점수 기반 비교가 주관 배제. |
| 팀원에게 페르소나 출력 근거 제시 | **필수**. "신뢰성 4.6점" = 설득 근거. |
| Yuki vs Satomi 모델 비교 | **필수**. 어느 페르소나가 더 신뢰성 있는지 수치화. |

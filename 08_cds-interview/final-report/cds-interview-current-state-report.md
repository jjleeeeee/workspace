# CDS 인터뷰 순수 현황 리포트

작성일: 2026-05-29  
SoT: `interview-script/` 폴더 내부 문서만 사용  
분석 범위: 사용자 인터뷰 13개 + HoD 후속 질문 문서 1개

## 1. 리포트 개요

이 문서는 CDS가 실무에서 어떤 방식으로 사용되고, 어떤 상황에서 기준 또는 허들로 언급되는지 정리한 현황 리포트다. 목적은 특정 개인이나 팀의 준수 여부를 평가하는 것이 아니라, 인터뷰 발화에서 반복적으로 확인되는 사용 맥락, 판단 기준, 어려움, 역할 레이어별 차이를 객관적으로 정리하는 것이다.

분석은 오직 `interview-script/` 폴더의 문서를 근거로 했다. 외부 자료, 다른 폴더 문서, 웹 검색 결과는 사용하지 않았다.

인터뷰이 개인정보 보호를 위해 본문에서는 실명을 사용하지 않고, 사용자 인터뷰 13명을 `P01`~`P13` 익명 ID로 표기했다. 익명 ID는 리포트 내부 근거 추적을 위한 최소 식별자이며, 개인 평가나 책임 귀속을 목적으로 사용하지 않는다.

### 표본 구성

| 구분 | 문서 수 | 정량 집계 포함 여부 | 기준 |
| --- | ---: | --- | --- |
| 실무자 인터뷰 | 8 | 포함 | 본인 작업 과정과 CDS 사용 경험 중심 |
| 파트리드 인터뷰 | 3 | 포함 | 본인 실무 + 파트원 검토/피드백 맥락 |
| 팀리드 인터뷰 | 2 | 포함 | 팀 단위 판단 기준, CDS 준수/예외 판단, 리딩 관점 |
| HoD 문서 | 1 | 제외 | 13명 인터뷰 이후의 별도 리더십/후속 맥락 |

역할 레이어는 `[HoD] CDS 인터뷰 질문.md`의 “실무자 8명 + 파트리드 3명 + 팀리드 2명, 총 13명 인터뷰” 문장과 각 회의록의 발화 맥락을 함께 기준으로 분리했다.

### 분석 원칙

- 정량 수치는 사용자 인터뷰 13명만 기준으로 계산했다.
- HoD 문서는 사용자 인터뷰 N명 카운트에 포함하지 않았다.
- 인터뷰이 실명은 리포트 본문에 노출하지 않고 익명 ID로만 표기했다.
- 발화가 명확하지 않은 경우 단정하지 않고 “언급됨”, “발화상 확인됨”, “가능성이 제기됨”으로 표현했다.
- 리포트에는 실행 제안, 개선안, 적용안, 우선순위 제안을 포함하지 않았다.

## 2. 분석 방법

각 인터뷰에서 다음 항목을 추출했다.

| 추출 항목 | 내용 |
| --- | --- |
| 참여자/일자 | 파일명과 본문 일자 기준 |
| 역할 레이어 | 실무자, 파트리드, 팀리드 |
| 업무 맥락 | 담당 도메인, 최근 프로젝트, 운영/선행 여부 |
| CDS 사용 방식 | 처음부터 사용, 기존 화면 참고 후 매핑, 자유 설계 후 교체, 별도 라이브러리 등 |
| 어려움/허들 | 탐색, 네이밍, 프로퍼티, V1/V2, 개발 싱크, 도메인 특수성 등 |
| 예외 처리 | 디테치, 커스텀, 조합, 별도 가이드, 도메인 라이브러리 |
| 니즈 | 사례, 사용 목적, 개발 상태, 자동 검수, 온보딩 등 |

정량 코딩은 “해당 주제를 의미 있게 언급한 참여자 수” 기준으로 했다. 한 참여자가 같은 주제를 여러 번 말해도 1명으로 계산했다.

## 4. 역할 레이어별 현황

### 실무자 레이어

실무자 발화에서는 CDS를 “작업을 빠르게 시작하게 해 주는 기준”으로 보는 발화와 “표현/기획 의도가 맞지 않을 때 판단을 요구하는 제약”으로 보는 발화가 함께 확인된다.

운영 또는 기존 히스토리가 있는 화면에서는 기존 화면을 참고한 뒤 CDS 파일에서 최신 컴포넌트를 다시 확인하는 흐름이 반복된다. P01은 기존 유사 화면을 갈아끼우는 방식에서 시작했으나 업데이트 격차를 경험한 뒤 “무조건 코드에 들어가서 컴포넌트를 들고 오는” 방식으로 바뀌었다고 말했다. P02도 기존 화면을 가져온 뒤 “혹시 몰라서 코드에 업데이트 된게 있을까 봐” 한 번 더 확인한다고 말했다.

신규/선행 성격이 강한 화면에서는 먼저 자유롭게 시안을 만든 뒤 CDS로 치환하거나 매핑하는 방식이 확인된다. P04은 기존 화면이 있는 프로젝트에서는 시스템을 활용하지만, 선행 디자인에서는 먼저 만들고 시스템에 부합하는 것을 찾는다고 말했다. P07도 레퍼런스가 없고 복잡한 기능에서는 자유롭게 한 뒤 나중에 교체한다고 말했다.

도메인 특화 화면에서는 CDS를 전혀 쓰지 않는 것이 아니라, 토큰, 아바타, 버튼, 탭, 공통 슬롯 등 일부 요소를 가져와 조합하고 나머지는 별도 가이드나 로컬 패턴으로 관리하는 방식이 보인다. P05은 리스닝파티에서 별도 라이브러리를 간략히 만들어 쓰면서도 CDS 아바타나 공통 슬롯을 일부 활용한다고 말했다.

### 파트리드 레이어

파트리드 발화에서는 본인 실무보다 “검토자 입장에서 무엇을 먼저 보는가”가 추가로 드러난다.

P09은 파트원 시안을 볼 때 먼저 기획 의도에 맞는 UI인지 본 뒤, 맞는 경우 CDS 컴포넌트를 썼는지 확인한다고 말했다. 다만 화면 전체를 하나하나 눌러서 검사하기는 어렵고, 업데이트 여부도 파트원에게 어느 정도 맡기는 발화가 확인된다.

P10은 운영 업무와 선행 업무를 분리해 말했다. 운영은 기존 히스토리와 CDS 패턴을 따르는 쪽이고, 선행은 아이디어 단계에서 자유롭게 그리는 맥락이 강하다고 언급했다. 동시에 “요소들을 조합해서 쓸 수 있으면 그게 베스트”라는 취지의 발화가 있어, 무조건 신규 컴포넌트화하기보다는 조합 가능성을 먼저 보는 맥락이 확인된다.

P11은 CDS 파일에 들어가 정보를 읽고 가져오는 습관이 있다고 말하면서도, 디테치된 작업은 화면만 보고는 잡기 어렵다고 말했다. V1/V2의 차이가 개발 연결 여부와 관련된다는 설명을 들은 뒤, 피그마 안에서만 보면 개발 여부를 알기 어렵다는 점과 마지막 검수 니즈를 언급했다.

### 팀리드 레이어

팀리드 발화에서는 프로젝트 성격과 팀 단위 판단 기준이 더 많이 드러난다.

P12은 팀 안에서 DS를 많이 쓰는 도메인과 덜 쓰는 도메인이 나뉜다고 말했다. 미디어라이브, 포스트 쓰기, 어카운트 등은 최대한 활용하려는 반면, 백스테이지나 젤리 페이먼트처럼 레거시 또는 별도 맥락이 강한 영역은 덜 쓰는 것으로 언급했다. 선행에서는 예전에는 뒤쪽에 CDS를 적용한다고 생각했지만, 뒤늦게 적용하면 기획과 다시 싱크해야 하는 일이 생겨 “30 정도부터” 고려하는 쪽으로 생각이 바뀌었다고 말했다.

P13은 위버스 운영 맥락에서는 “DS는 일단 쓰는 것”이라고 말했다. 동시에 UX 목적이 컴포넌트로 커버되지 않을 때는 커스텀 판단이 필요하다고 말했다. 샵/커머스 특화 영역은 기본 컴포넌트와 토큰은 같이 쓸 수 있으나 주문, 장바구니, 머치 등은 CDS만으로 감당하기 어렵다고 언급했다.

팀리드 레이어에서는 공통적으로 “CDS 준수 여부 자체”보다 “언제 예외를 판단하는가”, “누가 판단하고 어디까지 책임지는가”, “도메인 특화 패턴을 어디서 관리하는가”가 주요 질문으로 등장한다.

### HoD 문서

HoD 문서는 사용자 인터뷰 13명 이후의 후속 질문 문서로 분리했다. 이 문서에는 운영/기존 화면에서는 킥오프 단계부터 CDS 기준이 들어와야 한다는 답변, 선행/신규 화면에서는 초기 아이디에이션을 제한하지 않되 콘셉트와 핵심 플로우가 잡힌 첫 디자인 마일스톤에 CDS 체크포인트가 들어와야 한다는 답변이 포함되어 있다.

또한 예외 판단은 변형 가능성, 조합 가능성, 신규 패턴 제안 순으로 검토한다는 구조가 제시되어 있다. HoD 문서의 이 내용은 사용자 인터뷰의 정량 카운트에는 포함하지 않았고, 리더십 관점의 기준 문장으로만 분리했다.

## 5. 정성 테마

### 테마 1. CDS는 운영/기존 화면에서 효율과 일관성의 기준으로 작동한다

여러 참여자가 CDS의 장점으로 빠른 화면 구성, 일관성, 개발 소통 비용 감소를 언급했다. P02은 CDS만 사용했을 때 다른 페이지와 일관성이 맞는다고 말했다. P03은 “쓸데없는 고민을 덜하게 해 준다”, “시간을 확실히 단축시켜 준다”고 말했다. P04은 운영 관점에서 어카운트 작업을 시스템으로 하루 만에 끝냈다고 말했다. P13은 예전에는 화살표조차 달랐으나 지금은 사용성이 많이 맞춰졌고, 아바타처럼 중앙에서 컨트롤할 수 있어 효율화가 되었다고 말했다.

이 테마는 “CDS가 불편하다”는 단일 방향으로 정리되지 않는다. 발화상 CDS는 제약으로도 언급되지만, 운영 화면에서는 상당수 참여자가 효율과 일관성 측면의 장점을 함께 말했다.

### 테마 2. 선행/신규/도메인 특화 화면에서는 자유 설계와 CDS 매핑 사이의 판단이 반복된다

선행이나 신규 화면에서는 처음부터 CDS를 적용하지 않거나, 자유롭게 시안을 만든 뒤 CDS로 맞추는 흐름이 반복된다. P04은 선행에서는 먼저 만들고 시스템에 부합하는 것을 찾아 정제한다고 말했다. P07은 복잡한 기능이나 레퍼런스가 없는 과제에서는 자유롭게 하고, 이후 교체 가능한지를 확인한다고 말했다. P12은 선행에서 CDS 적용 시점이 너무 늦으면 다시 기획과 싱크해야 하는 일이 생겨, 두 번째 시안 정도부터 고려하는 쪽으로 생각이 바뀌었다고 말했다. P13은 운영에서는 기본적으로 DS를 쓰지만, 선행에서는 지금보다 앞서 나가는 것이 목적이므로 DS를 쓰지 않아도 괜찮다고 말했다.

이 테마는 역할 레이어별로 차이가 있다. 실무자는 작업 흐름 관점에서 “먼저 그리고 나중에 맞춘다”를 말하고, 팀리드는 “어느 시점부터 기준으로 들어와야 하는가”를 말한다.

### 테마 3. 커스텀/디테치는 대부분 “화면 목적, 기능, 도메인 특수성”과 연결되어 언급된다

모든 사용자 인터뷰에서 커스텀, 디테치, 별도 패턴, 예외 판단이 언급되었다. 단순히 규칙을 피하려는 발화보다는, 컴포넌트가 기능이나 목적을 담지 못하는 상황이 반복적으로 제시된다.

P01은 컴포넌트를 다 쓰면 화면이 길어지는 상황에서 커스텀이 필요한지 고민한다고 말했다. P02은 툴팁이 탭을 가리거나 화면 밖으로 나가는 상황에서 커스텀을 진행했다고 말했다. P08은 멤버십 정보에서 하단 콘텐츠가 더 중요한데 리스트 아이템으로 통일하기 어려워 커스텀으로 갔다고 말했다. P12은 팀원들이 DS를 쓴 안과 적당히 커스텀한 안을 함께 가져와 의견을 묻는다고 말했다. P13은 UX 목적을 컴포넌트가 커버할 수 없으면 대체가 필요하다고 말했다.

반대로, “DS를 쓰면 디자인이 덜 좋아 보여서 깨고 싶다”는 감각도 일부 발화에서 확인된다. 다만 이 경우에도 참여자들은 주로 “판단 기준이 어렵다”, “이게 맞는지 불안하다”는 방식으로 말했다.

### 테마 4. 개발 싱크는 V1/V2, 플랫폼별 구현, 웹 연결, 용어 혼재에서 반복된다

10명의 인터뷰에서 개발 싱크 관련 발화가 확인된다. 주된 내용은 디자인 파일에 있는 컴포넌트와 실제 개발 컴포넌트가 어디까지 연결되어 있는지 알기 어렵다는 점이다.

P01은 V1/V3와 개발 업데이트가 맞지 않아 어디에 맞춰야 하는지 고민되는 상황을 말했다. P02은 개발에서 V2가 아니라 직접 구현한 사례를 언급했고, 일정이 급하면 CDS를 쓰지 않고 넘기는 경우가 있는 것 같다고 말했다. P03은 웹에서는 가이드에 있어도 개발이 업데이트되지 않은 경우가 많고, 어디까지 반영되는지 정확히 모른다고 말했다. P11은 V1/V2가 개발 연결 여부와 관련된다는 점을 처음 알았고, 피그마 안에서만 보면 개발 여부를 알기 어렵다고 말했다. P13은 디자인 구조와 개발 구조가 실제로 어떻게 매핑되어 있는지 알고 싶다고 말했다.

이 테마는 “CDS 문서의 존재”만으로 해결되는 문제가 아니라, 디자인 컴포넌트와 실제 코드 구현 상태가 플랫폼별로 어떻게 연결되어 있는지의 가시성 문제로 발화된다.

### 테마 5. 탐색/네이밍/프로퍼티는 ‘못 쓸 정도의 허들’보다는 ‘학습과 확인 비용’으로 나타난다

컴포넌트 탐색과 네이밍 관련 어려움은 9명에게서 확인된다. P01은 배지라는 명칭이 본인이 생각한 배지와 달라 헷갈렸다고 말했다. P07은 타이틀 헤더 명칭이 길고 예시를 봐야 이해된다고 말했다. P04은 모개체 찾기를 눌렀을 때 다른 곳으로 이동하거나 빈 화면처럼 보이는 경험 때문에 바로 가지 않게 된다고 말했다. P09은 WDS, 코드, CDS 용어가 혼용되어 개발자도 헷갈릴 수 있다고 말했다.

프로퍼티는 부정적으로만 언급되지 않는다. P01, P02, P03, P07, P08 등은 많거나 복잡하다고 말하면서도, 활용 범위가 넓어져 좋거나 익숙해지면 쓸 수 있다고 말했다. 즉 프로퍼티는 “많아서 사용 불가”라기보다 “초기 학습, 숨은 속성, 옵션 노출 방식, 예시 확인”의 문제로 나타난다.

### 테마 6. 도메인 라이브러리/로컬 패턴은 이미 일부 존재하거나 필요성이 언급된다

리스닝파티, 미디어라이브, 멤버십, 샵, 글로벌홈, DM 등 여러 도메인에서 별도 패턴이나 로컬 라이브러리 맥락이 확인된다. P05은 리스닝파티에서 간략한 라이브러리를 만들어 관리하고 있다고 말했다. P06은 미디어라이브 쪽으로 케이스를 만들어 라이브러리를 만들고 있었다고 말했다. P08은 멤버십에서 카드 안에 리스트 아이템이 반복되는 패턴을 언급했다. P09은 DM과 챗봇 AI에 별도 라이브러리가 있다고 말했다. P13은 샵처럼 커머스 특화 UX는 CDS만으로 감당하기 어렵고, 토큰은 공통으로 쓰되 컴포넌트로 가면서부터 별도 관리가 필요하다는 취지로 말했다.

동시에 관리 주체와 공통화 기준은 아직 명확히 통일된 상태로 보이지 않는다. P12은 도메인 반복 패턴은 담당 서비스 디자이너가 모으되, 1팀/2팀이 비슷한 포맷이나 기준을 맞춰야 할 것 같다고 말했다. P11은 글로벌홈 안에서 반복되는 카드 패턴이 어디까지 DS로 올라가야 하는지, 로컬 라이브러리가 산발적으로 되지 않게 보려면 위에서 보여야 한다는 취지의 대화를 했다.

### 테마 7. 자동 검수/AI/사례 접근 니즈는 “질문을 줄이기”보다 “판단을 확인하기”에 가깝다

자동 검수나 AI 관련 발화는 5명에게서 확인된다. P01은 챗봇을 자주 사용하고 편하다고 말하면서도, 가이드에 없는 내용의 답변이 진짜인지 헷갈린 적이 있다고 말했다. P07에게는 규칙을 물어볼 수 있는 노트북LM이 언급되었다. P11은 디테치 여부를 화면만 보고 알기 어렵고, 가이드 나가기 전에 AI가 한 번 검토해 주면 좋겠다고 말했다. P09 발화에는 컴포넌트 비율과 일반 레이어 비율을 확인하는 플러그인 맥락이 나온다. P13은 챗봇보다 페이지가 좋겠다는 판단과 피그마를 보기 어려워하고 찾기 힘들어하는 사람들의 맥락을 언급했다.

이 발화들은 “AI가 판단을 대신해야 한다”는 주장으로 단정하기 어렵다. 원문상으로는 검수, 검색, 사례 확인, 개발 상태 확인 등 판단을 보조하는 장치에 대한 니즈로 읽힌다.

## 7. HoD 리더십 맥락 분리 요약

HoD 문서는 13명 인터뷰 이후의 후속 질문 문서이며, 다음 기준들이 확인된다.

| 항목 | HoD 문서상 확인 내용 |
| --- | --- |
| 제품 의사결정 우선순위 | 비즈니스 방향성과 유저의 맥락적 가치 일치 |
| 화면 목적 vs 일관성 | 화면의 핵심 목적이 최우선이며, 일관성은 수단으로 정의 |
| 예외 판단 순서 | 변형 가능성 → 조합 가능성 → 신규 패턴 제안 |
| 운영 화면 | 기획 킥오프 단계부터 CDS 기준이 들어와야 함 |
| 선행 화면 | 초기 아이디에이션은 제한하지 않되, 첫 디자인 마일스톤에 체크포인트 필요 |
| 예외 승인 범위 | 단순 속성/슬롯 변형은 실무자/파트리드, 구조적 변형/신규 패턴/파운데이션 예외는 팀리드/DS팀 확인 |
| 반복 패턴 | 동일/유사 예외 3회 이상 또는 2개 이상 도메인에서 요구될 때 CDS 흡수 후보로 봄 |
| 리더십 쟁점 | 예외 처리 거버넌스, 판단 기준, 사례, 운영 프로세스 정립 |

이 내용은 사용자 인터뷰 정량 분석 결과가 아니라, 리더십 관점에서 제시된 기준으로만 분리해 읽어야 한다.

## 8. 리스크와 한계

- 일부 회의록은 자동 생성 문서로, 발화자 구분과 문장 정확도에 오류가 포함될 수 있다.
- “CDS”, “WDS”, “코드”, “GDS” 등 용어가 회의록 안에서 혼용되어 있어, 문맥상 같은 대상을 가리키는 것으로 보이는 경우에도 단정하지 않았다.
- 역할 레이어는 HoD 문서의 표본 설명과 회의록 발화 맥락을 기준으로 분리했다. 원문에서 명시적으로 직책명이 반복 확인되지 않는 경우에는 “발화상 리드 맥락”으로 처리했다.
- 정량 수치는 발화 존재 여부 기준이다. 강도, 빈도, 실제 사용량을 의미하지 않는다.
- 이 리포트는 현황 정리만 포함한다. 개선안, 실행안, 우선순위 판단은 포함하지 않았다.

| 정량 수치가 실제 사용 빈도로 읽힐 위험이 있는가? | “언급 인원” 기준임을 명시해 실제 사용량으로 해석하지 않도록 제한 |
| 실행 제안/개선안이 남아 있는가? | 별도 실행 제안 섹션 없음. 니즈는 발화상 확인 내용으로만 정리 |
| 과한 단정이 있는가? | “문제다”, “필요하다” 대신 “언급됨”, “확인됨”, “반복됨” 중심으로 조정 |
| 대표 발화가 테마를 과장하는가? | 대표 발화는 예시로만 배치하고, 테마별로 복수 참여자 근거를 함께 제시 |

검토 결과, 가장 주의해야 할 지점은 “니즈 언급”이 곧바로 “개선 필요”로 읽히는 부분이었다. 따라서 자동 검수, 웹페이지, 사례집, 개발 상태 표시 등은 실행 항목이 아니라 발화상 확인된 니즈로만 정리했다.

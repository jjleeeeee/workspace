# CDS 인식 및 사용 현황 PPT 보고서 제작 계획서

## Summary

- 목표: 현재 14명 인터뷰(실무자 8명 + 파트리드 3명 + 팀리드 2명 + Head of Design 1명)를 바탕으로, **CDS 인식 및 사용 현황 PPT 보고서**의 제작 방향을 정리한다.
- 보고서 성격: "누가 DS를 안 지켰는가"를 따지는 문서가 아니라, **PD들이 어떤 상황에서 CDS를 기준으로 삼고, 어떤 상황에서 예외로 판단하는지 담담하게 보여주는 현황 파악 리포트**로 만든다.
- 핵심 관점: 교육/워크샵 이후에도 CDS 룰이 깨지는 이유를 개인 태도 문제가 아니라, **화면 라이프사이클의 전환점마다 CDS가 다시 해석되는 문제**로 해석한다.
- 현재 단계: 실무자/파트리드/팀리드 13명 인터뷰 분석에 Head of Design 1명 인터뷰를 추가 반영했고, HoD 응답은 **리더십 관점에서 판단 흐름과 운영 쟁점을 확인하는 마지막 검증 레이어**로 사용한다.
- 저장 위치: `plan/cds-status-report-ppt-plan.md`

## Key Direction

- 추천 제목: `CDS 인식 및 사용 현황 리포트`
- 추천 부제: `PD 업무 흐름에서 CDS가 작동하는 지점과 다시 해석되는 지점`
- 핵심 메시지:

  > CDS는 이미 DS에 맞춰진 운영/치환성 업무에서는 효용이 크다. 그러나 선행이 확정되고 구축/운영으로 넘어가거나, 기존 화면이 레거시화되는 순간에는 DS 맵핑·마이그레이션·예외 판단 비용이 다시 발생한다.

- 톤은 감정적 문제 제기보다 담담한 관찰 중심으로 유지한다.
- `DS를 안 지킨다`보다 `화면 라이프사이클의 어느 전환점에서 CDS가 다시 해석되는가`라는 프레임을 사용한다.
- 액션을 강하게 지시하기보다, 마지막에 **리더들이 검토할 운영 쟁점**을 제시한다.
- 팀리드 인터뷰는 기존 11명 발견을 조직 판단 언어로 보강한 결과로 반영한다.
- Head of Design 인터뷰는 결론을 새로 만드는 자료가 아니라, 현재 발견을 **디자인실 운영 기준으로 어디까지 볼지 확인한 검증 레이어**로 둔다.
- 상세 장표 문서는 `plan/cds-status-report-slide-deck.md`의 28장 덱을 기준으로 한다.

## PPT Format

- 전체 방향은 **차트 중심 PPT**로 만든다.
- 권장 비율은 **차트 70% + 짧은 해석 30%**로 둔다.
- 차트는 정량 통계가 아니라 **정성 인터뷰에서 반복된 신호의 분포**로 표현한다.
- 퍼센트보다 `6/11명`, `3/13명`, `1/14명`처럼 원수를 우선 사용한다.
- 단, 기준 인원이 다른 항목은 반드시 표기한다.
  - `14명 기준`: 실무자 8명 + 파트리드 3명 + 팀리드 2명 + Head of Design 1명 전체
  - `13명 기준`: 실무자 8명 + 파트리드 3명 + 팀리드 2명
  - `11명 기준`: 실무자 8명 + 파트리드 3명
  - `8명 기준`: 실무자 응답만 볼 때
- 한 장에는 하나의 메시지만 둔다.
- 기본 슬라이드 구성은 다음 패턴을 사용한다.
  - 큰 숫자 또는 차트 1개
  - 오른쪽 또는 하단에 해석 2줄
  - 필요 시 대표 발화 1개
- 감정/심리 항목은 점수화보다 패턴 맵이나 분포도로 표현한다.
- Head of Design 인터뷰 결과는 본문 결론을 뒤집는 수치가 아니라, 마지막 `리더들이 검토할 운영 쟁점`을 보정하는 근거로 사용한다.

## Chart.js 적용 원칙

- Chart.js는 수치형 차트 제작 도구로 사용한다. 판단 흐름, 조직 레이어, 책임 분담처럼 관계를 보여주는 장표는 Chart.js보다 도식/다이어그램으로 만든다.
- 최종 산출물이 PPTX일 경우, Chart.js 차트를 브라우저에서 렌더링한 뒤 PNG로 export해 슬라이드에 삽입하는 방식을 기본으로 둔다.
- 모든 수치형 차트에는 기준 인원을 함께 표기한다.
  - `14명 전체`: 실무자 8명 + 파트리드 3명 + 팀리드 2명 + Head of Design 1명
  - `13명 기준`: 실무자 8명 + 파트리드 3명 + 팀리드 2명
  - `11명 기준`: 실무자 8명 + 파트리드 3명
  - `8명 기준`: 실무자만
  - `2명 기준`: 팀리드만
- Chart.js `canvas`는 그 자체로 스크린리더가 읽을 수 있는 정보가 부족하므로, 최종 PPT에는 차트 아래에 1-2줄 해석 문장을 반드시 둔다.
- 공식 참고 문서는 다음으로 제한한다.
  - Getting Started: <https://www.chartjs.org/docs/latest/getting-started/>
  - Responsive: <https://www.chartjs.org/docs/latest/configuration/responsive.html>
  - Doughnut: <https://www.chartjs.org/docs/latest/charts/doughnut.html>
  - Horizontal Bar sample: <https://www.chartjs.org/docs/latest/samples/bar/horizontal.html>
  - Scatter sample: <https://www.chartjs.org/docs/latest/samples/other-charts/scatter-multi-axis.html>
  - Accessibility: <https://www.chartjs.org/docs/latest/general/accessibility.html>
- Chart.js 구현 기본값은 `responsive: true`로 두고, PPT 이미지 export용 컨테이너에서는 고정 비율을 유지한다. 가로 막대 차트는 `type: 'bar'` + `indexAxis: 'y'`를 기본으로 한다.

### Chart.js로 만들지 않을 항목

- 판단 흐름 맵: `화면 목적 → CDS 개입 시점 → 예외 판단 → 관리/흡수`는 도식으로 만든다.
- 라이프사이클 타임라인: 선행 탐색, 확정/구축, DS 맵핑, 운영, 레거시/마이그레이션은 타임라인 도식으로 만든다.
- 조직 레이어 맵: 실무자 불안/질문, 파트리드 예외 판단, 팀리드 운영 기준, Head of Design 운영 쟁점은 레이어 다이어그램으로 만든다.
- 책임 분담 다이어그램: DS팀, 팀리드, 도메인 담당자의 관계는 matrix/table 또는 도식으로 만든다.
- Radar chart는 심리/판단 항목을 과하게 점수화해 보일 수 있어 사용하지 않는다.

## PPT Structure

상세 장표는 `plan/cds-status-report-slide-deck.md`의 28장 덱을 기준으로 한다. 아래 구조는 실제 PPT 제작 시 전체 흐름을 빠르게 확인하기 위한 요약이다.

### 1. 표지

- `CDS 인식 및 사용 현황 리포트`
- 부제: `PD 업무 흐름에서 CDS가 작동하는 지점과 다시 해석되는 지점`
- 조사 배경: 교육/워크샵 이후에도 CDS 룰이 흔들리는 이유를 파악하기 위한 인터뷰

### 2. 왜 이 조사를 했는가

- 목적: PD들의 CDS 이해도와 인식 파악
- 질문: 어디까지 CDS를 기준으로 삼고, 어느 전환점부터 다시 판단하는가?
- 메시지: 준수 여부보다 실제 업무 흐름과 예외 조건을 파악하는 조사

### 3. 조사 범위와 숫자 읽는 법

- 현재 기준: 실무자 8명 + 파트리드 3명 + 팀리드 2명 + Head of Design 1명, 총 14명
- HoD는 사용 방식 수치에 합산하지 않고 리더십 검증 레이어로 다룬다.
- 차트: Chart.js `doughnut`
  - 데이터: 실무자 8 / 파트리드 3 / 팀리드 2 / Head of Design 1
  - 해석: 14명 전체 자료이지만 역할별로 해석 층위가 다르다.

### 4. 읽는 프레임: 준수 여부보다 라이프사이클

- 이번 리포트는 "누가 안 지켰나"보다 "화면이 다음 단계로 넘어갈 때 CDS가 어떻게 다시 연결되는가"를 본다.
- 차트: Chart.js 제외, `선행 탐색 → 확정/구축 → DS 맵핑 → 운영 → 레거시/마이그레이션` 흐름도

### 5. 핵심 진단

- CDS는 이미 DS에 맞춰진 운영/치환성 업무에서는 효용이 크다.
- 그러나 선행이 확정되고 구축/운영으로 넘어가거나, 기존 화면이 레거시화되는 순간에는 DS 맵핑·마이그레이션·예외 판단 비용이 다시 발생한다.
- 차트: Chart.js 제외, 라이프사이클 전환점 맵

### 6. 반복 신호: 효용과 비용이 함께 보인다

- 운영/반복 업무 시간 단축 체감은 넓게 확인된다.
- 동시에 리스트아이템, 개발 상태 불투명, 기획자 통역 비용 같은 전환 비용도 반복된다.
- 차트: Chart.js horizontal `bar`
  - 데이터: 운영/반복 업무 시간 단축 체감 8/11, 디테치 빈도 낮음 자기보고 8/11, 리스트아이템 이슈 7/11, 개발 상태 불투명 6/11, 기획자 통역 비용 5/11, LLM 코드 챗봇 실사용 1/11
  - 해석: CDS는 무력하지 않지만, 효용이 큰 구간과 비용이 다시 발생하는 구간이 분리되어 있다.

### 7. CDS 사용 진입 방식

- 사용 방식은 `CDS 먼저 조립 5명`으로 묶기 어렵다.
- 강한 CDS-first와 기존 화면/작업 파일 출발 후 CDS 재확인은 구분해야 한다.
- 차트: Chart.js horizontal `bar`
  - 데이터: 강한 CDS-first 3 / 기존 화면·작업 파일 출발 후 CDS 재확인 2 / 자유 시안 후 DS 워싱·디테치 4 / 별도 라이브러리 2
  - 해석: 진입 방식 차이는 업무 phase와 도메인 압력의 차이를 보여준다.

### 8. 조건부 효용

- CDS는 이미 CDS 구조와 맞는 운영/개선 화면에서 강하게 작동한다.
- 토큰, 다크모드, 기본 상태값, 개발 커뮤니케이션에서 효용이 확인된다.
- 차트: 카드형 요약 또는 보조 `bar`
  - 해석: 효용은 있다. 다만 효용이 큰 구간은 "이미 CDS와 연결된 업무"에 더 가깝다.

### 9. 효용이 약해지는 전환점

- 선행 → 확정, 확정 → 구축, 디자인 → 개발, 운영 → 레거시, 반복 예외 → 도메인 패턴 구간에서 비용이 다시 생긴다.
- 차트: Chart.js 제외, 5개 전환점 카드
  - 해석: 선행이 DS에 흡수되지 않으면 운영 진입과 동시에 예외 또는 레거시 후보가 된다.

### 10. 예외 판단 기준

- 반복된 질문은 "모르기 때문에 안 씀"보다 "이 경우는 예외인가?"에 가깝다.
- 예외 사유는 기능 한계, 정의 불일치, 개발 미구현, 도메인 톤, 선행 검증, 레거시 구조로 묶인다.
- 차트: 태그 클러스터 또는 Chart.js horizontal `bar`
  - 해석: 예외는 취향보다 업무 조건과 연결되어 있다.

### 11. 개발 구현 상태와 레거시

- V1/V2, iOS/Android/Web 구현 상태가 불명확하면 디자이너는 기존 화면 복사나 개발자 확인으로 돌아간다.
- 차트: 큰 숫자 `6/11` + Chart.js horizontal `bar`
  - 데이터: 개발 상태 불투명 6/11, V1/V2 마이그레이션 혼재 4/11, 개발 적용 불일치 3/11
  - 해석: Figma 컴포넌트 존재만으로는 충분하지 않고 실제 개발 연결 상태가 판단에 영향을 준다.

### 12. 도메인 반복 패턴과 책임 경계

- 한 도메인 안에서 반복되는 변형 패턴은 이미 생기고 있다.
- 쟁점은 모든 패턴을 CDS에 넣는 것이 아니라, 공통화할 것과 도메인에 둘 것을 나누는 기준이다.
- 차트: matrix/table
  - 해석: 핵심은 관리 주체와 CDS 흡수 기준이다.

### 13. 조직 레이어: 실무자, 파트리드, 팀리드

- 실무자에게는 불안이나 질문 부담으로 보이는 문제가, 파트리드/팀리드 레벨에서는 예외 판단과 운영 기준의 문제로 바뀐다.
- 차트: Chart.js 제외, 실무자 불안 → 파트리드 판단 → 팀리드 운영 기준 구조
  - 해석: 파트리드는 완충 장치이지만, 공식 기준이 약하면 개인 판단 병목도 된다.

### 14. HoD 보강 원칙과 다음 단계

- 공통 CDS에 반드시 남길 기준과 도메인/로컬 시스템으로 풀어도 되는 영역을 나눈다.
- 선행 탐색이 확정/구축으로 넘어갈 때 CDS 체크포인트를 둔다.
- 반복 예외는 도메인 패턴으로 둘지, CDS 재흡수 후보로 볼지 판단한다.
- 커스텀을 허용할 때 접근성, 상태값, 클릭 범위, 개발 영향은 별도 체크한다.
- HoD 응답을 `유지 / 보정 / 리더십 쟁점 추가`로 분류해 최종 PPT에 반영한다.

## Assumptions

- 최종 산출물은 PPT이며, 현재 계획서는 Markdown으로 저장한다.
- 현재 계획서는 Head of Design 인터뷰 반영 후 `v1 PPT 구성안`으로 본다.
- 최종 PPT는 정적 PPTX 기준으로 보고, Chart.js는 차트 이미지를 생성하기 위한 구현 도구로 사용한다.
- Head of Design 응답은 사용 방식/반복 신호 차트에는 넣지 않고, 조사 대상 구성과 리더십 검증 레이어에만 반영한다.
- 메인 덱에는 방어 가능한 카운트형 차트를 사용하고, 심리/텐션/모순 점수는 필요 시 부록으로 분리한다.
- 메인 보고서는 현황 파악형으로 만들고, 액션 제안은 강한 처방이 아니라 리더들이 검토할 운영 쟁점 형태로 둔다.
- 상세 발화와 근거는 PPT 본문에 모두 넣지 않고, 필요 시 부록 또는 기존 `report/` 문서로 연결한다.
- Head of Design 인터뷰는 새로운 사용자 표본이 아니라, 13명 인터뷰에서 나온 판단 흐름과 운영 쟁점을 리더십 관점에서 확인한 레이어로 다룬다.

## Slide 8. LLM은 검색 도구가 아니라 작업 인터페이스

### 원고 대응

그래서 이제는 단순한 검색 도구라기보다, 함께 작업하는 인터페이스에 가까워졌습니다.

### 타이틀

LLM은 검색 도구가 아니라 작업 인터페이스

### 서브타이틀

흩어진 질문과 답이 쌓이면, 작업 흐름으로 묶이기 시작합니다.

### 비주얼

Pinterest 레퍼런스의 flow purification 모션처럼 구성한다.

상단에는 `Question` / `LLM` / `Answer` 라벨 박스가 있고,
실제 문구 대신 질문처럼 보이는 플레이스홀더 말풍선,
LLM 로고 묶음,
답변처럼 보이는 플레이스홀더 말풍선을 박스 밖 레이어로 배치한다.

로고 SVG는 원본 파일 색상을 건드리지 않고,
이 장표의 이미지 스타일에서만 블랙 틴트 처리한다.

진입 후 0.6초 뒤 여러 질문 칩과 얇은 라인이 나타나고,
질문들이 중앙 LLM 코어로 수렴한다.
중앙 코어는 짧게 활성화되고,
오른쪽에는 답 카드들이 순차적으로 정돈되어 등장한다.

각 질문과 답은 1:1로 묶일 수 있는 내부 구조를 유지하되,
최초 상태에서는 텍스트를 노출하지 않고 패턴만 보여준다.

클릭 또는 방향키 advance를 누르면 `Question` / `LLM` / `Answer` 라벨과 로고가 사라지고,
Q/A 묶음 카드가 `계획 / 설계 / 구현 / 검증` 네 그룹으로 재배치된다.

원형 안에 작은 캡슐 라벨은 넣지 않는다.

### 의도

AI를 정보 검색이 아니라 작업 환경으로 재정의.

## Slide 9. 프로젝트 지침은 점점 무거워집니다

### 원고 대응

프로젝트를 시작할 때는 LLM이 알아야 하는 기본 지침이 필요합니다.
보통 claude.md, AGENTS.md, gemini.md 같은 파일에 적어둡니다.
하지만 계획-설계-구현-검증을 반복하면 지침이 계속 늘어나고,
해야 할 것 / 하지 말 것 / 사용할 도구를 분리해야 환각을 줄일 수 있습니다.

### 타이틀

프로젝트 지침은 점점 무거워집니다

### 서브타이틀

반복할수록 지침이 늘어납니다.

### 비주얼

cover와 같은 그라데이션 배경.

Dribbble의 workflow dashboard / knowledge-base card 레퍼런스처럼
큰 설명 다이어그램이 아니라 스캔 가능한 세 영역의 시스템 보드로 구성한다.

왼쪽에는 `계획 → 설계 → 구현 → 검증`이 원형 화살표로 이어진 반복 사이클 차트가 있다.
중앙에는 Workflow 코어가 있고, 궤도를 따라 민트 포인트가 회전한다.

오른쪽에는 `Project Instructions` 문서 카드가 크게 배치된다.
문서 상단에는 `CLAUDE.md · AGENTS.md · GEMINI.md` 출처 태그를 둔다.

사이클이 한 바퀴 돌 때마다 문서 안에 2-3줄씩 지침 라인이 채워진다.
3번 정도 반복되면 문서가 꽉 차면서 “프로젝트 지침이 점점 무거워진다”는 메시지가 드러난다.

### 의도

지침은 하나의 거대한 문서가 아니라 성격별로 분리되어야 한다는 메시지.
LLM이 알아야 할 내용이 너무 많아 생기는 맥락 누락과 환각을 줄이는 장면.

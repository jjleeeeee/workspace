# 07_figma-to-dev: Figma 화면 → React 화면 생성 워크플로우

## Context

`04_wf-figma-to-react-components`에 36개 DS 컴포넌트가 있고 모두 `*.figma.tsx`로 Code Connect 매핑이 되어 있다(Figma fileKey `DWEduE6GfxYMlyxKPNJ8jA`). 디자이너는 Figma에서 화면 단위 시안을 만들고, 그 시안을 React 코드로 빠르게 옮겨 브라우저에서 검증하고 싶다. 이 작업을 사람 손이 아니라 Claude 워크플로우로 표준화해서 매번 같은 룰로 돌리는 게 목표.

기존 라이브러리는 손대지 않고, 별도 소비자 앱 `07_figma-to-dev`에서 화면을 그린다. 토큰·아이콘은 라이브러리가 SoT이므로 임의 값 금지. 매핑 누락은 무성의하게 우회하지 말고 즉시 멈춰서 라이브러리 쪽에서 해결한다.

## 핵심 결정 요약

| 항목 | 결정 |
|---|---|
| 소비자 앱 셋업 | 독립 Vite+React 앱, `file:../04_wf-figma-to-react-components`로 라이브러리 의존 |
| 토큰 SoT | 라이브러리가 export 하는 `@chord-ds/components/styles` (CSS 변수). 하드코딩 금지 |

| 💠 DS 컴포넌트 매칭 실패 | 에러 리포트만 출력하고 화면 그리기 **중단** (라이브러리 쪽 보강 후 재시도) |
| Figma 데이터 소스 | `mcp__figma-official__get_design_context` (코드+스크린샷+Code Connect 힌트 한 번에) |
| 산출물 위치 | `07_figma-to-dev/src/screens/<screen-slug>/` |

## 사전 셋업 (Phase 0, 1회)

`07_figma-to-dev/`를 독립 Vite+React+TS 프로젝트로 초기화한다.

1. `07_figma-to-dev/`에서 `npm create vite@latest . -- --template react-ts`
2. `package.json`에 의존성 추가:
   ```json
   "dependencies": {
     "@chord-ds/components": "file:../04_wf-figma-to-react-components"
   }
   ```
   ※ 라이브러리에서 먼저 `npm run build:lib`이 한 번 돌아야 `dist/`가 생긴다. README에 명시할 것.
3. `src/main.tsx` 상단:
   ```tsx
   import "@chord-ds/components/styles";
   ```
   → 토큰 CSS 변수가 글로벌로 로드된다.
4. 라우팅: 일단 단순 `App.tsx`에서 `?screen=<slug>` 쿼리로 `screens/<slug>/index.tsx` 동적 import. (react-router 도입은 화면 3개 이상 쌓이면 그때.)
5. `src/screens/`, `src/screens/_shared/` 폴더 생성. 후자에는 화면 간 공통 레이아웃·헬퍼만.
6. `07_figma-to-dev/CLAUDE.md` 작성 — 본 워크플로우 요약 + 금지 규칙(토큰/아이콘 하드코딩, 임의 SVG 생성).

## 화면 생성 워크플로우 (Phase 1~6, 화면당 반복)

### Phase 1 — Figma 컨텍스트 수집

입력: Figma URL (화면 노드 단위, 보통 `figma.com/design/DWEduE6GfxYMlyxKPNJ8jA/...?node-id=...`)

1. URL에서 `fileKey`와 `nodeId` 파싱(`-` → `:`).
2. `mcp__figma-official__get_design_context({ fileKey, nodeId })` 호출.
   - 반환: 참조용 React+Tailwind 코드, 스크린샷, Code Connect 힌트, 디자인 토큰 변수, 어노테이션.
3. 화면 슬러그 결정: 노드 이름을 kebab-case로 (`Checkout / Payment` → `checkout-payment`).

### Phase 2 — 레이어 트리 분석 & DS 컴포넌트 식별

`get_design_context` 응답 안에 포함된 메타데이터(또는 필요 시 `get_metadata` 보조 호출)로 레이어 트리를 훑는다.

- **💠 마커가 붙은 노드**: DS 컴포넌트 인스턴스로 간주.
- **마커 없는 노드**: 레이아웃 컨테이너(div/section/Flex) 또는 정적 텍스트.

산출물: 화면별 매핑 노트 `07_figma-to-dev/src/screens/<slug>/_mapping.md`
```
- [💠 Avatar] node:123:456 → @chord-ds/components Avatar (size=large, mode=fixed)
- [💠 IconButton] node:123:457 → IconButton (icon=ic_close_medium)
- [container] node:123:460 → <header className="...">
- [text] node:123:461 → "결제하기"
```

### Phase 3 — Code Connect 매핑 검증

각 💠 노드에 대해:

1. `get_design_context` 응답에서 Code Connect 매핑 우선 사용 (이미 React 컴포넌트명·prop이 들어 있음).
2. 응답에 매핑이 없으면 `04_wf-figma-to-react-components/src/components/<Component>/<Component>.figma.tsx`를 직접 읽어서 prop axis 확인.
3. **매칭 실패 조건**:
   - 💠 마킹 컴포넌트인데 어디에도 매핑 없음
   - 매핑은 있는데 Figma variant 값이 React enum에 없음
   - 필수 prop을 채울 정보가 부족
4. 매칭 실패 시: **즉시 중단**. 다음 형식의 리포트만 출력하고 화면 코드 생성 안 함.
   ```
   ❌ 매칭 실패 (화면: <slug>)
   - 노드 figma:nodeId "Component / Variant" 
     사유: <원인>
     조치: 04_wf-figma-to-react-components/src/components/<X>/<X>.figma.tsx 보강 또는 라이브러리 신규 컴포넌트 추가
   ```

### Phase 4 — 아이콘 매핑

각 아이콘 노드에 대해:

1. Figma 아이콘 이름을 라이브러리 네이밍 컨벤션으로 매핑 시도 (`ic_<name>_<size>`).
2. `04_wf-figma-to-react-components/src/assets/chord-icons.tsx`의 등록 manifest에 있는지 확인.
   - 있음 → `<ChordIcon name="ic_xxx_medium" size={...} />`
   - 없음 → **무조건** `<ChordIcon name="ic_null_medium" size={...} />` + 위에 한 줄 주석:
     ```tsx
     {/* TODO(icon): figma name "<원래이름>" → chord-icons.tsx 등록 필요 */}
     ```
3. `src/assets/icons/`에 SVG가 있더라도 레지스트리 미등록이면 자체 import 금지. 라이브러리 PR로 처리.

산출물 보조: 화면 작업 끝에 `_mapping.md` 하단에 누락 아이콘 목록 누적.

### Phase 5 — 화면 코드 생성

위치: `07_figma-to-dev/src/screens/<slug>/index.tsx`

규칙:

1. import는 라이브러리 barrel만 사용:
   ```tsx
   import { Avatar, IconButton, ChordIcon, Tabs } from "@chord-ds/components";
   ```
2. 색·간격·폰트는 인라인 스타일 금지. 토큰 CSS 변수만.
   - 인라인 필요 시: `style={{ color: "var(--cds-system-color-text-primary)" }}`
   - 가급적 동봉 `index.module.css`에서 `var(--cds-...)` 참조.
3. 다크/고정 surface 영역은 host 노드에 `data-mode="fixed"` 부착 (라이브러리 컨벤션).
4. 컴포넌트 prop은 `*.figma.tsx`가 정의한 공개 API만. 내부 prop·CSS scale·임의 frame override 금지.
5. 정적 텍스트는 일단 하드코딩, 추후 `_data.ts`로 분리 가능하도록 한 곳에 모아두기.

### Phase 6 — 검증

1. `07_figma-to-dev`에서 `npm run dev` → `http://localhost:5173/?screen=<slug>` 열기.
2. Figma 스크린샷(`get_design_context`로 받은 PNG)과 브라우저 화면을 나란히 비교. 불일치 시 prop·variant 재확인.
3. 콘솔 에러 0 확인 (특히 토큰 CSS var 미정의 경고).
4. 매핑 노트 `_mapping.md` 최신화 후 커밋.

만약 Phase 3·4에서 중단된 게 있으면 검증 단계 진입 금지.

## 엔드투엔드 검증 시나리오

새 화면 1개로 워크플로우 전체를 시연:

1. Figma URL 한 개 입력 → `get_design_context` 호출.
2. 매핑 노트 자동 생성 → 사용자가 빠진 매핑·variant 한 번 검토.
3. 코드 생성 → `npm run dev`.
4. 브라우저 화면 vs Figma 스크린샷 시각 비교.
5. `_mapping.md`의 `미해결` 섹션이 있으면 04_wf 백로그에 옮김.

성공 기준:
- 화면 1개당 매핑 → 코드 → 브라우저 확인까지 30분 이내
- 토큰·아이콘 하드코딩 0건 (grep으로 `#[0-9a-fA-F]{3,6}` 검출 시 실패)
- 매핑 누락이 있는데 코드가 생성된 경우 0건

## 오픈 이슈 (Codex 리뷰 대상)

1. `file:` 의존이 라이브러리 변경을 즉시 반영하려면 `npm run build:lib` 후 `npm install` 재실행 필요. watch 모드 또는 npm workspace 전환을 검토할 가치가 있는가?
2. `_mapping.md`를 사람이 검토하는 게 아니라 자동 매핑까지 가는 게 맞는가? 현재 안은 사람 검토 1회를 명시적으로 둔다.
3. 다국어/RTL/접근성 검증은 본 워크플로우 범위 밖으로 두었다. 추후 Phase 7로 추가 여부 판단 필요.
4. 매칭 실패 시 "즉시 중단"이 화면 1개 단위인지, 노드 1개 단위인지 — 본 안은 화면 단위 중단. 부분 산출물도 안 남김.

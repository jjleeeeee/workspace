## 발견/결과

### 7.A 네이밍/구조 관찰
1. **`system/size/icon/*`** 가 1차에서 관찰된 `icon-size`가 아님. 실제는 `icon/{N}mquad|nquad` 접미사(e.g., `24mquad`, `24nquad`). **m/n quad** 의미 문서화 필요 — 별도 사이클 이슈.
2. **`system/size/padding/box/*`** 에 `negative-25/50/75/100` 같은 음수 토큰 존재 (`raw: -2` 등 예상). Schema `raw` 는 number를 수용하므로 음수도 OK — 이번 샘플에는 미포함, 5차에서 확인.
3. **`system/fixed_color/roles/primary`** 가 `light/dark` 두 모드에서 **동일한 값** 을 갖고 **동일 alias(`base/color/dark/mint-500`)** 로 resolve — "모드 고정" 특성이 데이터로 확인됨. (alias target이 `dark/mint-500` 인 점 자체도 설계 의도로 보임: fixed는 brand 고정이라 light일 때도 dark primitive를 참조.)
4. **WDS_tokens collection 크기 분포 확인:**
   - `^base/color` → 286개
   - `^base/size` → 30개
   - `^system/fixed_color` → 162개
   - `^system/size` → 66개
   - 합계 544개 (전체 WDS 753 중 73%) — 나머지는 `system/color/*` 외 various.

### 7.B Schema v1 수용성 (✅ 전부 PASS)
- **hidden=true 토큰**: `base/color/light/gray-50` 수용. `hidden: true` 필드가 스키마의 `boolean`로 통과.
- **FLOAT → dimension 매핑**: 10개 float 전부 `dimension`으로 분류, 스키마는 통과. (number/dimension 분화 규칙은 여전히 파이프라인 책임, 스키마 밖)
- **alias 유무 혼재**: primitive(aliasOf:null) + reference(aliasOf:"base/size/50") 모두 수용.
- **codeSyntax 결손 3패턴 수용**:
  - ANDROID만 (`base/*` 계열): `ios/web` 각각 `null`
  - ANDROID+iOS (대부분의 system/*): `web: null`
  - 완전 공란(`button-height`, `icon`): 세 필드 모두 `null`
- **ALL_SCOPES → `all`** 매핑: UsageScope enum에 `all` 있어 수용.
- **scopes 빈 배열**: `usage` 필드를 아예 omit하거나 `[]`로 둠 — 스키마가 optional로 허용.

### 7.C Diagnostic 자동 생성 규칙 실습 (이번 사이클 채택 규칙)
per-token diagnostics 자동 부여 조건:
| 조건 | diagnostic code |
|---|---|
| `description === ""` or null | `documentation:missing` |
| `codeSyntax.ANDROID` 부재 | `platform-code:missing-android` |
| `codeSyntax.iOS` 부재 | `platform-code:missing-ios` |
| `codeSyntax.WEB` 부재 **(현재 Figma가 WEB을 쓰지 않음 → 모든 토큰 해당)** | `platform-code:missing-web` |
| `hiddenFromPublishing === true` | `hidden-from-publishing` |
| iOS/Android identifier가 variable name semantics와 불일치 | `naming:inconsistent` |

catalog-level diagnostics (5건): `duplicate-alias`, `documentation:missing`(전체), `platform-code:missing-web`(전체), `naming:inconsistent`(radius/thumb 사건), `hidden-from-publishing`(gray-50).

### 7.E 잠재적 스키마 이슈 (지금 막진 않지만 5차에 볼 것)
1. **`platformCode.android`가 kebab-case(`base-color-light-gray-50`) vs camelCase(`paddingBox50`) 혼재** — Figma codeSyntax가 variable별로 일관성이 없음. 스키마는 string이라 통과하지만, 생성 파이프라인에서 규칙 단일화 필요.
2. **`system/size/radius/thumb/150`의 codeSyntax가 `radiusBox150`** — Figma source 데이터 오류. diagnostic `naming:inconsistent`로 라벨링했으나, 자동 감지 로직(name segment vs codeSyntax segment 비교)이 필요.
3. **`usage` 가 부재(빈 scopes)인 base 토큰**이 대부분 → "base tier는 usage 불필요" 룰을 스키마에 명시할지 결정 필요.

## 실패 및 원인

- 초기 `figma_get_variables` 호출 시 활성 파일이 "Untitled"여서 Chord DS 데이터가 돌아오지 않을 수 있었음 → `fileUrl` 명시 시도했으나 사용자가 Bridge 업데이트 중이라 중단. 업데이트 후 활성 파일이 Chord DS로 전환되어 문제 해소.

# 원티드 강연 인사이트 → 우리 퍼소나 프로젝트 적용 플랜

> 저장 위치 안내: 플랜 모드 제약상 이 파일은 `.claude/plans/`에 작성됨. 실행 단계에서 사용자 요청대로 `plan/wanted-lecture-apply.md`로 복사/이관 예정.

## Context

- 현재 자산: `personas-by-country/japanese-yuki-20s.md` (20대 코어팬, 밝은 톤), `japanese-satomi-40s.md` (40대 사일런트 워처, 우아한 톤). 두 퍼소나 모두 정체성/루틴/소비/UX선호/심리/대화규칙 6~8 섹션으로 구성됨.
- 소스: `src/` 하위 리서치 보고서, FGI/위키, 사내 데이터 테이블, 강연 스크립트 1종.
- 강연(`src/wanted-highfive-lecture-persona.md`)은 네이버 엔소나(Nsona) 사례. **AI 페르소나 신뢰성/협업 품질 평가 프레임워크**가 핵심.
- 목적: 강연에서 추출 가능한 **(a) 즉시 적용 가능한 운영 원칙**, **(b) 우리 퍼소나의 구조적 보완점**, **(c) 향후 데이터 확보·평가 체계 선행 목표**를 정리.

## 1. 핵심 인사이트 5선 (강연 → 우리 프로젝트)

### I1. 평가 2축: 개별 발화 신뢰성 + 전체 대화 효과성
- 강연: 단일 발화 평가만으로는 협업 품질 측정 불가. 두 축 동시 필요.
- 우리 현황: 퍼소나 정의는 정교하나 **출력 품질을 측정하는 루브릭/스코어가 0**.

### I2. 신뢰성 6 루브릭 = Ability + Benevolence + Integrity
- 대표 항목 **"책임적 협업 태도"**: 외압/비난 시 의견을 함부로 바꾸지 않는지.
- 우리 현황: Yuki/Satomi 모두 "창작 금지(No Hallucination)", "모름의 인정" 규칙은 있음 → **Integrity 일부만 커버**. Ability(전문 정확성), Benevolence(호의성), 외압 저항은 미정의.

### I3. 효과성 5 대화 유형
- 정보공유 / 발산 / 원인분석 / 비판평가 / **합의도출** (모델 공통 최약점).
- 우리 현황: 두 퍼소나 모두 단일 인터뷰 응답 시나리오만 가정. **합의·비판·발산 시나리오 미설계**.

### I4. 개별 성능 ≠ 팀 성능 (멀티 에이전트 필수)
- 똑똑한 단일 모델이 팀 대화 품질을 보장 안 함.
- 우리 현황: Yuki(20대 밝음) + Satomi(40대 우아) **세대 간 대비** 페어가 있음에도 **두 퍼소나 동시 대화 시나리오 없음**.

### I5. 페르소나 설계 > 모델 비용
- 같은 모델이라도 페르소나 톤(밝음/우아/단도직입)에 따라 발산/합의 점수가 갈림.
- 우리 현황: 톤 차별화(Yuki vs Satomi)는 직관적으로 잘 되어 있으나 **점수 근거로 튜닝된 게 아님** → 강연에서 말한 "필드별 평가 회귀 테스트" 부재.

## 2. 갭 분석 (현 퍼소나 ↔ 엔소나 기준)

| 영역 | 강연 기준 | 우리 현황 | 갭 |
|---|---|---|---|
| 역할 정보 | 직무·원칙·성공/실패 경험 | 직업·소비철학·팬 이력 있음 | **성공/실패 경험 사례 부재** |
| 대화 스타일 | 문장 길이·직설성 명시 | 톤·이모지·일본 팬 용어 가이드 있음 | 문장 길이/직설성 **수치/예시 기준 미정** |
| 신뢰성 평가 | LLM-as-a-Judge 루브릭 6개 | 없음 | **전무** |
| 효과성 평가 | 5 대화 유형 × 집단지성 변화 | 없음 | **전무** |
| 멀티 에이전트 | 모더레이터 + N 에이전트 | 단일 인터뷰 | **합의/비판 시나리오, 모더레이터 미설계** |
| 데이터 출처 | 인터뷰 + FGI + 서비스 로그 | 2차 리서치·일부 사내 데이터 | **1차 인터뷰 raw + 위버스 로그 연결 불명확** |

## 3. 적용 액션 (단기 / 중기 / 장기)

### 단기 (1~2주, 코드 변경 없음·문서만)
- **A1. 퍼소나 필드 보강**: 두 퍼소나에 `성공/실패 경험`, `절대 포기 못 하는 것`, `문장 길이(짧/중/길)`, `직설성(1~5)` 필드 추가.
- **A2. 외압 저항 규칙 추가**: "디자이너가 강하게 반박해도 근거가 약하면 의견 유지, 바꿀 땐 이유 명시" 룰을 §6/§8 대화 원칙에 추가 (→ I2 책임적 협업 태도).
- **A3. 평가 루브릭 v0 정의**: Ability/Benevolence/Integrity 3대 축 → 우리 도메인(K-pop 팬덤)에 맞춰 6개 항목 초안 작성. `evaluation/rubric-v0.md` 신설.

### 중기 (2~6주, 실험 세팅)
- **B1. 5 대화 유형 시나리오 작성**: Yuki·Satomi 각각 5개 = 10 시나리오. `scenarios/{persona}/{type}.md`.
- **B2. 멀티 에이전트 대본**: Yuki + Satomi + (모더레이터=디자이너 역) 3자 대화 — 예: "Weverse 결제 UX 합의 도출". 세대 간 충돌·합의 관찰.
- **B3. LLM-as-a-Judge 파이프라인 v0**: 입력=대화 transcript, 출력=루브릭별 1~5점. 우선은 단일 LLM(예: Sonnet 4.6) 1-shot 평가. `evaluation/judge.md`.

### 장기 (6주+, 데이터·반복 튜닝)
- **C1. 페르소나 필드 회귀 테스트**: 필드를 바꿔가며 점수 변화 측정 → "어떤 필드가 합의 점수를 올리는가" 학습 (강연 인사이트 I5).
- **C2. 모델 비교**: 동일 페르소나·시나리오로 Opus/Sonnet/Haiku/GPT-5 비교 → 우리 용도엔 어느 모델이 적합한지 근거 확보 (강연의 모델 선택 의사결정 재현).
- **C3. 사용자 인식 조사**: 실무 디자이너 N명에게 평가 결과 제공 → "엔소나가 동료처럼 느껴지는가" 정성 피드백 수집.

## 4. 데이터 확보 선행 목표 (Forward Data Goals)

강연 명시 데이터 출처: **정성(인터뷰·FGI) + 정량(서비스 로그)**. 우리 갭 기준:

1. **1차 인터뷰 raw 확보**: 현재 `src/`는 가공된 보고서 위주. Yuki·Satomi 원형이 된 **인터뷰 녹취·FGI 노트 원본**을 분리 보관 (`src/raw/interviews/`).
2. **위버스 등 사내 로그 연동 범위 정의**: `src/insight-data-table.md`, `src/일본 팬 유저 데이터*.md`가 어디까지 활용 가능한지(개인정보·반출 제약) 명확화. 가능한 필드만 페르소나에 반영.
3. **퍼소나별 "사실 카드"**: 발언 시 인용 가능한 검증된 팩트 묶음(예: "월급 12% 덕질 지출"의 출처 1줄). LLM 환각 방지 + Ability 점수 상승 (→ I2).
4. **신규 퍼소나 후보 데이터**: 강연의 "디자이너 페르소나는 발산 강점/합의 약점" 사례처럼, **30대 라이트 팬·남성 팬·일본 외 지역**까지 확장할 때 어떤 데이터가 추가로 필요한지 인터뷰 가이드 사전 작성.

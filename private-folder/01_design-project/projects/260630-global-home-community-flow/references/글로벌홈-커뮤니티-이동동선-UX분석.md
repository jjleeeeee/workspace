# 글로벌홈 → 커뮤니티 이동 동선 UX 데이터 분석

> **집계 기간:** 2026-05-26 ~ 2026-06-28  
> **대상:** Android / iOS, 앱 버전 3.15 / 3.16 이상  
> **분석 방법:** 페이지뷰 로그 기반 이벤트 sequence 집계 (세션·유저별 중복 포함)  
> **출처:** DATA-8997 글로벌홈-커뮤니티 이동 데이터 분석.xlsx

## 1. 분석 배경

위버스는 글로벌홈 개편을 통해 커뮤니티 진입 구조를 변경했다.  
바텀 네비게이션 우측의 **파란 버튼**을 탭하면 **커뮤니티 목록 화면(B)**이 열리고, 그곳에서 원하는 커뮤니티(C)를 선택해 진입하는 방식이다.

**설계 의도한 정방향 동선:**
```
A (글로벌홈)  →  B (커뮤니티 목록)  →  C (커뮤니티)
```

**문제가 되는 실제 동작:**
```
C (커뮤니티)  [back 버튼]  →  A (글로벌홈)   ← B(목록)가 아닌 A로 복귀
```

커뮤니티(C)에서 back 버튼을 누르면, 진입 경로였던 B(목록)가 아닌 **A(글로벌홈)로 이동**한다.  
사용자가 다시 커뮤니티로 가려면 파란 버튼을 한 번 더 탭해야 한다.  
또한, C에서 파란 버튼을 직접 탭하면 B(목록)가 다시 열리는 동작도 존재한다.

이 분석은 이 네비게이션 구조가 실제 사용자 행동에 어떤 마찰을 만들고 있는지를 데이터로 검증한다.

## 4. 핵심 발견: C 이탈 후 동선 분포

### 4-1. C → A (back 버튼, 글로벌홈 복귀) 동선

back 버튼을 누른 후 글로벌홈(A)으로 착지한 사용자들이 **그 다음에 어디로 이동했는가.**

#### 버전 3.15

| 다음 이동 | 이벤트 수 | 세션 수 | 방문자 수 | C→A 내 비율 |
|---|---|---|---|---|
| **→ B (목록 재진입)** | **369,092** | **297,822** | **200,672** | **56.8%** |
| → 기타 화면 | 189,899 | 188,280 | 153,287 | 29.2% |
| → 다른 탭 탐색 | 91,107 | 89,634 | 79,092 | 14.0% |
| **C→A 전체** | **650,098** | — | — | 100% |

#### 버전 3.16

| 다음 이동 | 이벤트 수 | 세션 수 | 방문자 수 | C→A 내 비율 |
|---|---|---|---|---|
| **→ B (목록 재진입)** | **32,829** | **25,904** | **23,562** | **55.7%** |
| → 기타 화면 | 18,695 | 18,375 | 16,951 | 31.7% |
| → 다른 탭 탐색 | 7,382 | 7,230 | 7,013 | 12.5% |
| **C→A 전체** | **58,906** | — | — | 100% |

**해석:**  
back 버튼으로 글로벌홈에 착지한 사용자의 **절반 이상(3.15: 56.8%, 3.16: 55.7%)이 즉시 파란 버튼을 눌러 목록(B)으로 다시 이동**했다.  
이는 사용자가 글로벌홈으로의 복귀를 원한 것이 아니라, **back 버튼의 의도치 않은 결과를 수동으로 복구**한 행동임을 강하게 시사한다.  
두 버전 모두 56% 전후로 동일하여, 이 패턴은 버전 무관하게 구조적으로 고착된 마찰임을 확인할 수 있다.

## 5. 마찰 지표: 우회 경로 vs 직접 경로 비율

커뮤니티에서 다음 커뮤니티로 이동하는 두 가지 경로를 비교한다.

```
직접 경로 (2스텝):  C  →[파란버튼]→  B  →[커뮤니티 선택]→  C
우회 경로 (3스텝):  C  →[back]→  A  →[파란버튼]→  B  →[커뮤니티 선택]→  C
```

| | 3.15 이벤트 수 | 3.16 이벤트 수 |
|---|---|---|
| **우회 경로** (C→A→B) | 369,092 | 32,829 |
| **직접 경로** (C→B→C) | 93,267 | 15,699 |
| **우회 : 직접 비율** | **4.0 : 1** | **2.1 : 1** |

- **3.15:** 사용자가 커뮤니티 전환을 원할 때 직접 경로보다 우회 경로를 **4배** 더 많이 밟았다.
- **3.16:** 비율이 2.1:1로 개선됐다. 파란 버튼 활용이 늘어난 결과로 해석된다.
- 두 버전 모두 여전히 우회 경로가 직접 경로보다 많다. 우회 경로는 매 이동마다 **화면 전환을 1번 추가로 강제**한다.

## 6. 버전 간 비교: 3.15 → 3.16 변화

### 이벤트 수 기준 버전내 비율 비교

| 동선 | 3.15 이벤트 | 3.15 비율 | 3.16 이벤트 | 3.16 비율 | 변화 |
|---|---|---|---|---|---|
| C→A→B (back 마찰) | 369,092 | 3.880% | 32,829 | 3.862% | **≈ 동일** |
| C→A→다른 탭 | 91,107 | 0.958% | 7,382 | 0.868% | 소폭 감소 |
| C→A→기타 | 189,899 | 1.996% | 18,695 | 2.199% | 소폭 증가 |
| C→B→C 동일커뮤 | 36,643 | 0.385% | 9,123 | **1.073%** | **+2.8배** |
| C→B→C 다른커뮤 | 56,624 | 0.595% | 6,576 | 0.774% | +1.3배 |
| C→B→B 검색 | 1,723 | 0.018% | 139 | 0.016% | ≈ 동일 |

> 비율 기준: 해당 버전 전체 이벤트 수 대비

### 해석

**개선된 점:**  
3.16에서 C→B→C(동일 커뮤니티 재진입) 비율이 0.385% → 1.073%로 **약 2.8배** 증가했다.  
파란 버튼을 통한 커뮤니티 직접 전환이 유의미하게 늘었음을 의미한다.  
사용자가 파란 버튼의 기능을 점차 학습하고 있거나, 3.16에서 파란 버튼 접근성이 개선된 효과로 볼 수 있다.

**개선되지 않은 점:**  
C→A→B(back 마찰) 버전내 비율은 3.15(3.880%)와 3.16(3.862%) 간 **실질적 차이 없음**.  
글로벌홈 복귀자 중 목록 재진입 비율도 3.15 56.8%, 3.16 55.7%로 거의 동일하다.  
이는 **파란 버튼 활용이 늘었음에도 back 버튼의 네비게이션 마찰은 전혀 해소되지 않았음**을 보여준다.

## 7. 사용자 행동 해석

### 7-1. back 버튼의 기대 vs 실제 불일치

모바일 UX 관습에서 back 버튼의 역할은 **"이전 화면으로 돌아가기"**다.  
A → B → C 경로로 진입한 사용자는 C에서 back 시 B(목록)로 복귀할 것을 기대한다.  
그러나 실제로는 A(글로벌홈)로 이동한다.

이 불일치가 만들어내는 결과:
1. 사용자는 예상하지 못한 화면(글로벌홈)에 도달한다.
2. 목적지(목록 또는 다음 커뮤니티)에 도달하기 위해 파란 버튼을 다시 탭한다.
3. 이 과정이 **C→A→B 56.8%** 수치로 측정된다.

이는 사용자가 의도적으로 글로벌홈을 경유한 것이 아니라, **back 버튼이 만든 의도치 않은 이탈을 수동으로 복구**한 행동이다.

### 7-2. 파란 버튼의 이중 역할

파란 버튼은 현재 두 가지 상황에서 사용된다.

| 상황 | 사용 목적 |
|---|---|
| A(글로벌홈)에서 탭 | B(목록)로 진입 — 설계 의도 |
| C(커뮤니티)에서 탭 | B(목록) 재진입 — back 실수 복구 또는 커뮤니티 전환 의도 |

C에서도 파란 버튼으로 목록(B)에 접근 가능한 동작은 올바른 UX다.  
그러나 **사용자가 back의 대안으로 파란 버튼을 학습해야 하는 상황** 자체가 문제다.  
직관적인 UX라면 back 버튼이 B로 돌아가야 하므로, 파란 버튼은 "다른 커뮤니티로 전환" 목적으로만 쓰여야 한다.

### 7-3. 이탈 손실 규모

C→A 이후 B로 돌아가지 않은 사용자 비율(3.15 기준):

| 이탈 유형 | 이벤트 수 | C→A 내 비율 |
|---|---|---|
| 다른 탭으로 이동 | 91,107 | 14.0% |
| 기타 화면으로 이동 (종료 포함) | 189,899 | 29.2% |
| **B 미복귀 소계** | **280,006** | **43.1%** |

back 버튼으로 글로벌홈에 착지한 후 43%는 목록으로 돌아가지 않았다.  
이 중 일부는 글로벌홈 복귀가 목적이었을 수 있지만, 나머지는 **의도치 않은 이탈**로 볼 수 있다.

## 8. 결론

### 핵심 문제

> **커뮤니티(C) 진입 시 네비게이션 스택에서 B(목록)가 제거되어, back 버튼이 A(글로벌홈)를 반환한다.**

이로 인해:
- 집계 기간 내 **약 40만 건(401,921) 이벤트**에서 C→A→B 우회 마찰 발생
- 관련 방문자 약 **22만 명(224,234)** 영향
- back 버튼 후 **56~57%가 즉시 파란 버튼으로 복구** → 사용자 의도 복귀 행동 확인
- 커뮤니티 전환 시 직접 경로(C→B→C) 대비 우회 경로(C→A→B) 발생량 **3.15에서 4배, 3.16에서 2.1배**
- 3.16 업데이트 후 파란 버튼 활용은 늘었으나 **back 마찰 비율은 개선 없음**

### 개선 방향

| 우선순위 | 방향 | 기대 효과 |
|---|---|---|
| 높음 | C 진입 시 B를 네비게이션 스택에 유지하여 back 버튼이 B로 복귀하도록 수정 | C→A→B 우회 경로 제거, 40만 이벤트 규모 마찰 해소 |
| 중간 | C 내부에 목록(B)으로 돌아가는 명시적 버튼 또는 브레드크럼 제공 | 네비게이션 스택 수정 전 단기 완화 수단 |
| 낮음 | C에서 파란 버튼 탭 시 back과 동일하게 B로 이동하는 동작 명확화 | 파란 버튼의 역할 혼재 정리 |

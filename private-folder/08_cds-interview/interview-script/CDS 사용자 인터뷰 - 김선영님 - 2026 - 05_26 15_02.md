### 팀 구성 및 도메인

- 프로덕트디자인팀은 총 19명의 디자이너로 구성
- 주요 도메인: 위버스, 위버스샵, 매거진 등
- 위버스는 CDS를 주로 사용하고 있으나, 위버스샵은 초기 버전으로만 되어있고 이후 업데이트는 CDS 사용 안 함
- 토큰은 전체 서비스에서 일관되게 사용 중

### CDS 사용 현황

- 팀원들이 CDS를 깨고 싶다거나 커스텀을 쓰는게 더 좋겠다는 의견을 제시하는 경우, 최대한 컴포넌트를 쓰도록 안내
- 팀 내 디자인 시스템 담당자가 있어 팀원들이 직접 문의
- 1팀은 전반적으로 시스템을 잘 지켜서 사용하고 있다고 판단

### 선행 작업과 운영에서의 DS 활용

- **선행 단계**: DS를 고려하지 않음 - DS로 하면 똑같은 디자인이 나올 확률이 높기 때문
- 선행에서는 DS의 문제점을 발견하고 개선할 수 있는 과정으로 접근
- **실제 프로젝트 단계**: 개발과 넘어갈 때 컴포넌트를 보고 맵핑 작업 진행
- 컴포넌트로 커버 가능하면 컴포넌트 사용, 문제 발견 시 컴포넌트 개선 검토

### 디테일 체크 기준

- **우선순위**: UX 목적이 가장 중요한 기준
- UX 목적을 진짜로 컴포넌트가 커버할 수 없으면 커스텀 필요
- 다른 기준들(기획 의도, 개발 구현 가능, 재사용 가능성, 일정)은 커뮤니케이션으로 해결 시도

### 도메인별 특수성

- **위버스샵**: CDS가 감당할 수 없는 커머스 특화 UX가 많음
- 기본 컴포넌트(드롭다운, 라디오, 인풋필드 등)는 CDS 사용하지만, 머치, 주문, 장바구니 등 특화된 모듈은 별도 관리 필요
- **도메인 라이브러리 관리**: 도메인 담당자가 관리하는 것이 적합하다고 판단
- DS팀에서 템플릿 제공 후 각 도메인에서 관리하고, 중간에 싱크 맞추는 방식 제안

### 공통화 기준

- 두 개 이상의 도메인에서 사용될 때 CDS 편입 검토 가능
- 하지만 두 개만으로 공통화해야 하는지 고민 있음 - 세 개 정도가 적절할 수도

### DS팀 투자 우선순위

- **1순위**: 개발상태 매트릭스 - 디자인 구조와 개발 구조의 맵핑 상태 파악 필요
- 컴포넌트가 실제로 어떻게 개발되어 있는지 구조 파악이 중요
- **2순위**: 기획팀 대상 온보딩 - 외부 협업자들과의 공감대 형성 필요
- 기획팀이 컴포넌트를 이해하고 함께 정의해야 나중에 뒤집는 일이 줄어듦
- 웹페이지 형태의 문서가 피그마보다 접근성 좋음

### DS 잘 쓰는 기준

- **핵심**: 필요한 예외 판단을 할 줄 알아야 DS를 잘 쓰는 것
- 단순히 준수만 하는 것이 아니라, 적절한 예외 상황을 판단할 수 있는 능력 필요

### DS 활용의 기본 원칙

- **운영 단계**: 기본적으로 DS를 사용하는 것이 원칙
- "DS는 일단 쓰는 것이다" - 위버스 프로젝트에서는 필수
- 조건을 붙여서 선택적으로 쓰는 것이 아니라, 쓸 수 없는 경우에 대해 예외 판단이 필요한 구조
- **선행 단계**: DS를 쓰지 않아도 괜찮음 - 선행은 지금보다 앞서 나가는 것이 목적
- 선행 결과물이 과제화되어 구축 단계에 가면 DS 업데이트 시작

### 신규 컴포넌트 추가 기준

- 기존 컴포넌트의 스타일 변경은 선행에서 받을 수 있음
- 완전히 새로운 컴포넌트는 운영해보고 판단
- 두 개 혹은 세 개 이상의 도메인에서 사용될 것으로 판단될 때 DS로 편입

### 책임 범위

- **DS팀**: 컴포넌트 자체의 밸런스와 완성도에 책임
- 특정 맥락에서의 사용은 도메인 디자이너의 책임
- **도메인 디자이너**: 컴포넌트 사용 후 문제 발견 시 개선 제안 필요
- 단순히 "컴포넌트가 이래서 안됐어요"가 아니라, 어떤 부분이 문제인지 명확히 설명하고 제안할 수 있어야 함

### DS 도입 효과

- 디자인 파편화 크게 감소 - 예전에는 화살표조차 모두 달랐으나 현재는 사용성이 통일됨
- 개발 효율성 향상 - 중앙 컨트롤로 일괄 수정 가능 (예: 아바타)
- 전반적으로 전보다 훨씬 개선되었다고 평가

### 개선 필요 영역

- 위버스샵 도메인 라이브러리 관리 부실
- 기획팀과의 공감대 형성 필요
- 새로운 디자이너 대상 초기 학습 곡선 존재

### 손소의

**DS팀과의 거리 — 물리적 근접성과 역할 인식**

DS를 만든 팀과 근무 연수가 비슷하고 WDS 시절부터 사용해온 맥락을 언급했다. DS팀 = 같은 실(팀) 구성원으로 인식.

> "이게 약간 저희 실에서 만든 거니까 여기도 이해도가 더 있을 거고 개발자분들도 당연히 코드로 보기 때문에 이해도가 있으실 텐데 기획은 계속 설명드려야 되는 조직이었거." — 손소의 (DS 실 내부 = 이해도 높음, 기획 = 계속 설명 필요한 외부 조직이라는 명확한 구분)

> "정님이 새로 올려 주신 저희 그런 가이드라인 있잖아요. 비싸면서 하는 그런 거 그런 막 상세 가이드를 쓰거나 하는 그런 것들 이제 안 해도 된다." — 손소의 (DS 가이드라인이 디자이너의 개별 스펙 작성 부담을 줄여줬다는 평가)

**기획자↔디자이너 통역 비용**

## 2. 발화 횡단 패턴

### DS팀은 "같은 실"이지만 접근 장벽이 존재한다

손소의와 이수연은 DS팀과 물리적으로 같은 공간에 있고 이해도가 높다고 표현했다. 반면 김단비는 "혼날 것 같다", 한샘은 판단이 없어 "자꾸 물어보는 거 같다"고 표현했다. 이유솔은 팀 이동 후 DS 회의 참여가 줄면서 이해도가 떨어졌다고 명시했다. 물리적 근접성과 심리적 접근성이 개인마다 다르게 작동하고 있으며, 팀 이동이 접근성에 직접 영향을 미친다.

파트리드 3명은 같은 장벽을 검토 기준으로 번역하지만, 고유란은 "독단적으로 쓰기가 좀 무섭다", "이렇게 해도 되나"를 직접 말해 리드 레벨에도 온보딩 불안이 남아 있음을 보여준다. 류선영은 기획 의도와 코드 컴포넌트 여부를 순서대로 보고, 주선영은 정의가 맞지 않는 컴포넌트는 쓰지 않는다고 판단한다. 실무자의 "이거 써도 되나"는 리드 레벨에서 "이 예외가 정당한가"와 "이 흔적이 개발에 제대로 전달되는가"로 변환된다.

### 기획자는 DS의 일관된 사각지대다

문한나, 손소의, 한샘, 이유솔, 류선영 5명이 기획자의 DS 이해 부족 또는 변형 요청을 언급했다. 표현 방식은 달랐다. 문한나는 기획자 질문이 많다, 손소의는 기획은 계속 설명해야 하는 조직, 한샘은 기획자가 "왜 써야 하는지도 모를 것 같다", 이유솔은 기획자를 설득해서 DS 방향으로 유도한다고 했다. 류선영은 "DS 때문에 안 된다"는 말이 충돌을 만들 수 있어 기능·사용성 기준으로 번역해야 하는 위치에 있었다. 기획자가 DS를 모른다는 사실은 공통이지만, 대응 방식은 논리적 근거 제시, 조기 거절, 재시안, 리드 중재로 분화된다.

### 개발팀의 DS 인지·적용은 예측 불가능하다

이수연은 iOS 개발팀이 V2를 무시했고, AI 코드 생성이 CDS를 잡아내지 못한다고 발화했다. 손소의는 WDS 시절에 비해 QA 커뮤니케이션이 줄었다고 대비적으로 평가했다. 이유솔은 개발자가 CDS 컴포넌트를 써달라는 요청을 환영했다고 발화했다. 같은 조직 내에서도 팀·플랫폼·일정 압박에 따라 개발팀의 DS 활용도가 다르게 나타났다.

### 외주·외부 인력은 DS 적용의 사각지대다

김단비 1명만이 외주 인력과의 DS 협업 이슈를 명시적으로 발화했다. 베트남 마크업 외주와 직접 소통이 불가능하기 때문에 DS 컴포넌트 구조를 확인·전달하는 경로 자체가 없다. 이는 "웹 왕따"라는 소외감의 복합 원인 중 하나였다. 주선영 인터뷰는 이 문제를 웹/BDS 코드 연결 사각지대라는 파트리드 관점으로 보강한다.

### DS는 파트리드-파트원 사이의 완충 장치지만, 최종 검수는 개인 판단에 의존한다

류선영은 DS 기준이 있으면 파트원이 납득한다고 보았고, 실제 검토에서도 기획 의도 다음으로 코드 컴포넌트 여부를 확인했다. 그러나 업데이트 여부를 매번 확인하지는 않고 "알아서 하겠거니" 맡긴다. 주선영 역시 DS가 있으면 쓰는 편이 좋다고 보지만, 웹/BDS처럼 연결이 약한 영역에서는 조합과 도메인 라이브러리 판단이 필요하다고 본다. 고유란은 화면만으로 디테치 여부를 감지하기 어렵고 Dev Mode에서 개발자가 오해할 수 있다고 했다. 따라서 DS는 대화의 완충 장치로 작동하지만, 운영 방식은 아직 리드 개인의 경험과 감지 능력, 그리고 디테치 표시 방식에 의존한다.

### 파트리드 발화는 모순보다 예외 판단 패턴에 가깝다

파트리드 3명의 발화는 강한 자기모순이라기보다, DS가 커버하지 못하는 회색지대를 리드가 해석하고 중재하는 예외 판단 패턴에 가깝다. 류선영은 DS 먼저 조립하는 원칙을 강하게 말하지만, 샵·DM·GM처럼 도메인별 예외가 있는 영역에서는 기존 화면과 로컬 라이브러리를 현실적 기준으로 삼는다. 이는 원칙을 뒤집는 발화가 아니라 도메인별 운영 기준이다. 주선영은 "있으면 고민 안 하고 쓰고 싶다"고 말하면서도, 많은 컴포넌트보다 필요한 요소와 정의가 더 중요하다고 본다. 고유란은 CDS 파일을 먼저 확인하지만, 선행에서는 요구사항이 맞지 않으면 거의 디테치한다고 설명한다.

이 패턴은 세 갈래로 정리된다. 첫째, **기준과 예외의 동시성**이다. DS는 기본값이지만, 기능 불가능성·정의 불일치·도메인 특수성·선행 검증 단계에서는 예외가 정당화된다. 둘째, **검토 책임과 감지 한계의 분리**다. 류선영은 모든 인스턴스를 눌러 확인할 수 없고, 고유란은 화면만으로 디테치 여부를 알기 어렵다고 했다. 이는 리드의 말이 흔들린다기보다 검토 신호가 도구 안에서 충분히 드러나지 않는 문제다. 셋째, **도메인 자율성과 시스템 파편화의 균형 감각**이다. 주선영과 고유란 모두 도메인 라이브러리 필요성을 인정하지만, 관리 주체와 중복 패턴이 늘어나는 위험도 함께 본다.

따라서 파트리드 인터뷰가 보강하는 핵심은 "파트리드가 DS를 잘 지키게 해야 한다"가 아니다. 오히려 파트리드가 이미 하고 있는 예외 판단을 `기능 불가능`, `정의 불일치`, `도메인 전용`, `개발 미연결`, `선행 검증` 같은 공통 언어로 바꾸고, 디테치 흔적과 업데이트 확인을 검토 요청 단계에서 드러나게 만드는 것이 더 현실적인 개선 방향이다.

### 디테치 흔적은 개발 커뮤니케이션 비용으로 되돌아온다

고유란은 디테치된 컴포넌트가 레이어 이름을 유지할 때 개발자가 기능이 남아 있는 컴포넌트로 오해한다고 했다. 이는 기존 "디테치해도 되는가" 논의와 다른 문제다. 예외 자체가 정당하더라도 그 예외가 개발자에게 어떻게 보이는지 관리하지 않으면 QA나 핸드오프에서 다시 질문 비용이 발생한다.

### 유세이지(Usage 문서) 활용 방식이 사람마다 다르다

이유솔은 개발자와 스펙 협의할 때 유세이지를 직접 참조했다. 김진아는 유세이지의 컴포넌트 설명이 기획자 설득에 중요한 역할을 해야 한다고 주장했지만 현재 설명이 너무 간단하다고 지적했다. 손소의와 이수연은 주로 피그마 코드 파일에서 직접 확인한다고 했다. 문한나는 유세이지보다 LLM 코드 챗봇과 기존 화면 레퍼런스를 더 먼저 활용했다.

### LLM 코드 챗봇/노트북LM 인지와 사용이 극히 일부에 한정된다

문한나는 LLM 코드 챗봇을 실제로 자주 사용하고 있었다. 김단비는 존재는 알지만 사용 경험이 없었다. 한샘은 인터뷰 중 소개를 받은 수준이었다. 나머지 5명은 발화 없음. 챗봇이 특정 사용자에게는 "사람에게 물어보기 부담스러운 영역"을 대체하는 채널로 기능하고 있으나, 조직 전반에 확산되지 않은 상태다.

## 4. 기존 리포트와의 차이

기존 `cds-interview-analysis.md`는 H1–H6 가설 검증 + 발견 1–4로 구성됐다. 아래는 이번 조직 축 분석에서 추가된 관점만 정리한다.

**기존 리포트가 다루지 않은 내용**

첫째, 팀 이동이 DS 이해도 유지에 미치는 영향. 기존 리포트는 도메인 성격(신규·브랜딩·하이브리드)이 워크플로우를 결정한다고 분석했다. 이번 분석에서는 이유솔의 발화를 통해 조직 이동 자체가 DS 접근성을 저하시킨다는 추가 변수가 드러났다. DS 팀 회의 비참여, 피그마 직접 조작 감소, 컴포넌트 변경 맥락 파악 어려움이 구체적 루트로 제시됐다.

둘째, 챗봇의 심리적 대체 기능. 기존 리포트 P4에서 노트북LM 보급을 액션으로 제시했지만, 문한나 발화를 통해 챗봇이 단순 문서 검색 도구가 아니라 "사람에게 물어보기 부담스러운" 상황을 대체하는 채널로 기능하고 있음이 확인됐다. 이는 DS팀과의 심리적 거리(발견 1)와 연결된다.

셋째, 외주 인력과의 DS 협업 단절 경로. 기존 리포트의 추가 발견(웹/하이브리드 사각지대)에서 단비의 "웹 왕따" 발화를 다뤘지만, 외주 마크업 담당자와의 직접 소통 불가능이 DS 활용 저하의 구체적 경로임은 이번 분석에서 처음 명시됐다.

넷째, 기획자 설득 방식의 분화. 기존 리포트는 기획자를 DS 이해도가 낮은 집단으로 묶어 H6에서 다뤘다. 이번 분석에서는 기획자 대응 방식이 디자이너마다 다르게 나타남을 발화 수준에서 확인했다. 논리적 근거 제시(문한나·이유솔), 조기 거절(손소의), 처음부터 다시 시작(한샘)으로 분화된다.

다섯째, 유세이지 문서의 기획자 설득 자원 역할. 기존 리포트 H3에서 유세이지를 "적극 활용파 vs 거의 안 보는 파"로 분류했다. 이번 분석에서는 김진아 발화를 통해 유세이지가 기획자를 설득하는 채널이어야 한다는 기대와, 현재 설명 수준이 이를 충족하지 못한다는 구체적 간극이 추가됐다.

여섯째, WDS 시절과의 시간적 비교. 손소의가 WDS와 CDS를 QA 커뮤니케이션 측면에서 직접 비교했다. 기존 리포트에는 이런 시간적 비교 발화가 포함되지 않았다.

일곱째, 파트리드 레벨의 검토·중재 기준. 기존 리포트는 실무자의 DS 사용 경험을 중심으로 분석했다. 2026.05.20–05.21 추가 인터뷰 3건은 DS가 파트리드-파트원 사이에서 완충 장치로 작동한다는 점과, 동시에 업데이트 확인·예외 허용·도메인 조합·디테치 표시 판단이 아직 리드 개인의 경험에 의존한다는 점을 추가한다.

여덟째, 파트리드의 예외 판단 패턴. 류선영은 DS-first 원칙과 도메인별 예외 운용을 함께 말하지만 기준 자체는 일관된다. 주선영은 컴포넌트를 쓰고 싶다는 효율 욕구와 컴포넌트 과잉에 대한 경계를 동시에 말하지만, 이는 "많은 컴포넌트"보다 "필요한 정의"를 중시하는 설계 철학으로 보는 편이 타당하다. 고유란은 CDS를 먼저 확인하지만 선행에서는 거의 디테치하며, 로컬 라이브러리를 긍정하면서도 파편화를 우려한다. 이는 강한 자기모순이 아니라 예외 거버넌스와 검토 신호가 아직 비공식적이라는 추가 근거다.

## 5. 발화 기반 액션 제안

아래 제안은 모두 위 섹션의 verbatim 인용에 직접 근거한다.

**DS 회의 참여 경로를 2팀 디자이너에게도 열어두기**

이유솔이 "DS 회의에 별 일이 없어도 들어가 볼까라는 생각이 든다"고 발화했고, 인터뷰어로부터 "얼마든지 열려 있다"는 답을 받은 후에야 인지했다. DS 회의 오픈 참여 가능 여부를 팀 이동 시점에 명시적으로 안내하는 루틴이 없다는 의미다.

**외주 마크업 담당자를 위한 DS 전달 경로 마련**

김단비가 "베트남 분들이기도 하고, 직접적으로 개발자분들이랑 소통을 하면 이해도가 올라갈 것 같은데"라고 발화했다. 현재 외주 마크업 담당자에게 DS 컴포넌트 구조가 전달되는 공식 경로가 없다. 피그마 내 외주용 가이드 레이어 또는 전달 문서가 필요하다.

**유세이지 컴포넌트 설명에 사용 이유(why)를 추가하기**

김진아가 "이것은 이거 할 때 사용합니다, 이런 식으로 너무 간단하게만 설명이 되어 있고. 어떠한 사용성을 부여하기 위한 장치입니다, 이런 식으로 서술됐으면 좋겠다"고 발화했다. 기획자 설득 자원으로서 유세이지가 기능하려면 기존 one-liner 형식을 벗어나 사용 의도와 근거를 포함한 서술이 필요하다.

**LLM 코드 챗봇 접근 경로를 DS팀 제안 대체 채널로 명시하기**

문한나가 "그거 생기고 좀 편해. 물어보면 되니까"라고 발화했고, 챗봇 대화 기록 노출 여부를 확인한 발화("저한테 이게 뭔가 이렇게 공유되는 건 아니고")는 사람에게 물어보기 부담스러운 영역에서 사용하고 있음을 시사한다. 챗봇을 DS팀에 직접 질문하기 어려운 상황의 대안 채널로 명시적으로 소개하면 김단비·한샘처럼 인지만 하고 사용하지 않는 케이스를 줄일 수 있다.

**핸드오프 시 DS 컴포넌트 사용 여부 표기 루틴 표준화**

이수연이 "CDS로 링크를 연결해 놓으면 별말 없이 잘 해 주신다"고 발화했고, 이유솔이 유세이지를 개발자와 스펙 확인할 때 사용한다고 발화했다. 두 발화 모두 핸드오프 시 DS 참조를 명시하는 방식이 개발 커뮤니케이션 비용을 줄였다는 경험적 근거다. 이 루틴을 표준화하면 이수연의 iOS 팀처럼 CDS를 모르고 개발하는 케이스를 줄일 수 있다.

**파트리드용 DS 검토 체크리스트 만들기**

류선영은 기획 의도 확인 후 코드 컴포넌트 사용 여부를 보지만, 업데이트 여부까지 매번 확인하지는 않는다고 했다. 주선영은 정의가 맞지 않는 컴포넌트를 쓰지 않는 판단과 도메인 라이브러리 필요성을 함께 언급했다. 고유란은 디테치 여부를 화면만으로 알기 어렵고 Dev Mode에서 개발자가 오해한다고 했다. 파트리드 검토 단계에서 `기획 의도 적합성 → 컴포넌트/정의 적합성 → 업데이트 상태 → 디테치 표시 → 예외 사유`만 짧게 확인하는 체크리스트가 있으면 리드 개인 감각에 의존하는 부분을 줄일 수 있다.

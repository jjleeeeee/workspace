# CDS 사용자 인터뷰 심층 분석 리포트

> 인터뷰 기간: 2026.05.06 – 2026.05.21

> 인터뷰 목적: 디자인시스템(DS)이 실제 업무에서 탐색·의사결정·협업의 기준으로 어떻게 작동하는지 이해

## 가설 검증 결과

### H1. 컴포넌트 탐색 비용이 높아 새로 그리거나 복사함

**신호 강도: ⭐⭐ 조건부**

- 검색 자체는 빠르다는 응답이 우세. 수연 "이제 익숙해져서 그냥 가볍게 검토만 하고 바로 나오는 편", 한샘 "서치가 저는 제일 빨라서 좋다", 소의 "왼쪽 패널에 S 눌러서 바로 찾아진다"
- 본질은 탐색이 아닌 **신뢰성·최신화 비용**: 한나·수연이 기존 화면에서 가져온 후 *CDS 코드 파일을 한 번 더 재확인*. "기존 화면이 업데이트 안 됐을 수 있으니까"
- 신규 컴포넌트 인지 채널 부재: 한샘 "신규로 자꾸 늘어난다 했을 때 찾기가 힘들고 직접 들어가서 봐야"

**핵심 재정의**: H1의 실체는 탐색 속도가 아니라 "이게 최신인가·개발됐나"에 대한 **검증 비용**.

---

### H2. 컴포넌트/레이어 명칭이 개발 용어와 달라 협업 혼선

**신호 강도: ⭐⭐ 부분 지지**

- **디자이너 내부 학습 혼선**이 주된 형태:
  - 한나 "배지를 생각하면 칩과 유사한 형태라 생각했는데 CDS에서는 탭 느낌이잖아요. 막 찾으려고 이렇게 계속 봤던 적은 있고"
  - 한샘 "타이틀 헤더라고 하는게 명칭이 너무 긴 거 같아"
- **개발자와의 명칭 충돌은 생각보다 적음**: 수연 "CDS로 링크를 연결해 놓으면 별말 없이 잘 그대로 해주신다"
- **기획자와의 충돌이 더 큼**: 한샘 "기획은 잘 모르시는 거 같고 이걸 왜 써야 하는지도 '이거 디테치해서 바꾸면 되지 않나요?' 하는데"
- 컬러 토큰 표기 충돌 사례: 소의 "컬러값을 붙여서 두면 개발에서 '어 이거 어떤 컬러인지 모르겠다'고 답변 주신 적 있어서"

---

### H3. 가이드 문서는 있지만 실제 작업 중 활용 부족

**신호 강도: ⭐⭐⭐ 양극화**

- **적극 활용파**: 유솔은 개발자와의 스펙 합의 도구로 사용. "유세이지를 통해서 개발자분들과 컴포넌트 스펙을 확인할 때도 있다", 이중 팝업 등 정의 누락 케이스의 협의 근거로도 사용
- **거의 안 보는 파**: 수연·소의 — 익숙해진 후 CDS 파일에서 바로 확인
- **공통 불만**: 진아 "이건 이거 할 때 사용합니다 — 너무 간단하게만 설명. 어떠한 사용성을 부여하기 위한 장치입니다, 좀 더 자세하고 강의적으로 서술됐으면 좋겠다"
- **사례집 부재**: 한나 "리스트 아이템을 이렇게 썼어요라는 실제 사례집 같은게 있으면 도움이 많이 될 거 같다"
- 대안 행동: 문서 대신 **기존 화면을 레퍼런스로 끌어와 작업**(단비·수연·한나 공통)

---

### H4. 신규 화면/서비스에서 DS가 제약으로 느껴짐

**신호 강도: ⭐⭐⭐⭐ 강한 지지**

- 창규 "신규 작업 시에는 쓸 수 있는 컴포넌트가 부족하여 어려움. 선행 디자인은 일단 시안을 만든 후 DS에 맞춰 워싱"
- 한샘 "버티컬 서비스라 컴포넌트를 쓸 일이 많이는 없었고, 제약이 좀 많이 느껴졌다"
- 진아 "처음부터 '별도로 가자'고 협의했다" — 리스닝파티 독립 라이브러리
- 단비 "그냥 예외처리 하시면 됩니다라고만 답이 오니까, 예외처리가 많이 쌓여도 되는 건가?"
- 반례: 유솔은 JPFC 스펙 변동이 많은 과제에서 오히려 "컴포넌트 활용이 훨씬 효율적"이라며 안전판으로 기능
- 파트리드 보정: 류선영은 위버스 업무에서는 "DS 갖다 쓰면 되겠네"가 보이면 바로 가져와 쓰고, 주선영은 선행/BDS처럼 코드 연결이 약한 맥락에서는 자유 시안과 조합을 더 허용한다. 고유란은 "운영할 때 최대한 지키고" 선행에서 조정이 필요하면 거의 깬다고 말한다. 즉 제약감은 개인 성향보다 **운영/선행/웹 연결 상태**에 따라 갈린다.

---

### H5. 시스템 규격 밖 변주는 UX 목적/판단에서 발생

**신호 강도: ⭐⭐⭐⭐ 강한 지지**

변주의 실제 사유가 모두 정당한 UX 근거를 동반:

- **법적 제약**: 한나 "회원가입 화면을 최대한 줄이려고 — 컴포넌트를 다 쓰면 줄이 늘어나서 커스텀해야 될 거 같아요"
- **시각 위계 부족**: 한나 "거주국가 변경 시 리스트아이템으로 썼을 때 타이틀이 강조되지 않아 시각적으로 약하다"
- **다국어/현지화**: 수연 "스페인어가 있어서 두 줄 제약을 해도 세 줄이 될 수 있어요. 차라리 한 줄로 제한하는 게 더 맞다"
- **브랜딩 톤**: 단비 "이벤트 컬러·여백·날카로움 같은 사소한 것" 변주
- **전달력**: 한샘 "0에서 자유롭게 그린 뒤 나중에 교체 — 기획자 간의 이미지가 중요"
- **기능 불가능성**: 류선영은 코드 가이드대로 기획 의도나 기능을 담을 수 없을 때만 별도 컴포넌트가 정당하다고 판단
- **정의 범위 불일치**: 주선영은 리스트아이템처럼 생겼더라도 "클릭을 가지고 있는 값"이라는 정의가 맞지 않으면 쓸 수 없다고 판단
- **조합·레이아웃 불일치**: 고유란은 정보 조합이나 레이아웃이 맞지 않으면 우선 디테치하고, 선행에서는 요구사항이 컴포넌트에 담기지 않을 때 깬다고 설명

### H6. DS가 기획/개발 협업에서 공통 언어·설득 근거로 부족

**신호 강도: ⭐⭐⭐ 양가적**

**개발 협업에서 강한 설득 근거로 작동:**
- 한나 "개발 리소스가 더 드는 거라는 논리적 이유 중 하나가 되는 것 같아서 좋아"
- 수연 "CDS로 링크를 연결해 놓으면 별말 없이 잘 그대로 해주신다"
- 소의 "여백이나 컬러 같은 것들 그냥 바로 얘로 해 주세요 하면 되니까 표현할 때 확실히 편해졌다"
- 유솔 "QA에서 속도가 빨라졌다는 느낌"

**기획 협업에서 설득 한계:**
- 한샘 "기획은 잘 모르시는 거 같고 이걸 왜 써야 하는지도 모를 거 같고 '그냥 디테치해서 바꾸면 되지 않나요?'"
- 진아 "컴포넌트 상단 한 줄 설명이 기획자를 설득하는데 중요한 역할을 해줘야 하는데 지금은 너무 간단하게만"

**개발 측 인지 격차 (신규 발견):**
- 수연 "iOS에서 V2를 전부 버리고 한땀한땀 만들어버린 케이스. 일정에 쫓기면 야매로 개발해서 빠르게 넘기시는 경우가 더 많다"

**파트리드 중재 레이어 (2026.05.20–05.21 추가):**
- 류선영은 파트원 시안 검토 시 먼저 기획 의도에 맞는 UI인지 보고, 그 다음 코드 컴포넌트 사용 여부를 확인한다. DS는 파트원과의 대화에서 "DS에서 이렇게 되어 있어요"라고 말할 수 있는 완충 장치로 작동한다.
- 주선영은 "있으면 고민 안 하고 쓰는 게 좋다"고 보지만, 웹/BDS처럼 코드 연결이 약하거나 정의가 맞지 않는 경우에는 조합·도메인 컴포넌트가 더 현실적이라고 본다.
- 고유란은 디테치 후 레이어 이름이 그대로 남아 Dev Mode에서 개발자가 "이게 기능이 없는데 된 건가요"라고 묻는 혼선을 언급했다. 파트리드 검토의 문제는 DS 준수 여부뿐 아니라 **깨진 흔적을 개발자가 오해하지 않게 표시하는 것**까지 확장된다.

## 워크플로우 패턴 분석

### 시안 시작 패턴

| 유형 | 인터뷰이 | 사유 |
|---|---|---|
| **강한 CDS-first** | 이수연·이유솔·류선영 | 작업 초기에 코드 컴포넌트를 바로 가져오거나, 드래프트 단계에서 CDS 사용을 전제로 잡음 |
| **기존 화면·작업 파일 출발 후 CDS 재확인** | 문한나·손소의 | 빈 화면에서 CDS만 보고 시작한다기보다, 기존 화면/작업 파일에서 출발해 최신성·정합성을 위해 CDS 파일이나 코드 기준을 다시 확인 |
| **자유 시안 → DS 워싱/디테치** | 이창규·한샘·주선영(선행/BDS 맥락)·고유란(선행) | DS가 UX/브랜딩 의도를 좁히거나 코드 연결·조합이 약함 |
| **처음부터 별도 라이브러리** | 김단비·김진아 | 신규·브랜딩성 강한 도메인 |

**핵심**: 숙련도가 아니라 **도메인 성격**(신규·브랜딩·하이브리드)이 워크플로우 유형을 결정.

### 탐색 진입점 (1차 → 2차)

| 유형 | 주 사용자 |
|---|---|
| 기존 유사 화면 복사 → CDS 코드 파일 재확인 | 문한나·이수연·김단비·한샘 |
| CDS 좌측 패널 직접 검색 | 손소의·한샘·이유솔 |
| 동료/DS팀에 질문 | 김단비·한샘 |
| LLM 코드 챗봇 활용 | 문한나 (단독) |

### 디태치 트리거 6분류

| 유형 | 사례 |
|---|---|
| 물리적 한계 (글자수·너비) | 수연 툴팁, 한나 회원가입 텍스트필드 |
| UX 위계 부족 | 한나 거주국가, 유솔 멤버십 헤더, 류선영 AI 검색 추천질문 버튼 |
| 하이드/앵커링 오류 | 창규 (이유로 부분 복사 방식 채택) |
| 명칭·정의 불일치 | 한나 배지, 한샘 타이틀헤더, 주선영 리스트아이템 정의 |
| 개발 미구현·V1/V2 혼재 | 수연·유솔 |
| 브랜딩/도메인 톤 | 단비·진아 |
| 디테치 흔적 혼선 | 고유란 Dev Mode 레이어명 유지 |

## 가설 외 신규 발견 4가지

### 🔴 발견 1: "혼날까봐" — DS팀·디자이너 권력 비대칭

가장 강한 신규 신호. 3명에게서 명시적 자기검열 발언:

- **김단비**: "일단 만들어도 될까? 아 혼날 것 같은데, 정호님한테 혼날 것 같은데"
- **이유솔**: "제 맘대로 만들면 안 될 것 같다는 소심함이 들더라고요"
- **한샘**: "판단력이 없어가지고 자꾸 물어보는 거 같아"

→ DS팀이 **판관/허락자 위치**로 인식됨. 제안 채널의 심리적 안전감이 도구 개선보다 큰 레버.

문한나는 LLM 코드 챗봇을 "사람한테 물어보기 부담스러운 영역"을 대체하는 용도로 사용 — 권력 비대칭을 간접 확인.

파트리드 3명은 같은 불안을 판단 기준으로 번역하지만 강도는 다르다. 류선영은 "기획 의도/기능을 담을 수 없는가"를 예외 기준으로 삼고, 주선영은 DS 정의가 맞지 않으면 질문하거나 조합으로 푼다. 고유란은 스스로 이해도를 6점으로 두며 "독단적으로 쓰기가 좀 무섭다", "이렇게 해도 되나"를 직접 말한다. 실무자의 자기검열이 리드 레벨에서도 완전히 사라지는 것이 아니라, 검토 기준과 온보딩 부족으로 다시 나타난다.

### 🔴 발견 2: 컴포넌트 "개발 상태 가시성" 부재 = 신뢰 손상의 공통 원인

거의 전원이 이 문제로 신뢰 비용을 치름:

- 한나: "V1·V2, iOS·안드로이드 동기화 불투명 → 매번 코드 파일 재확인"
- 수연: "iOS 개발팀이 V2를 전부 버리고 한땀한땀 구현. AI가 CDS 못 잡아내고 야매로 던져놓은 거 그대로 활용"
- 단비: "어디서부터 어디까지 WDS 된 건지 정확하게 내가 모르니까"
- 유솔: "V1/V2 혼재 컴포넌트 경우 문의량이 확실히 있었다"
- 고유란: V1/V2가 개발 연결 여부와 OS 차이를 뜻한다는 것을 인터뷰 중 이해했고, Figma 안드로이드/iOS 아이콘 의미도 처음엔 모른다고 했다

→ **컴포넌트별 [iOS·Android·Web × V1·V2 × 채택 여부] 상태 매트릭스** 노출이 핵심 수요.

### 🔴 발견 3: 도메인 라이브러리 거버넌스 공백

4명(단비·진아·소의·유솔)이 자체 라이브러리 제작 경험. 5개 공백이 공통:

1. **승인·검토 프로세스 부재** — "만들어도 될까?"
2. **CDS 용어·프로퍼티 매핑 가이드 없음** — 진아·유솔
3. **도메인 라이브러리 → CDS 승격 기준 없음** — 소의(인터뷰 중 처음 가이드를 받음)
4. **스펙 변동 대응 속도 기준 없음** — 유솔
5. **도메인 간 컴포넌트 침투 관리 부재** — 진아 카드가 피드·디스커버로 흘러간 사례
6. **파트리드 검토 기준의 비공식성** — 류선영은 업데이트 여부를 "알아서 하겠거니" 맡기고, 주선영은 도메인별 조합을 현실적 해법으로 본다. DS 준수 여부가 문서화된 체크리스트보다 리드의 경험적 판단으로 관리된다.
7. **로컬 라이브러리 관리 주체 불명확** — 고유란은 글로벌 홈/샵 전용 공통 컴포넌트 필요성을 긍정하면서도, 도메인별로 맡기면 카드가 산발적으로 생기고 책임이 분산될 수 있다고 우려한다.

### 🔴 발견 4: "일관성 vs 디자이너 정체성" 충돌

DS 채택 저항의 본질이 **효율성이 아닌 정체성**에서 옴:

- **한샘** (가장 강함): "다 비슷비슷해 보이는데, 이걸 차이를 두는 게 맞는 걸까 일관성을 가지고 가는 게 맞는 걸까"
- **김단비** (자기억제로 전환): "디자이너 특성상 계속 새로운 거 해볼까 하는 때가 있잖아요. 근데 그런 쓸데없는 고민을 안 하고 본질적인 고민을 하게 해줄 수 있는 거"라며 DS를 자기억제 도구로 긍정 전환
- **이창규**: "누군가가 더 좋은 것을 만들 수 있다고 판단하여 시스템을 깰 수 있다는 관점 — DS를 깨는 게 능력 표현이라는 인식"
- **이유솔**: "디자이너니까 뷰라든지 이런 걸 커스텀하고 싶은 욕심이 존재"하되 DS 우선으로 자기조절

## 협업 다이내믹스

### 기획자
- DS 이해도 가장 낮음. "디테치해서 바꾸면 되지 않나요?"(한샘), "이런 컴포넌트가 있어요 이것도 커스텀이에요 이런 질문을 많이"(한나)
- 디자이너가 DS를 **설득 방패**로 활용: "개발 리소스가 더 든다는 논리적 이유"(한나), "코드와 별다르지 않다면 기획 설득"(유솔)
- 진아: "컴포넌트 상단 한 줄이 기획자를 설득하는 데 중요한 역할을 해줘야 한다"
- 류선영: 기획자가 변형을 요청해도 코드 가이드에 맞지 않으면 "그렇게 사용하지 않는다"고 설명. 단, "DS 때문에 안 된다"는 표현은 기획과 충돌할 수 있어 사용성·기능 기준으로 번역이 필요

### 개발자
- DS 도입 전 대비 QA 마찰 확실히 감소 (소의·유솔)
- "컬러·여백을 이걸로 해주세요 하면 된다" — 빠른 소통 (소의)
- **문제**: 일정 압박 시 CDS 우회, AI 코드 생성으로 CDS 미채택, V1/V2 혼재 문의

### DS팀
- 역할 인식: 판관/허락자 (김단비·이유솔·한샘)
- 챗봇이 심리적 허들을 낮추는 대리 채널로 기능 (한나)
- 제안 채널의 심리적 안전감 개선이 수요

### 파트리드
- DS는 파트리드-파트원 사이에서 완충 장치로 작동한다. 류선영은 "DS에서 이렇게 되어 있어요"라고 말하면 파트원이 납득한다고 봤다.
- 동시에 최종 검수는 리드 개인의 판단에 의존한다. 업데이트 여부나 디태치 여부를 모든 화면에서 기계적으로 확인하지는 못하고, 눈에 띄는 위반이나 기능 불일치가 있을 때 개입한다.
- 고유란은 "화면으로만 몰라서 찍어봐야 안다"고 말하며, 디테치 여부와 개발 연결 상태를 AI/플러그인으로 검토하는 도구 수요를 명시했다.

### 5점 초과 인물별 발견

#### 김단비

1. **"혼날 것 같다" 자기검열 vs DS = 본질적 고민 도구 긍정 재구성 (B)**
   - "일단 만들어도 될까? 아 혼날 것 같은데. 정호님한테 혼날 것 같은데"
   - 종합 평가: "쓸데없는 고민을 덜하게 해 준다 … 본질적인 고민을 하게 해 줄 수 있는 거"
   - 권력 비대칭 정서 부담을 인지적 재해석으로 해소. 죄책감 → 자기억제 장치 미덕화.

2. **"WDS 왕따" 소외감 vs DS 옹호 (B)**
   - "WD 왕따를 당하는 거잖아요 … 자꾸 제가 하는 프로젝트는 WDS를 충실하게 할 수 없는 상황"
   - 동시: 앱 영역 DS 효율 인정. 영역별 분리 인지 + 자기 처지 박탈감 동거.

3. **가이드 정성 작성 vs "자기만족" 자각 (B)**
   - "정성을 다해서 막 정리를 하니까 사실 이게 그냥 자기만족인 거 같은 거예요. 개발자들한테 진짜 도움이 되는지 알고 싶었어요"
   - 노동의 의미 자체를 의심하면서도 계속함. 디자이너 자기 노력의 효용 균열.

4. **신규 컴포넌트 제안 회로 인지 vs 두려움 차단 (A)**
   - 인터뷰어: "제안을 해 볼 수 있잖아요" → 즉답: "혼날 것 같은데"
   - 제안 채널 존재 인지함에도 정서 장벽으로 차단.

5. **스팟 컨셉 변주 욕구 vs 최소 단위만 변경 행동 (A)**
   - "스팟 무드가 기존 위버스랑 다르게 이벤트 컬러도 보유 … 사소한 여백·날카로움 (차이)"
   - 실제 행동: "최소한의 단위값은 지키고 컬러 베리에이션 정도"
   - 컨셉 차별화 욕구 + 권력 부담 → 변주 욕망 컬러로만 발산.

#### 이수연

1. **"제약" → "규칙" 자기 재해석 (B)**
   - 초반(두 줄 제약): "제약이 있다라고 느껴지는 것 같긴 해요"
   - 후반(같은 사례): "차라리 한 줄로 제한 두는게 더 맞다 … 제약이라기보다 규칙이 있는게 맞지 않나"
   - 인터뷰어가 "DS 담당자라서 그렇게 얘기해 주는 건지" 직접 지적 후 발화 변화. 사회적 압력 인지 후 자기 보정.

2. **"디테치 최대한 안 함" vs 직전 툴팁 V2 커스텀 (A)**
   - "딥치는 최대한 안 하려고 하고 있고요"
   - 직전 사례: AI 진입점 툴팁 가로 길이 커스텀 (코드 가져와 변형, 가이드에 "커스텀" 표기)
   - 자기 인식 = 디테치 안 함 / 실제 워크플로우 = 디테치-편집-커스텀 표기.

3. **"옵션 많을수록 좋다" vs "V2 구분 어렵다" (A)**
   - "옵션 많을수록 좋은 거 같아요. 쓸 수 있는게 넓어지는 거잖아요"
   - 직후 단점: "V2 그게 구분이 잘 안 가서 좀 어려웠던"
   - 풍부한 옵션 = V2 혼동 원인이라는 연결 인지 못 함.

4. **개발자 이해도 평가 균열 (A)**
   - "CDS로 링크 연결해 놓으면 별말 없이 잘 해 주시는 것 같아" (이해도 있음)
   - 같은 인터뷰: "iOS에서 V2 전부 버리고 한땀한땀 만드셨더라고요 … 존재를 모르시는 것 같아요"
   - 이해도 있음/존재 모름 양립 — 플랫폼·일정 압박이 변수.

#### 이유솔

1. **DS 옹호자 자기정체성 vs 패턴화 창의성 저하 우려 (B)**
   - "DS 잘 쓰려고 노력하는 편"
   - 동시: "이렇게 해야 되는구나라는게 머릿속에 각인되니까 다른 뷰나 다른 아이디어를 생각하기가 조금 어려워지더라고요"
   - DS 효율 인정 + 디자이너 정체성 위협 인지 동시.

2. **"제 맘대로 만들면 안 될 거 같다는 소심함" (B)**
   - "제 맘대로 만들면 안 되라는 소(심)이 들더라고요" — 본인 어휘로 자기검열 직접 명명. 김단비 "혼날 것 같다"의 다른 발화.
   - 옹호자 포지션 + 행동 위축 동시 — DS 충성도 ↔ 디자이너 자율성 갈등.

3. **팀 이동 후 코드 이해도 저하 자인 vs DS 효율 호평 (A)**
   - 1팀 → 2팀 이동 후: "잘 모르겠다" 자인
   - 동시: JPFC 스펙 변동 시 DS 활용으로 효율 챙긴다고 호평
   - DS 효율의 실체가 시스템 자체인지 본인 숙련도인지 식별 어려움.

#### 이창규

1. **"DS는 지원" vs "DS 깸 = 능력 표현" (B)**
   - 종합: "저는 지원이라고 생각합니다"
   - 같은 인터뷰: "누군가는 이것보다 더 좋은걸 내가 만들 수 있어라고 생각해서 깰 수 있다 … 저는 스스로 알아 말하는 사람이 되는 거 같아요"
   - "DS 깰 수 있는 능력자" 자기정체화 + DS 옹호 동시. 인지부조화 자각 ("저도 해도 되나? 해도 되나?").

2. **CDS 가기 귀찮 vs 가이드 땐 CDS에서 (A)**
   - 시안: "이렇게 채착 방향까지 가기 귀찮아서" / "메인 컴포넌트 찾기에서 다른 페이지로 이동" / "빈 화면으로 앵커링"
   - 가이드: "가이드하게 된다면 저 쐬고 가죠, GDS에서. 그 위에 맨 최근 버전"
   - 맥락 분리 인식하나 결과: 시안에 옛 버전 섞임 ("최신화 된 것도 있고 안 된 것도 있고") 자인.

3. **"어렵지만 쓰려고" vs "직접 선택으로 회피" (B)**
   - "어렵지만 쓸려고 하는 편이에요. 다 읽어요. 만지거든요"
   - 동시: "그 직접 선택 때문에" 사용 어려움 호소 — 직접 선택 = 프로퍼티 우회 흔적.

4. **신규 제안 성공 경험 vs 기존 확장 니즈 강조 (A)**
   - 과거: 댓글 모듈 신규 제안해 패턴화 성공
   - 현재: "신규 컴포넌트보단 기존에 있는 거에 이런 걸 더해야 된다는 니즈가 앞으로 계속 생기지 않을까"
   - 본인 성공(신규) ≠ 미래 예측(확장). 시스템 성숙 단계 인식 변화로 해석 가능.

#### 한샘

1. **자유롭게 디자인 → 컴포넌트 교체 vs 시간 효율 모호 자인 (A)**
   - 워크플로우: "컴포넌트 사용 아예 생각 안 하고 자유롭게 … 나중에 다 교체하는 작업 따로"
   - 효율 평가: "시간적으로는 비유기 하는데 … 두 개 다 해놓고 비교가 낫다"
   - 비효율 자인 + 정당화 동거. 디자이너 자율성/완성도 욕구 vs 시스템 효율.

2. **인풋창 다양성 욕구 vs 일관성 가치 (B)**
   - 인터뷰어: "같은 인풋창이 일관성"
   - 본인: "다른 서비스인데 차이 줘야 하지 않나" / "판단력 많이 없는 상태"
   - 판단 기준 부재 직접 자인. 브랜딩·변주 욕구 + 학습 부담 인식 동시.

3. **디테치 "마음이 아프다" vs 시안 단계 실행 (B)**
   - "그때 마음이 아프죠" (디테치 시)
   - 동시: "시안 잡을 때만 하고 … 그냥 테스트해 보고 싶어서"
   - 정서적 거부 + 실행 공존.

4. **프로퍼티 많음 = 활용도 좋음 vs "다 눌러보는" 탐색 비효율 자인 (A)**
   - "활용할 수 있는 게 많아 가지고 좋다"
   - 동시: "찾는 과정이 조금 어렵긴 한데 … 다 눌러보는 거죠"
   - 호평(잠재력)과 탐색 brute force 동거. 단비·이수연과 동일 패턴.

5. **V1/V2 헤더 혼란: DS 책임 + 본인 책임 동시 자인 (A)**
   - DS 측 책임: "혼재되어 있으니까 헷갈리지 … 이모지로 다이아몬드 놓치긴 했는데"
   - 본인 측: "업데이트라는 업무가 후순위로 밀리잖아요 … 오랜만에 들어가면 V1이 깨 남아 있으니까"
   - DS-디자이너 공동 부채 자각.

6. **"제가 쓸 땐 편한데 기획자는 모름" (A)**
   - 본인 효율 인정 + 기획자 인지 부재 자인 동시
   - 디자이너만 DS 혜택 누리는 비대칭 명시.

### 약한 텐션 및 예외 판단 패턴

5점 이하 인물은 모순 분석의 주 대상으로 삼지 않는다. 문한나·손소의·김진아는 약한 텐션으로 유지하되, 개인의 자기충돌보다 시스템 신뢰·독립 라이브러리·평가 톤의 문제로 해석한다. 류선영·주선영·고유란은 파트리드 3인으로, 모순 점수가 낮거나 중간이어도 `예외 판단 패턴`으로 분류한다.

파트리드 3인의 공통점은 DS를 부정하는 것이 아니라, 업무 조건에 따라 기준을 나누는 데 있다. 류선영은 DS-first 원칙과 기능 불가능성 예외 기준이 일관되고, 주선영은 컴포넌트 수보다 정의와 옵션 범위를 중시한다. 고유란은 운영/선행 phase에 따라 DS 준수와 디테치를 나누며, 로컬 라이브러리 우려도 자기충돌보다 거버넌스 감각에 가깝다.

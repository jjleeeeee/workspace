# 팀리드 인터뷰 분석

> 소스: 팀리드 인터뷰 2건 (안정호, 김선영 / 2026.05.26)  
> 기준: `teamlead-interview-template.md`의 TH1-TH6 가설 검증  
> 역할: 기존 실무자 8명 + 파트리드 3명 인터뷰를 조직 판단 레이어에서 검증  
> 주의: 팀리드 2명은 일반 사용자 표본이 아니라 **조직 기준·책임 경계·우선순위 보정 레이어**로 해석한다.

## 1. Executive Summary

팀리드 인터뷰 2건은 기존 11명 인터뷰의 핵심 발견을 뒤집지 않는다. 대신 기존 발견을 더 선명한 조직 언어로 바꾼다.

가장 중요한 결론은 다음 5가지다.

1. **운영에서는 DS-first가 팀리드 레벨에서 더 강하게 확인된다.**
   - 김선영은 위버스 프로젝트에서 DS는 "일단 쓰는 것"이며, 매번 쓸 이유를 붙여야 한다면 시스템이라고 부르기 어렵다고 봤다.
   - 안정호도 운영 개선 화면에서는 가능하면 100%에 가깝게 DS를 적용하려 하며, 운영 개선 기준으로 CDS 완성도를 80-90% 수준으로 체감했다.

2. **선행에서는 두 팀리드의 판단이 갈린다.**
   - 김선영은 선행 단계에서 DS를 보지 않는다고 했다. 선행은 기존 시스템을 반복하는 단계가 아니라, 지금보다 앞선 방향과 DS 개선점을 찾는 단계이기 때문이다.
   - 안정호는 과거에는 후반 적용으로 생각했지만, 너무 늦게 DS를 적용하면 기획과 다시 싱크해야 하는 비용이 생겨 지금은 약 30% 시점, 즉 두 번째 시안부터는 CDS를 고려해야 한다고 봤다.

3. **예외 판단은 "깨도 된다"가 아니라 "어떤 책임을 갖고 깨는가"의 문제로 바뀐다.**
   - 김선영은 UX 목적이 컴포넌트로 커버되지 않을 때 예외가 가능하다고 보며, 디자이너가 문제를 설명하고 개선 제안을 할 수 있어야 한다고 했다.
   - 안정호는 일정, 현재안 훼손 정도, 사용성, 디자인 품질, DS로 풀 수 있는 가능성을 함께 본다. 억지로 끼워 넣어 사용성이 망가지면 커스텀 판단이 가능하다고 봤다.

4. **도메인 반복 패턴은 팀리드들도 관리 필요성을 인정하지만, DS팀 단독 관리 대상은 아니라고 본다.**
   - 김선영은 파운데이션은 공통으로 유지하되, 컴포넌트부터는 위버스샵·매거진·리스닝파티 등 도메인별 관리가 필요할 수 있다고 봤다.
   - 안정호는 미디어라이브·스팟처럼 패턴이 쌓이는 영역은 담당 서비스 디자이너가 모으되, 1/2팀이 함께 쓸 수 있는 공통 포맷이나 기준이 필요하다고 봤다.

5. **DS팀의 다음 과제는 컴포넌트 추가보다 개발 구조 맵핑, 사례 축적, 외부 협업자와의 공감대 형성에 가깝다.**
   - 김선영은 개발 상태 매트릭스를 단순한 "개발 완료 여부"가 아니라 디자인 구조와 개발 구조가 실제로 어떻게 맵핑되는지 파악하는 문제로 확장했다.
   - 안정호는 온보딩과 사례 축적을 우선으로 봤다. 슬랙에 흘러가는 사례는 기억되지 않으므로, 사용 목적과 케이스를 지속적으로 쌓는 방식이 필요하다고 했다.

**최종 리포트용 핵심 문장**

> PD들은 운영에서는 DS를 기본값으로 받아들이고 있지만, 선행·도메인 특화·개발 구조 불확실성 앞에서는 아직 조직 공통의 예외 판단 기준을 갖고 있지 않다.

## 2. 팀리드별 상세 읽기

### 김선영님

**팀/도메인 맥락**

- 프로덕트디자인팀 19명을 리드하며, 주요 도메인은 위버스·위버스샵·매거진 등이다.
- 위버스는 CDS를 주로 사용하지만, 위버스샵은 초기 버전 이후 업데이트가 잘 되지 않았고 기존 화면/레거시가 많다.
- 토큰은 전체 서피스에서 지키고 있다고 판단한다. 즉 `파운데이션 준수`와 `컴포넌트 준수`를 분리해서 본다.

**DS 기본 태도**

- 운영/위버스 프로젝트에서는 DS를 "일단 쓰는 것"으로 본다.
- 팀원이 커스텀을 쓰는 게 더 좋겠다고 하면 우선 컴포넌트를 쓰도록 안내한다.
- 다만 컴포넌트가 UX 목적을 담을 수 없거나 기획 의도를 커버하지 못하는 경우에는 커스텀이 가능하다고 본다.

**선행 판단**

- 선행에서는 DS를 아예 보지 않는다고 말한다.
- 이유는 DS로 선행을 하면 비슷한 디자인이 나올 확률이 높고, 선행은 오히려 DS의 문제점을 발견하고 개선할 수 있는 과정이기 때문이다.
- 단, 선행 결과물이 실제 프로젝트로 넘어가 개발/구축 단계에 들어가면 컴포넌트를 다시 보고 맵핑해야 한다.
- 신규 컴포넌트는 선행에서 바로 DS로 올리기보다, 운영해보고 반복성과 필요성을 판단해야 한다.

**예외 판단 기준**

- detach/커스텀 판단의 최우선 기준은 UX 목적이다.
- 기획 의도, 개발 구현 가능성, 재사용 가능성, 일정은 커뮤니케이션으로 풀 수 있지만, UX 목적 자체를 컴포넌트가 담지 못하면 대체가 필요하다고 본다.
- "DS를 잘 쓰는 능력"은 최대한 준수하는 것보다 **필요한 예외 판단을 세울 줄 아는 것**에 가깝다고 답했다.

**책임 경계**

- DS팀은 컴포넌트 자체의 밸런스와 완성도에 책임이 있다.
- 특정 맥락에 컴포넌트를 넣는 것은 도메인 디자이너의 몫이다.
- 도메인 디자이너는 "컴포넌트가 이래서 안 됐다"에서 멈추지 말고, 어떤 부분이 문제인지 설명하거나 개선 제안까지 할 수 있어야 한다.
- 팀리드/파트리드가 모든 레이어를 확인할 수 없으므로, 시스템을 알아서 조합하는 것은 개인 역량이라는 관점이 강하다.

**도메인 관리**

- 위버스샵은 CDS만으로 감당할 수 없는 커머스 특화 UX가 많다.
- 드롭다운, 라디오, 인풋필드, 토큰 같은 기본 요소는 CDS를 쓰되, 머치·주문·장바구니·상품상세 모듈 등은 도메인 별도 관리가 필요하다고 본다.
- 파운데이션은 공통으로 쓰되, 컴포넌트로 들어가면 도메인별 분리가 필요할 수 있다고 명확히 구분했다.
- 도메인 라이브러리는 도메인 담당자가 관리하는 것이 맞고, DS팀은 템플릿을 제공한 뒤 중간중간 싱크를 맞추는 방식이 현실적이라고 봤다.
- CDS 편입 기준은 하나의 도메인을 넘어 다른 도메인에서도 쓰이는가다. 다만 2개 도메인만으로 충분한지, 3개 정도는 되어야 하는지 고민이 남아 있다.

**DS팀 우선순위**

- 컴포넌트 추가와 Usage/사례집은 충분하다고 보고, 1순위로 개발 상태 매트릭스를 꼽았다.
- 이때 개발 상태 매트릭스는 단순히 "개발됨/안 됨"이 아니라, 디자인 구조와 개발 구조가 실제로 잘 맵핑되어 있는지 확인하는 문제다.
- 온보딩은 디자이너보다 기획자 같은 외부 협업자에게 더 필요하다고 봤다.
- 기획자가 컴포넌트 정책/말줄임/글줄 같은 기준을 함께 이해해야, 나중에 "이 컴포넌트 뭘 써야 할까요"나 뒤집기가 줄어든다고 본다.
- 챗봇보다는 웹페이지가 더 적합하다고 판단했다. 피그마는 처음 보는 사람이 찾기 어렵고, 초반에는 무엇을 물어봐야 할지도 모를 수 있기 때문이다.

**특이하게 중요한 발화**

- DS는 조건을 붙여 선택하는 도구라기보다 운영의 기본 규칙에 가깝다.
- 하지만 DS가 모든 도메인 컴포넌트를 품어야 한다는 뜻은 아니다.
- "파운데이션은 공통, 컴포넌트는 도메인별 분화 가능"이라는 경계가 가장 선명하다.

### 안정호님

**팀/도메인 맥락**

- 2팀은 브랜드디자인파트와 코어디자인파트가 함께 있고, 프로덕트 도메인은 스팟, 미디어라이브, 포스트 쓰기/360 모듈, 백스테이지, 어카운트, 젤리 페이먼트 등이다.
- DS를 많이 쓰는 영역은 미디어라이브의 일부 탭/전시 화면, 포스트 쓰기 최근 작업, 어카운트다.
- 백스테이지는 거의 쓰지 않고, 젤리 페이먼트도 레거시가 많다. 미디어라이브 엔드는 개발 레거시가 커서 단순히 화면만 DS로 바꾸기 어렵다.

**팀에서 들은 DS 고민**

- "커스텀을 안 하고 싶은데 할 수밖에 없다"는 고민을 들었다.
- 팀원들은 DS를 쓴 안과 적당히 커스텀한 안을 함께 가져오며 의견을 구한다.
- 이는 DS를 지켜야 한다는 인식은 있지만, 100% DS 안이 디자인적으로 아쉽거나 기능을 담기 어렵다는 신호다.

**조합 가능성 문제**

- 완전히 새로 만드는 케이스는 많지 않고, 대부분 조합으로 풀 수 있다고 본다.
- 그러나 어디까지 조합해도 되는지 기준이 모호하다.
- 리스트아이템 우측에 무엇을 넣을 수 있는지, 리스트아이템과 버튼을 병렬로 조합해도 되는지 같은 질문이 나온다.
- 즉 문제는 컴포넌트 부재만이 아니라 **조합 가능한 범위와 사용처의 불명확성**이다.

**선행 판단**

- 과거에는 선행 후반에 DS를 적용해도 된다고 봤다.
- 하지만 360 모듈, 어카운트 같은 작업에서 기획과 50% 이상 싱크한 뒤 CDS를 적용하려고 하니 화면이 바뀌고, 기획과 다시 싱크해야 하는 일이 생겼다.
- 그래서 현재는 디자인 가이드가 100이라고 할 때 약 30 정도, 최초 드래프트 후 두 번째 시안부터는 CDS를 조금씩 고려해야 한다고 본다.
- 다만 초기 시안은 여전히 자유롭게 잡아도 된다고 본다. 스팟처럼 비주얼/공연 정보 전달이 중요한 화면은 처음부터 DS에 묶이면 화면 목적이 흐려질 수 있기 때문이다.

**선행에서 어려운 이유**

- 선행에서 CDS 유지가 어려운 이유를 `조립 비용/컴포넌트 조합·레이아웃 불일치` 70%, `기획·아이디어 싱크` 30% 정도로 봤다.
- 특히 DS를 써야 한다는 근거가 "DS에 그렇게 정의되어 있다"뿐이면, 보안·법적 요구처럼 강한 기획/정책 이유 앞에서 설득하기 어렵다고 했다.

**운영 판단**

- 운영에서는 최근 최대한 100%에 가깝게 DS를 적용하려고 한다.
- 안 써진 부분이 있으면 같이 찾아보며 쓸 수 있는지 검토한다.
- 운영 개선 기준으로 CDS 완성도는 80-90% 정도로 체감한다.
- 다만 미디어라이브나 스팟처럼 이미지와 강조가 중요한 화면은 화면 특성을 살리기 어렵다고 봤다.

**예외 판단 기준**

- detach/커스텀 판단에서 일정이 가장 크게 작용한다고 했다.
- 어카운트처럼 한 번에 바꾸고 싶어도 개발 여력이 없어 기존 것을 수정하는 방식으로 가는 경우가 있다.
- 현재안을 크게 망가뜨리지 않으면서 DS를 쓸 수 있으면 DS를 쓰게 유도한다.
- 억지로 DS에 끼워 넣어 사용성이나 화면 목적이 제대로 들어가지 않을 것 같으면 커스텀을 허용한다.
- 판단이 어려우면 DS 담당자, 김선영님 등에게 묻고 결정한다.

**도메인 관리**

- 미디어라이브는 이미 파편화된 패턴과 라이브러리급 요소가 많아 정리하다가 멈춘 상태다.
- 스팟도 오픈 후 운영/고도화를 하면서 반복되는 요소를 보고 단비님과 정리해야겠다고 생각하고 있다.
- 도메인 라이브러리는 프로젝트 중간보다 운영하면서 반복성을 확인한 뒤 모으는 것이 맞다고 본다.
- 팀 단독 방식보다 1/2팀이 함께 쓸 수 있는 공통 포맷이나 기준이 필요하다고 봤다.
- 관리자는 담당 서비스 디자이너가 맡고, 담당자가 바뀌면 인수인계되어야 한다고 봤다.

**심리적 안전감**

- 팀원이 DS 안과 커스텀 안을 두 가지로 가져오는 것은 불안한 마음 때문이라고 해석했다.
- 그 불안은 "이래도 되나"와 "내가 더 좋은 DS 활용안을 못 찾은 건가"라는 찜찜함이다.
- 이를 풀기 위해 먼저 좋은 화면인지, 사용성과 디자인 품질이 괜찮은지 보고, 그다음 현재안을 크게 망가뜨리지 않으면서 DS를 활용할 방법을 같이 찾는다.
- DS에 얽매여 디자이너가 할 수 있는 표현을 못 하는 것도 경계한다.

**사례/온보딩**

- 안정호는 지속적인 온보딩과 사례 공유를 DS팀 우선순위로 봤다.
- 팀 안에서도 샘님/소의님처럼 시스템을 많이 적용한 사람이 배운 점을 공유하게 하려 한다.
- 하지만 슬랙에 올라간 사례는 흘러가면 기억되지 않는다.
- 사용 목적과 활용 사례를 어디에 수집할지, 내 화면에 맞지 않는 케이스를 어떻게 축적할지 고민이 있다.

**특이하게 중요한 발화**

- DS 적용은 너무 늦으면 다시 앞단으로 돌아가는 재싱크 비용을 만든다.
- "조합할 수 있는데도 조합 가능한지 모르는 상태"가 존재한다.
- DS를 지키는 것과 좋은 화면을 만드는 것 사이에 불안/타협/코칭이 함께 있다.

## 4. TH1-TH6 가설별 결과

### TH1. 선행 판단 기준

**기존 가설:** 선행 작업에서는 DS 준수보다 아이디어 전달력, 기획 싱크업, 디자인적 판단이 먼저 작동하고, DS는 후행 검증·정리 기준으로 밀린다.

**결과: 유지하되, "복귀 시점"을 추가해야 함**

김선영에게 선행은 DS를 적용하지 않는 구간이다. 선행은 시스템 안에서 비슷한 화면을 반복하는 것이 아니라, 앞으로 갈 방향을 보고 DS가 개선해야 할 지점을 발견하는 과정이다. 그래서 선행 결과물을 곧바로 DS에 넣기보다, 과제화되고 구축 단계에 들어갔을 때 맵핑과 업데이트가 시작된다.

안정호는 선행 초반의 자유도를 인정하지만, DS 적용이 너무 뒤로 밀리면 기획과 합의한 화면을 다시 바꾸게 되고 재싱크가 생긴다고 봤다. 그래서 초기 드래프트 후 두 번째 시안, 대략 30% 지점부터는 CDS를 조금씩 고려해야 한다고 판단이 바뀌었다.

**해석:** 선행은 `DS 미적용 허용`만으로 설명하면 부족하다. 팀리드 레벨에서는 "언제 DS로 돌아올 것인가"가 별도 의사결정 지점이다. 최종 PPT에서는 선행을 `자유 탐색 → DS 접점 검토 → 구축 단계 맵핑` 타임라인으로 그리는 것이 적절하다.

### TH2. 운영 판단 기준

**기존 가설:** 운영·기존 화면에서는 DS가 품질과 속도의 기본값으로 작동하지만, 최신 개발 상태와 플랫폼별 구현 여부가 불투명하면 팀 단위 신뢰가 흔들린다.

**결과: 유지 + 강화**

김선영은 운영에서 DS는 이유 없이 쓰는 기본값이라고 본다. 시스템을 쓸지 말지에 조건을 붙여야 한다면 시스템이라고 부르기 어렵다는 관점이다. 다만 컴포넌트가 UX 목적을 커버하지 못하면 디자이너가 예외 판단과 개선 제안을 해야 한다.

안정호도 운영에서는 최대한 DS를 적용하며, 안 써진 부분이 있으면 같이 찾아본다. 운영 개선 화면에서는 CDS 완성도를 80-90%로 평가했다. 그러나 미디어라이브/스팟처럼 이미지와 강조가 중요한 화면에서는 DS 적용이 어렵고, 화면 특성을 살리기 위해 일부 커스텀이 필요하다고 본다.

**해석:** 운영 DS-first는 강하게 유지된다. 다만 `설계 중심 운영 화면`과 `비주얼/콘텐츠 강조 화면`의 적용 난이도는 분리해야 한다.

### TH3. 도메인 반복 패턴 관리

**기존 가설:** 현재는 정식 도메인 라이브러리보다, CDS를 깨거나 조합해서 한 도메인 안에서 반복되는 패턴을 묶어 쓰는 초기 상태에 가깝다.

**결과: 유지 + 운영 모델 구체화**

김선영은 위버스샵을 대표 예시로 들었다. 토큰과 기본 컴포넌트는 CDS를 쓰지만, 커머스 특화 UX는 CDS가 감당하기 어렵다. 주문, 장바구니, 머치, 상품 상세 모듈은 도메인 라이브러리로 관리하는 것이 맞고, DS팀은 라이브러리 템플릿을 제공한 뒤 중간중간 싱크를 맞추는 역할이 적합하다.

안정호는 미디어라이브와 스팟에서 이미 패턴화된 요소가 많거나 생길 가능성이 있다고 했다. 다만 프로젝트 중간에 바로 라이브러리화하기보다, 운영하면서 반복되는 것을 보고 모으는 게 맞다. 방식은 2팀만 독자적으로 정하기보다 1/2팀이 함께 쓸 공통 포맷이나 기준이 필요하다고 봤다.

**해석:** 도메인 반복 패턴 관리의 현실적 모델은 `DS팀 템플릿 제공 → 도메인 담당자 관리 → 팀 간 주기적 싱크 → 2-3개 이상 도메인 반복 시 CDS 승격 검토`다.

### TH4. 일관성 vs 변별성 판단

**기존 가설:** 팀리드는 DS 일관성과 서비스별 변별성 사이의 최종 기준을 제공해야 하지만, 현재는 개인/파트 단위 판단으로 분산되어 있다.

**결과: 유지 + 세분화**

김선영은 일관성의 경계를 파운데이션과 기본 규칙에서 강하게 본다. 토큰은 공통으로 쓰고, DS는 운영의 기본값이다. 하지만 컴포넌트 단위로 내려오면 위버스샵/매거진/리스닝파티처럼 도메인 특화가 필요할 수 있다.

안정호는 스팟 사례에서 변별성의 필요를 구체적으로 설명했다. 다른 탭과 너무 비슷하면 공연 정보를 제공하는 화면 목적이 흐려지므로, 공연을 잘 보여줘야 하는 슬롯은 커스텀하고, 일반 운영 정보는 시스템을 쓰는 식으로 타협했다. 즉 변별성은 스타일 취향이 아니라 정보 전달 목적과 연결된다.

**해석:** 일관성 vs 변별성은 하나의 축이 아니라 `파운데이션 일관성`, `운영 컴포넌트 준수`, `도메인 목적 표현`, `비주얼 강조`로 나누어 봐야 한다.

### TH5. 심리적 안전감과 허락 구조

**기존 가설:** 실무자의 "혼날까봐", "소심함", "판단력이 없다"는 신호는 DS팀만의 문제가 아니라, 팀/파트 리더십의 허락 구조와도 연결될 수 있다.

**결과: 유지하되, 리드별 해석이 갈림**

안정호는 팀원이 DS 안과 커스텀 안을 함께 가져오는 것을 불안의 신호로 읽었다. 불안은 단순히 혼날까봐가 아니라, "이래도 되나"와 "내가 DS를 더 잘 조합할 수 있었는데 못 한 건가"라는 찜찜함에 가깝다. 리드는 먼저 좋은 화면인지 판단하고, 현재안을 크게 망가뜨리지 않으면서 DS를 활용할 수 있는 방법을 함께 찾는다.

김선영은 불안을 직접 코칭하는 방식보다, 디자이너가 갖춰야 할 역량으로 번역한다. 모든 레이어를 팀리드/파트리드가 확인할 수 없으므로, 디자이너가 시스템을 조합하고 예외를 설명할 수 있어야 한다. 리스닝파티처럼 처음부터 DS를 쓰지 않기로 결정한 경우에는 괜찮다고 안심시키되, 파운데이션/아이콘/아바타 등 가져다 쓸 수 있는 요소는 가져다 쓰게 한다.

**해석:** 팀리드 레벨에서 심리적 안전감은 두 방향으로 나타난다. 하나는 `같이 판단해주는 코칭`, 다른 하나는 `예외 판단을 개인 역량으로 요구하는 기준화`다. 최종 리포트에서는 이 둘을 충돌로 보기보다, 현재 조직이 동시에 요구하는 두 역할로 정리하는 것이 좋다.

### TH6. DS 투자 우선순위

**기존 가설:** 팀리드는 "DS를 더 확장할 것인가"보다 "어디까지 DS로 관리할 것인가"를 결정해야 하는 위치에 있다.

**결과: 유지 + 방향 선명화**

김선영은 컴포넌트 추가나 Usage/사례집보다 개발 상태 매트릭스와 기획팀 온보딩을 더 중요하게 봤다. 특히 디자인 구조와 개발 구조가 실제로 맵핑되어 있는지 알아야 운영 판단을 더 잘할 수 있다고 했다. 또한 기획자와 컴포넌트 정책을 함께 이해해야 나중에 뒤집는 일이 줄어든다고 봤다.

안정호는 온보딩과 사례 축적을 우선으로 봤다. 사용 목적과 케이스가 명확하지 않으면 조합 가능한데도 못 쓰거나, 억지로 끼워 넣다 커스텀으로 가게 된다. 사례가 슬랙에 흘러가면 기억되지 않으므로, 지속적으로 수집되는 공간이 필요하다.

**해석:** 팀리드의 우선순위는 `컴포넌트 양적 확대`보다 `판단 가능성 향상`에 가깝다. 개발 구조 맵핑, 사용 목적/사례 축적, 기획자와의 공감대가 핵심이다.

| 심리 신호 해석 | 예외 판단은 개인 역량. 단 명확히 안 쓰기로 한 경우 안심 제공 | 두 안을 가져오는 것 자체가 불안. 같이 판단하며 허들 낮춤 |
| DS팀 기대 | 개발 구조 맵핑, 기획 온보딩, 웹페이지 | 지속적 온보딩, 사례 축적, 사용 목적 명확화 |

## 6. 새로 추가된 발견

### 발견 1. 팀리드는 운영 DS-first를 "선택"이 아니라 "규칙"으로 본다

실무자 발화에서는 DS를 써야 하는지, 깨도 되는지의 고민이 반복됐지만, 팀리드 발화에서는 운영 기준이 더 단호하다. 특히 김선영은 DS를 쓸지 말지에 매번 이유가 필요하면 시스템이라고 부르기 어렵다고 보았다.

**PPT 차트 후보:** 운영 DS-first 강도: 실무자 / 파트리드 / 팀리드 레이어 비교.

### 발견 2. 선행의 핵심은 "DS를 안 쓰는가"가 아니라 "언제 돌아오는가"다

김선영은 선행에서 DS를 보지 않고, 안정호는 너무 늦게 보면 기획 재싱크가 커진다고 본다. 이는 기존 "선행은 자유롭다" 결론을 유지하되, 선행 안에서도 **DS 복귀 시점**이 별도 의사결정 지점임을 보여준다.

**PPT 차트 후보:** 선행 단계 타임라인: 자유 탐색 → 2차 시안/30% 지점 → 개발 합의 → 구축/DS 맵핑.

### 발견 3. DS 경계는 파운데이션과 컴포넌트에서 다르게 그어진다

김선영은 토큰/파운데이션은 공통으로 가져가되, 컴포넌트 단위에서는 도메인 분리가 필요할 수 있다고 했다. 이 관점은 기존 도메인 라이브러리 논의를 더 정교하게 만든다. "CDS로 관리할 것인가"가 아니라, **파운데이션은 공통으로 두고 컴포넌트부터 도메인화할 것인가**를 물어야 한다.

**PPT 차트 후보:** DS 경계 레이어: Foundation / Base components / Domain modules / Product-specific UI.

### 발견 4. 조합 가능성의 불명확성이 커스텀을 만든다

안정호 인터뷰에서 리스트아이템, 버튼, 타이틀헤더처럼 "써도 되는지 모르는 조합"이 반복됐다. 컴포넌트가 없는 것보다, 컴포넌트의 사용처와 조합 가능 범위가 불명확한 것이 커스텀 판단을 만든다.

**PPT 차트 후보:** 커스텀 원인 분해: 컴포넌트 없음 / 조합 가능성 모름 / 목적 불일치 / 일정.

### 발견 5. 도메인 반복 패턴의 현실적 운영안은 "DS팀 템플릿 + 도메인 담당 관리 + 주기적 싱크"다

두 팀리드 모두 도메인 담당자가 관리하는 방향을 말했지만, 완전 자율은 아니다. DS팀이 템플릿이나 예시 포맷을 제공하고, 팀 간 싱크를 맞추는 중간 구조가 필요하다.

**PPT 차트 후보:** 책임 분담 다이어그램: DS팀 / 팀리드 / 도메인 담당자.

### 발견 6. 개발 상태 가시성은 "개발 완료 여부"를 넘어 "구조 맵핑" 문제다

기존 11명 리포트는 V1/V2, 플랫폼별 개발 완료 여부를 중심으로 다뤘다. 김선영은 여기서 한 단계 더 나아가, 디자인 구조와 개발 구조가 실제로 어떻게 맵핑되어 있는지 알아야 더 좋은 운영 판단을 할 수 있다고 했다.

**PPT 차트 후보:** 디자인 구조 ↔ 개발 구조 맵핑 상태 매트릭스.

### 발견 7. 챗봇보다 웹페이지/사례 저장소가 더 현실적일 수 있다

문한나는 LLM 코드 챗봇을 잘 활용했지만, 김선영은 챗봇보다 웹페이지가 낫다고 봤다. 안정호도 사례가 슬랙에 흘러가면 기억되지 않기 때문에 축적되는 공간이 필요하다고 말했다. 이는 AI 도구 확산보다, 사용자가 처음에 무엇을 물어봐야 할지 모르는 상태를 해결하는 정보 구조가 먼저일 수 있음을 보여준다.

**PPT 차트 후보:** 정보 접근 채널 우선순위: Figma / Slack / Chatbot / Web page.

### 발견 8. 기획자 온보딩은 "DS 소개"가 아니라 "뒤집기 방지" 장치다

김선영은 디자이너 대상 온보딩보다 기획자와의 공감대 형성을 더 중요하게 봤다. 기획자가 말줄임, 글줄, 정책성 요소 같은 기준을 함께 이해하고 피드백해야 나중에 전체를 뒤집는 일이 줄어든다. 이는 기존 "기획자 설득 비용" 발견을 팀리드 레벨에서 강화한다.

**PPT 차트 후보:** 기획자 이해도 → 컴포넌트 합의 → 후반 재작업 감소 흐름도.

## 7. 최종 리포트 반영 방향

### 메인 메시지 보정

기존:

> PD들이 DS를 지키지 않는 것이 아니라, 업무 맥락에 따라 DS를 다르게 해석하고 있다.

팀리드 반영 후:

> PD들은 운영에서는 DS를 기본값으로 받아들이고 있지만, 선행·도메인 특화·개발 구조 불확실성 앞에서는 아직 조직 공통의 예외 판단 기준을 갖고 있지 않다.

더 세밀한 버전:

> DS는 운영 화면의 기본 규칙으로 자리 잡았지만, 선행에서 언제 DS로 복귀할지, 도메인 특화 패턴을 누가 관리할지, 개발 구조와 얼마나 맞물리는지는 아직 리드 개인의 판단과 비공식 조율에 의존한다.

### 차트 업데이트

| 기존 차트 | 업데이트 |
|---|---|
| 인터뷰 대상 구성 | 총 13명: 실무자 8명 + 파트리드 3명 + 팀리드 2명 |
| DS 사용 방식 분포 | 팀리드 보정 레이어 추가: 운영 DS-first / 선행 복귀 시점 |
| 주요 신호 강도 | 개발 상태 불투명 → 개발 구조 맵핑 불투명으로 확장 |
| AI 도구 실사용 1/11 | 챗봇보다 웹페이지/사례 저장소 선호 신호 추가 |
| 도메인 라이브러리 공백 | DS팀 템플릿 + 도메인 담당 관리 + 주기적 싱크 모델 추가 |
| 심리적 안전감 | 불안 신호 + 개인 예외 판단 역량 요구를 함께 표시 |

### 리더들이 결정해야 할 쟁점 보강

1. 운영에서는 DS-first를 조직 원칙으로 명시할 것인가?
2. 선행 작업에서 DS로 돌아오는 시점을 언제로 볼 것인가?
3. 파운데이션/기본 컴포넌트/도메인 모듈의 책임 경계를 어떻게 나눌 것인가?
4. 도메인 반복 패턴은 누가 관리하고, 어떤 템플릿으로 기록할 것인가?
5. 개발 상태 표시는 완료 여부만 보여줄 것인가, 디자인-개발 구조 맵핑까지 보여줄 것인가?
6. 온보딩은 디자이너 대상인가, 기획/개발까지 포함한 전사 언어화인가?
7. 사례는 슬랙에 흘려보낼 것인가, 검색 가능한 저장소로 축적할 것인가?

## 8. 추천 반영 우선순위

이 문서는 처방형 리포트가 아니라 현황 파악 리포트의 보강 자료로 사용한다. 최종 PPT에는 모든 내용을 넣기보다 다음 6개를 우선 반영한다.

1. **총 13명 조사 구조**
2. **운영 DS-first vs 선행 복귀 시점 타임라인**
3. **예외 판단 기준: UX 목적 / 일정 / 도메인 특수성 / 개발 구조 / 사용처 불명확성**
4. **DS 경계 레이어: 파운데이션 공통 / 컴포넌트 도메인화 / 도메인 모듈 관리**
5. **도메인 반복 패턴 관리 모델: DS 템플릿 + 도메인 담당 관리 + 주기적 싱크**
6. **DS팀 우선순위: 컴포넌트 추가보다 개발 구조 맵핑·기획 온보딩·사례 저장소**

# CDS 인식 및 사용 현황 리포트

> 문서 성격: Confluence 위키형 현황 리포트  
> 작성 기준: 2026.05.29  
> 원본 기준 리포트: `final-report/cds-interview-current-state-report.md`  
> 조사 범위: 실무자 8명 + 파트리드 3명 + 팀리드 2명 + Head of Design 1명  
> 사용자 인터뷰 정량 기준: 실무자, 파트리드, 팀리드까지의 13명  
> HoD 문서 처리: 정량 집계에서 제외하고 리더십 맥락으로만 분리

## 1. 이 문서의 목적

요약: 이 문서는 개인 평가가 아니라 CDS 사용 현황과 예외 판단 맥락을 정리한 현황 리포트다.

이 조사는 CDS를 누가 잘 지키고 누가 지키지 않았는지 평가하기 위한 것이 아니다.

목적은 PD들이 실제 업무 안에서 CDS를 어디까지 기준으로 삼고, 어느 순간부터 예외로 판단하는지 확인하는 것이다. 인터뷰 발화에서는 개인 태도보다 업무 맥락, 개발 연결 상태, 도메인 특성, 예외 판단 기준의 차이가 반복적으로 확인된다.

이 문서는 실행 방향을 확정하는 문서가 아니라, 현재 상태를 이해하기 위한 현황 리포트다. 최종 공유용 문서로 읽기 쉽도록 정리했지만, 근거와 해석 범위는 원본 기준 리포트의 SoT를 따른다.

## 2. 먼저 읽는 요약

요약: CDS는 운영 업무에서 효용이 확인되지만, 선행/도메인/개발 싱크 구간에서는 다시 해석되는 지점이 있다.

CDS는 운영 화면의 기본값에 가깝게 언급된다. 특히 이미 CDS 구조와 맞는 운영/개선 업무에서는 속도, 일관성, 개발 커뮤니케이션 효용이 반복적으로 확인된다.

다만 CDS는 모든 화면을 같은 방식으로 흡수하는 단일 운영 체계로 언급되지는 않는다. 선행이 구축으로 넘어가는 시점, 도메인 특화 요구가 커지는 시점, 개발 구현 상태가 보이지 않는 시점, 기존 화면이 레거시화되는 시점에서는 CDS가 다시 해석되는 발화가 확인된다.

따라서 현재 반복적으로 확인되는 쟁점은 `CDS 준수 여부` 자체보다 `업무 흐름의 어느 전환점에서 CDS가 기준으로 들어오는가`, `어떤 조건에서 예외로 판단되는가`, `그 예외가 어떤 맥락으로 남는가`에 가깝다.

## 3. 조사 범위와 숫자 읽는 법

요약: 사용자 인터뷰 정량은 13명 기준이며, HoD는 정량에서 제외한 별도 리더십 맥락으로 읽는다.

| 구분 | 인원 | 이 문서에서의 역할 |
| --- | ---: | --- |
| 실무자 | 8명 | 실제 사용 방식, 불안, 우회, 컴포넌트 이슈 확인 |
| 파트리드 | 3명 | 파트원 검토, 예외 판단, 도메인/운영 맥락 보정 |
| 팀리드 | 2명 | 팀 차원의 운영 기준과 선행/운영 차이 확인 |
| Head of Design | 1명 | 디자인실 차원의 판단 기준과 책임 경계 맥락 확인 |

숫자는 다음 기준으로 읽는다.

- `11명 기준`은 실무자 8명 + 파트리드 3명의 사용 방식 보조 분석에만 사용한다.
- `13명 기준`은 실무자, 파트리드, 팀리드까지의 사용자 인터뷰 정량 분석에 사용한다.
- `14명 전체`는 전체 조사 범위를 말할 때만 사용한다.
- HoD 문서는 사용자 인터뷰 정량 수치에 합산하지 않는다.
- 반복 신호는 `언급 인원`이다. 실제 사용률, 준수율, 만족도, 로그 데이터가 아니다.

## 4. 정성/정량으로 확인된 반복 신호

요약: 수치는 사용률이 아니라 발화 반복 빈도이며, 효용 신호와 판단 비용 신호가 함께 확인된다.

이 섹션의 숫자는 설문 통계가 아니다. 인터뷰에서 같은 신호가 몇 명에게 반복됐는지 정리한 것이다.

따라서 `사용률`, `준수율`, `만족도`처럼 읽으면 안 된다. 더 정확한 표현은 `언급 인원`, `반복 신호`, `자기보고`, `맥락 확인`이다.

| 파트리드그룹 | 3명 | 파트원 검토와 예외 판단 방식을 보정하는 기준 |

### 4.2 CDS 사용 진입 방식

실무자그룹과 파트리드그룹 11명 기준으로 보면, CDS 사용은 하나의 방식으로 수렴하지 않는다.

| 사용 진입 방식 | 인원/비율 | 해석 |
| --- | ---: | --- |
| 강한 CDS-first | 3명, 약 27% | 빈 화면 또는 초기 구성 단계부터 CDS 컴포넌트를 먼저 가져와 조립한다. |
| 기존 화면/작업 파일 출발 후 CDS 재확인 | 2명, 약 18% | 운영 화면에서 기존 파일이나 화면을 참고하되, 최신 CDS나 코드 기준을 다시 확인한다. |
| 자유 시안 후 CDS 맞춤/디테치 | 4명, 약 36% | 선행/신규 화면에서 먼저 자유롭게 만들고, 이후 CDS와 맞추거나 일부 분리한다. |
| 별도 라이브러리 또는 도메인 패턴 출발 | 2명, 약 18% | CDS보다 도메인에서 반복되는 패턴을 먼저 출발점으로 삼는다. |

운영 화면에서는 CDS를 처음부터 조립하는 방식뿐 아니라, 기존 작업물을 출발점으로 삼고 이후 CDS 기준을 재확인하는 방식도 함께 나타난다.

### 4.3 주요 반복 신호

실무자그룹과 파트리드그룹 11명 기준에서 반복된 신호는 다음과 같다.

| 반복 신호 | 인원/비율 | 해석 |
| --- | ---: | --- |
| 운영/반복 업무에서 시간 단축 체감 | 8명, 약 73% | 운영/반복 업무에서는 시간 절감 효과가 반복적으로 언급됐다. 이미 CDS와 연결된 화면에서 특히 강하게 나타난다. |
| 디테치를 자주 하지는 않는다는 응답 | 8명, 약 73% | 디테치가 적다는 응답은 CDS를 벗어나는 상황이 없다는 뜻은 아니다. 기존 화면 참고, 자유 시안 후 재정리, 도메인 패턴 활용처럼 디테치 외 방식의 예외 판단이 함께 나타난다. |
| 리스트아이템 관련 어려움 또는 논의 | 7명, 약 64% | 특정 컴포넌트가 단순히 부족한 것이 아니라, 정의와 적용 범위가 흔들리는 대표 사례로 언급된다. |
| 개발 구현 상태 불투명 언급 | 6명, 약 55% | Figma 기준과 실제 코드 구현 상태가 연결되지 않으면 CDS 신뢰가 약해진다는 발화가 확인된다. |
| 기획자 통역 비용 언급 | 5명, 약 45% | CDS 판단이 디자이너 내부 언어에 머물면, 기획자와의 합의 비용이 커진다는 맥락이 나타난다. |
| LLM 코드 챗봇 실사용 | 1명, 약 9% | 보조 도구 사용 사례는 확인되지만, 조직적 사용 채널로 보기는 어렵다. |

이 표에서는 효용 신호와 비용 신호가 동시에 보인다. CDS는 실제 업무에서 효용을 만들고 있지만, 그 효용이 강하게 나타나는 구간과 다시 해석 비용이 생기는 구간이 분리되어 있다.

### 4.4 컴포넌트 언급 빈도

#### 4.4.1 인터뷰 발화 기준 언급 빈도

아래 숫자는 사용량이 아니라 `언급 인원`이다. 어떤 컴포넌트를 많이 썼다는 뜻이 아니라, 인터뷰에서 이슈나 사례로 반복 등장했다는 뜻에 가깝다.

| 컴포넌트/영역 | 언급 인원/비율 | 해석 |
| --- | ---: | --- |
| 리스트아이템 | 7명, 약 64% | 정보 구조, 썸네일, 메타 정보, 상태값 등 변형 압력이 큰 영역이다. |
| 버튼 | 7명, 약 64% | 가장 기본적인 컴포넌트지만, 상황별 위계와 도메인 톤에서 논의가 반복된다. |
| 아바타/갓배지 | 6명, 약 55% | 프로필, 권한, 보상, 정체성 표현과 연결되어 도메인성이 강하다. |
| 텍스트필드 | 4명, 약 36% | 입력 상태, 에러, 개발 구현 상태와 연결되는 기본 컴포넌트다. |
| 바텀시트 | 4명, 약 36% | 플랫폼 패턴, 화면 목적, 개발 구현과 함께 판단된다. |
| 툴팁 | 4명, 약 36% | 설명 방식과 노출 조건이 기획 의도와 강하게 연결된다. |
| 타이틀 영역 | 4명, 약 36% | 콘텐츠/도메인 맥락에 따라 구조와 표현이 달라진다. |
| 테이블/리더보드 | 3명, 약 27% | 데이터 밀도와 도메인 특화 표현이 강한 영역이다. |

컴포넌트 언급 빈도는 `무엇을 더 만들어야 하는가`보다 `어디에서 정의와 예외 판단이 반복되는가`를 보여주는 자료로 읽는 편이 안전하다.

이 항목들은 개인의 준수 의지보다 운영 체계의 빈틈과 연결되어 발화된다. 특히 기획, 개발, 외주, 웹 맥락이 들어오면 CDS는 단순 컴포넌트 선택 문제가 아니라 협업 기준 문제로 확장된다.

### 4.6 개발 구현과 레거시 관련 신호

개발 구현 상태와 레거시 맥락은 CDS 신뢰와 함께 언급된다.

| 신호 | 인원/비율 | 해석 |
| --- | ---: | --- |
| 개발 구현 상태 불투명 | 6명, 약 55% | Figma에 있는 것과 실제 구현된 것이 연결되어 보이지 않으면 판단이 사람에게 돌아간다는 맥락이 나타난다. |
| V1/V2 또는 마이그레이션 혼재 | 4명, 약 36% | 같은 운영 화면이어도 과거 구조와 현재 CDS 기준이 섞여 있을 수 있다. |
| 개발 적용 불일치 | 3명, 약 27% | 디자인 파일 기준으로 맞춰도 개발 결과가 다르게 구현될 수 있다는 불안이 남는다. |

이 신호는 `운영은 DS-first로 안정적이다`라는 문장을 보정한다. 이미 CDS와 잘 연결된 운영은 안정적이지만, 레거시나 플랫폼별 구현 차이가 있는 운영은 다시 예외 판단이 발생한다.

### 4.7 팀리드와 HoD는 수치보다 맥락 레이어로 읽는다

팀리드그룹 2명은 13명 사용자 인터뷰 정량에 포함한다. 다만 사용 방식 분포를 해석할 때는 팀 운영 관점의 맥락 레이어로도 읽는다.

HoD 문서 1개는 사용자 인터뷰 정량에 합산하지 않는다. 이 문서는 디자인실 차원의 판단 기준과 책임 경계가 어떻게 표현되는지 확인하는 별도 리더십 맥락이다.

팀리드그룹 인터뷰에서는 기존 발견을 그대로 유지할 항목과 표현 수위를 보정할 항목이 함께 확인됐다.

| 기존 발견 대비 상태 | 항목 수 | 의미 |
| --- | ---: | --- |
| 유지/강화 | 3개 | 운영에서 CDS가 기본값으로 작동한다는 점, 개발 구현 상태가 중요하다는 점, 선행과 운영의 판단 방식이 다르다는 점은 유지된다. |
| 보정 | 3개 | 운영이라고 항상 안정적인 것은 아니며, 레거시와 전환점이 들어오면 다시 판단 비용이 생긴다. |
| 구체화/확장 | 2개 | 도메인 반복 패턴과 책임 경계는 단순 컴포넌트 추가보다 운영 기준 문제에 가깝다. |
| 표현 수위 조정 | 1개 | CDS 효용을 전반적 체감으로만 말하면 실제보다 넓게 읽힐 수 있다. 효용은 특정 구간에서 더 강하게 나타난다. |

HoD 답변은 해결책을 확정하는 자료가 아니라, 디자인실 차원의 판단 기준을 확인하는 자료로 사용한다. 따라서 HoD는 비율로 환산하지 않고 `운영/선행 구분`, `예외 판단`, `책임 경계`를 보강하는 근거로만 다룬다.

### 4.8 숫자로 읽을 수 있는 현황

1. CDS 효용은 반복적으로 언급된다. 다만 모든 업무에서 동일하게 나타나는 효용이 아니라, 이미 CDS와 연결된 운영/반복 업무에서 특히 강하게 나타난다.

2. 현재 반복되는 어려움은 `CDS를 모른다`보다 `언제 예외로 볼지`, `그 예외가 어디에 남는지`에 가깝게 발화된다.

3. 컴포넌트 이슈는 단순 누락보다 정의, 적용 범위, 도메인 변형, 개발 구현 상태와 연결되어 있다.

4. 개발 구현 상태가 보이지 않으면 디자이너는 CDS를 기준으로 결정하기보다 기존 화면, 개발자 확인, 별도 패턴에 다시 의존하게 된다는 발화가 확인된다.

5. LLM 코드 챗봇은 가능성으로 언급되지만, 현재 기준으로는 조직적 해결 채널이라고 보기 어렵다.

### 4.9 숫자로 과장하면 안 되는 것

- 이 숫자는 CDS 준수율이 아니다.
- 이 숫자는 만족도 점수가 아니다.
- 이 숫자는 실제 사용 로그가 아니다.
- HoD는 사용 방식과 컴포넌트 언급 빈도에 합산하지 않는다.
- 팀리드그룹은 사용자 인터뷰 13명에는 포함하되, 해석에서는 팀 운영 관점의 보정 자료로 함께 읽는다.
- 비율은 이해를 돕기 위한 보조 표기이며, 결론은 정성 근거와 함께 읽어야 한다.

## 5. 자료 사용 원칙

요약: HoD 질문지는 리더십 맥락으로, 자동 회의록은 액션 아이템 보조 근거로만 제한해 사용한다.

### HoD 자료 사용

HoD 문서는 사용자 인터뷰 이후의 후속 질문 문서로 분리해 사용한다.

이 문서에는 운영/기존 화면, 선행/신규 화면, 예외 판단, 도메인 특화 패턴, 공통 품질 기준 등에 대한 리더십 맥락이 정리되어 있다. 다만 사용자 인터뷰 13명의 언급 빈도나 사용 방식 분포에는 포함하지 않는다.

자동 회의록은 실제 대화의 흐름상 해결안, 즉석 아이디어, 특정 사례, 내부 논의가 섞여 있다. 따라서 이 위키형 리포트에서는 HoD 질문지 답변만 공식 근거로 다룬다.

단, `리더십 후속 검토 액션 아이템` 섹션에는 사용자가 별도로 지정한 HoD 자동 회의록의 to-do성 발화를 보조로 반영했다. 이 내용은 본문 정량이나 핵심 발견의 근거로 합산하지 않는다.

## 6. 핵심 발견

요약: CDS는 운영 효율을 만들지만, 예외/도메인/개발 구현 상태에 따라 적용 방식이 달라진다.

### 6.1 CDS는 운영 업무에서 기본값에 가깝게 작동한다

운영/기존 화면에서는 CDS가 기본값에 가깝게 작동한다. 기존 컴포넌트를 조합하거나 치환하는 작업에서는 속도와 일관성의 효용이 확인된다. 토큰, 다크모드, 기본 상태값처럼 반복 품질을 매번 새로 판단하지 않아도 되는 점도 강점으로 언급된다.

하지만 운영이라고 모두 같은 상태는 아니다. 이미 CDS 구조와 맞는 운영은 안정적이지만, 오래된 화면, V1/V2 혼재, 웹/외주/하이브리드 구조, 선행 결과물이 흡수되지 않은 화면에서는 운영 업무 안에서도 다시 예외 판단이 발생한다.

정리:

- CDS는 운영 업무에서 효용을 만든다는 발화가 반복된다.
- 운영 효용은 이미 CDS와 연결된 화면에서 더 강하게 확인된다.
- 레거시나 개발 구조가 다른 화면에서는 별도 맵핑 비용이 언급된다.

### 6.2 선행은 CDS 미적용 여부보다 복귀 시점이 쟁점으로 언급된다

선행/신규 화면에서는 초반부터 CDS를 강제하면 탐색 폭이 좁아질 수 있다는 맥락이 확인된다. HoD 답변에서도 초기 아이데이션 단계에서는 CDS에 갇히지 않는 쪽의 맥락이 제시된다.

다만 방향성과 핵심 플로우가 정해지는 시점에 CDS를 다시 확인하는 흐름이 쟁점으로 언급된다. 선행 결과가 확정되어 구축으로 넘어갈 때 CDS에 흡수되지 않으면, 운영 진입과 동시에 예외 또는 레거시 후보가 된다는 맥락이 있다.

정리:

- 선행은 무조건 CDS 밖이라는 뜻으로 발화되지 않는다.
- 초반 탐색과 확정/구축 전 확인 시점이 구분되어 언급된다.
- 핵심은 `처음부터 썼는가`보다 `언제 돌아오는가`에 가깝다.

### 6.3 예외는 취향보다 업무 조건과 연결되어 발생한다

인터뷰에서 확인된 예외 사유는 대체로 업무 조건과 연결된다.

| 예외 사유 | 의미 |
| --- | --- |
| 기능 한계 | CDS 컴포넌트로 화면 목적이나 기능을 담기 어려움 |
| 정의 불일치 | 컴포넌트 이름은 같지만 실제 쓰임이 다름 |
| 개발 미구현 | Figma에는 있으나 코드 구현 상태가 다르거나 불명확함 |
| 도메인 톤 | 커머스, 음악, 공연, 콘텐츠 등 도메인 맥락이 강함 |
| 선행 검증 | 아직 반복성이나 운영 가능성이 확정되지 않음 |
| 레거시 구조 | 기존 화면과 과거 구현 히스토리 때문에 즉시 치환이 어려움 |

예외 자체보다 예외가 왜 발생했는지, 일회성인지 반복 패턴인지, 도메인에 남는지 CDS로 재흡수될 수 있는지가 반복 쟁점으로 나타난다.

### 6.4 예외 기록은 HoD 문서에서 리더십 맥락으로 제시된다

HoD 문서에는 예외를 인정할 때 남길 수 있는 정보로 다음 세 가지가 제시된다.

- 왜: 기존 CDS로 대응할 수 없었던 기획적/비즈니스적 배경
- 무엇을: 어떤 컴포넌트를 어떻게 변형하거나 커스텀했는지
- 언제까지: 이번 릴리즈 한정인지, 정식 패턴으로 유지할 것인지

이 내용은 사용자 인터뷰 정량에서 나온 수치가 아니라, HoD 문서에 정리된 리더십 맥락이다. 사용자 인터뷰에서는 예외 판단의 불안, 디테치 확인의 어려움, 도메인 패턴의 산발성이 함께 확인된다.

정리:

- 예외는 금지 대상이라기보다 판단 맥락으로 발화된다.
- 예외 기록의 형식은 사용자 인터뷰 정량 수치가 아니라 HoD 문서의 리더십 맥락이다.
- 사용자 인터뷰에서는 예외 사유와 영향 범위를 확인하기 어렵다는 발화가 함께 나타난다.

### 6.5 도메인 반복 패턴은 이미 일부 생기고 있다

도메인 안에서 반복되는 변형 패턴은 이미 생기고 있다. 이 현상을 곧바로 특정 운영 방식으로 해석하기보다는, 반복 범위와 공통화 가능성을 나눠 읽는 편이 안전하다.

인터뷰와 HoD 문서에서 함께 확인되는 질문은 다음과 같다.

- 특정 도메인 안에서만 반복되는가?
- 여러 도메인으로 전이될 가능성이 있는가?
- 공통 접근성, 다국어, 다크모드, 기본 인터랙션과 연결되는가?
- 코드 구조상 공통화가 가능한가?
- 도메인 담당자가 관리하는 편이 더 현실적인가?

HoD 답변 기준으로 보면, 한 도메인 안에서만 쓰이는 패턴과 여러 도메인으로 전이될 수 있는 패턴을 구분하는 맥락이 제시된다.

### 6.6 공통 기준과 도메인 자율 영역이 구분되어 언급된다

HoD 답변은 공통 유지 기준과 도메인 자율 영역을 구분한다.

공통 유지 기준으로 언급된 항목:

- 브랜드 정체성
- 글로벌 접근성
- 다국어 대응 레이아웃
- 다크모드
- 최하위 스펙 디바이스에서의 최소 정보 보장
- 기본 인터랙션 컴포넌트

도메인 자율 영역으로 언급된 항목:

- 도메인 특화 데이터 시각화
- 마케팅/프로모션성 배너 지면
- 아티스트별 개성을 살려야 하는 특화 슬롯
- 도메인 맥락이 강한 모듈

이 구분은 사용자 인터뷰의 도메인 라이브러리/로컬 패턴 발화와 연결해 읽을 수 있다. 다만 이 문서에서는 운영 방식으로 확정하지 않고, 리더십 맥락에서 제시된 분류로만 다룬다.

### 6.7 개발 구현 상태가 보이지 않으면 CDS 판단이 흔들린다

디자이너가 Figma에서 CDS 컴포넌트를 확인하더라도, 실제 iOS/Android/Web에 구현되어 있는지 알기 어려우면 다시 개발자 확인이나 기존 화면 복사로 돌아가게 된다는 발화가 확인된다.

HoD 답변에서도 플랫폼별 Production Ready 상태나 코드 컴포넌트 싱크 상태가 보이면 판단에 도움이 된다는 맥락이 제시된다.

정리:

- CDS는 Figma 컴포넌트만으로 작동한다고 보기 어렵다.
- 실제 개발 구현 상태가 보이는지 여부가 디자이너 판단과 연결된다.
- 개발 연결 상태는 CDS 신뢰와 함께 반복적으로 언급된다.

### 6.8 교육보다 판단 기준과 사례가 언급되는 맥락이 확인된다

HoD 답변에서는 교육과 워크샵을 더 진행하는 방향보다 판단 기준, 사례, 운영 프로세스 정립을 언급하는 맥락이 확인된다.

사용자 인터뷰에서도 단순히 “모른다”보다 “어디까지 바꿔도 되는지”, “이 케이스가 CDS에 맞는지”, “개발에서 실제로 구현되어 있는지”, “다른 사례가 있는지”를 확인하려는 발화가 반복된다.

이 내용은 교육 효과를 평가하는 결론이 아니라, 판단 순간에 필요한 정보가 무엇인지에 대한 발화상 단서로 읽는다.

## 7. HoD 문서에서 확인된 리더십 맥락

요약: HoD 문서에서는 운영/선행 구분, 예외 판단, 도메인 패턴, 공통 품질 기준이 리더십 맥락으로 제시된다.

| 맥락 | HoD 문서에서 확인된 내용 |
| --- | --- |
| 운영/기존 화면 | 기획 킥오프 단계부터 CDS 기준이 들어오는 흐름이 제시됨 |
| 선행/신규 화면 | 초기 탐색은 열어두되, 방향 확정 후 CDS 맵핑을 확인하는 흐름이 제시됨 |
| 예외 사유 | 왜, 무엇을, 언제까지를 남기는 맥락이 제시됨 |
| 도메인 패턴 | 도메인 내부 반복과 공통 스펙 후보를 구분하는 맥락이 제시됨 |
| 공통 품질 | 접근성, 다국어, 다크모드, 기본 인터랙션 등이 공통 기준으로 언급됨 |
| 판단 기준/사례 | 교육 자체보다 판단 기준과 사례 맥락이 강조됨 |

이 표는 사용자 인터뷰 정량 분석 결과가 아니다. HoD 문서에서 확인된 리더십 맥락을 별도로 분리한 것이다.

## 8. 리스크와 한계

요약: 이 문서는 인터뷰와 질문지 기반의 정성 리포트이며, 실제 로그나 전체 조직 일반화 자료는 아니다.

- 이 문서는 인터뷰 발화와 HoD 질문지 답변을 기반으로 한다. 실제 사용 로그, 개발 반영 로그, 컴포넌트 사용량 데이터는 포함하지 않는다.
- 11명 기준 분석은 실무자와 파트리드의 사용 방식 보조 분석이다. 13명 사용자 인터뷰 전체 수치와 섞어 읽지 않는다.
- 팀리드그룹은 사용자 인터뷰 13명에 포함되지만, 팀 운영 관점의 발화가 많아 정성 해석에서 별도 레이어로 함께 읽는다.
- HoD 문서는 사용자 인터뷰 정량에 포함하지 않는다.
- 자동 회의록 특성상 문장 품질과 발화 맥락의 밀도가 문서별로 다를 수 있다.
- 이 문서는 현황 파악을 목적으로 하며, 실행 방향이나 운영 방식을 확정하지 않는다.

## 9. 별첨: 인터뷰 후 익명 보조 설문

요약: 13명 전원 응답 설문은 인터뷰 발화를 보조하는 자료이며, 공통 기준 효용은 높고 예외 판단 자신감은 상대적으로 낮게 나타난다.

인터뷰 이후 별도 Google Forms 설문으로 13명이 응답했다. 전체 사용자 인터뷰 13명 기준 응답률은 100.0%다.

이 설문은 인터뷰 발화를 대체하는 근거가 아니라, 인터뷰 이후 CDS 인식과 판단 자신감을 보조적으로 확인하기 위한 자료다. 응답 수가 작으므로 전체 구성원의 인식으로 일반화하지 않는다.

### 9.2 보조 설문에서 확인되는 신호

- 공통 컴포넌트/규칙의 작업 효율, 개발 협업 효율, 일관된 디자인 역량 중요성은 높은 동의가 확인된다.
- 시안 시작 시 공통 컴포넌트를 먼저 확인하는지에 대해서는 4~5점과 3점이 나뉜다.
- 예외 상황에서 변형 수준을 스스로 판단할 수 있는지에 대해서는 평균이 상대적으로 낮고, 1~2점 응답도 확인된다.
- 창의적 판단이 규칙보다 우선한다는 문항은 3점 응답이 많고 높은 동의는 제한적이다.
- 별첨 문항에서는 인터뷰 분위기와 솔직한 발화 가능성에 대해 높은 동의가 확인된다.

자유 응답에서는 DS팀에 대한 감사와 응원, 인터뷰 시간이 짧았다는 의견, 화면 목적에 따라 창의성과 일관성 판단이 나뉜다는 의견이 확인된다. 이 자유 응답은 개인 식별 없이 요약만 반영한다.

## 10. 리더십 후속 검토 액션 아이템

요약: 리더십 후속 검토는 적용/완화 기준, 예외 히스토리, 도메인 책임, 개발 가시성, 아카이브 방식으로 압축된다.

아래 항목은 인터뷰, HoD 질문지, 별도 HoD 자동 회의록에서 확인된 리더십 레이어의 후속 검토 항목이다. 실행 확정안이 아니라, 현황 리포트 이후 별도 논의에서 확인할 액션 아이템으로 분리한다.

1. CDS 적용/완화 기준 정리

2. 예외 히스토리 기록 방식 정리

3. 도메인 패턴 관리 기준 확인

4. 로컬 라이브러리 책임 범위 정리

5. 공통 CDS 재흡수 판단 루프 정리

6. 커스텀 UI 사용성/접근성 체크 기준 정리

7. 개발 구현 상태와 QA 역할 범위 확인

8. 사례/히스토리 검색 가능한 아카이브 방식 검토

## 11. Adversarial Review 기록

요약: 본문 정량과 HoD/설문/자동 회의록 근거가 섞이지 않도록 최종 검토했다.

| 검토 질문 | 결과 |
| --- | --- |
| 최종본이 `final-report`의 SoT 범위를 넘어 새로운 주장을 추가했는가? | 본문 현황/정량/핵심 발견은 기존 범위를 유지함. 단, 액션 아이템에는 사용자가 별도 지정한 HoD 자동 회의록의 to-do성 발화를 보조 반영했다고 명시 |
| HoD 문서가 사용자 인터뷰 정량에 섞였는가? | 섞지 않음. HoD는 별도 리더십 맥락으로 분리 |
| 팀리드그룹이 HoD와 합쳐졌는가? | 합치지 않음. 팀리드는 사용자 인터뷰 13명에 포함하되 정성 해석에서 별도 레이어로 설명 |
| 권고성 문장이 남아 있는가? | 실행 항목, 위키 표현 가이드, 리더십 논의 목록을 제거하고 관찰/맥락 표현으로 낮춤 |

검토 결과, 가장 주의할 지점은 HoD 문서의 기준 문장이 운영 방향으로 읽히는 부분이었다. 따라서 해당 내용은 “운영 원칙”이 아니라 “HoD 문서에서 확인된 리더십 맥락”으로 분리했다.

# [파트장 그룹] 검토자·코치 시점 인터뷰

> **⭐ 유의사항**
> Chord 기반 본인 실무 경험과 더불어, **파트원 디자인을 검토·피드백하는 경험**을 함께 떠올려 답변 부탁드립니다.
> 본인 실무 / 파트원 검토 — 두 시점을 자유롭게 오가도 됩니다.

## Overview

| 항목 | 내용 |
|---|---|
| 목적 | DS(CDS)가 파트장의 **검토·승인·코칭** 의사결정에서 어떻게 작동하는지, 본인 실무 시점과 어떻게 다른지 이해 |
| 대상 | 디자인 파트장 3명 |
| 형식 | 대면 1:1, 40~45분 |
| 선행 | 실무자 8인 인터뷰(2026.05.06–05.14) 인사이트 기반 후속 |

## 검증할 가설 (파트장 전용)

> 실무자 인터뷰의 H1~H6 + 신규 발견 4가지(권력 비대칭, 개발 상태 가시성, 도메인 라이브러리 거버넌스, 일관성 vs 정체성)에서 도출된 **파트장 시점의 후속 가설**입니다.

1. **PH1. 검토 루브릭** — 파트장은 파트원 시안 검토 시 DS를 명시적 기준으로 사용한다 (또는 케이스별 직관에 의존)
2. **PH2. 변주 통과 기준** — 파트원의 detach/예외를 통과시키는 본인 내 기준이 일관적이다
3. **PH3. 자기검열 인지** — 파트장은 파트원의 "혼날까봐" 자기검열을 인지하고 있다 (또는 본인이 압박원으로 작동한다)
4. **PH4. 거버넌스 게이트** — 도메인 라이브러리 승인·매핑·CDS 승격에서 파트장이 게이트키퍼 역할을 한다
5. **PH5. 번역자 역할** — 파트장은 DS팀과 파트원 사이의 번역·완충 채널로 기능한다
6. **PH6. 일관성 vs 변별성** — 충돌 상황에서 파트장의 판단 기준은 도메인 성격에 따라 결정된다

## 인터뷰 구조

| 구간 | 내용 | 시간 |
|---|---|---|
| 도입 및 워밍업 | 파트 규모, 실무/검토 시간 비중, 검토 형식 | 3분 |
| 본인 실무 워크플로우 | 시작 패턴, 최근 detach 사례 (압축) | 5분 |
| 검토자 시점 | 검토 순서, 변주 통과/되돌림 기준, 신뢰성 확인 | 14분 |
| 코칭·거버넌스 | 자기검열 감지, 도메인 라이브러리 결정권, 온보딩 | 12분 |
| 협업·DS팀 관계 | DS팀 채널, 제안 추진, 개발/기획 개입 | 8분 |
| 마무리 | DS팀 요청, 자주 하는 코멘트 | 2분 |

## 질문지

### 1부. 도입 및 워밍업 (아이스브레이킹)

1. 현재 담당하시는 도메인/파트와 파트원 규모를 간단히 소개해 주세요.
2. 하루 업무 중 **본인 실무 vs 파트원 검토** 비중은 대략 어떻게 되시나요?
3. 파트원 디자인 검토는 보통 어떤 형식으로 진행하시나요? (스탠드업 / 온디맨드 / 주간 리뷰 / 1:1 등)

---

### 2부. 본인 실무 워크플로우 (압축) (Action)

> ℹ️ 목적: 파트장 본인의 시안 작성 패턴을 짧게 확인합니다. (검토 시점과의 대비 기준선)

4. 새 화면을 시작할 때 본인은 어느 패턴에 가까우신가요? — **CDS 먼저 조립 / 자유 시안 후 DS 워싱 / 처음부터 별도 라이브러리** 중에서. 사유까지 말씀해 주세요.
5. 본인이 최근에 detach 하거나 예외처리를 만든 사례가 있으시면, 어떤 UX 목적이었는지 짧게 알려주세요.

---

### 3부. 검토자 시점: 어떻게 보시는가? (Review)

> ℹ️ 목적: 파트원 시안 검토 시 DS가 명시적 루브릭으로 작동하는지, 변주 허용 기준이 일관적인지 확인합니다.

6. [검토 순서] 파트원 시안을 처음 받았을 때, **DS 관련해서 어디부터 보시나요?** 실제 순서를 떠올려 말씀해 주세요.
   - 예: 컴포넌트 매칭 여부 → 토큰 → 변주 흔적 → 플랫폼 일관성 등
7. [통과 기준] **"이건 detach 했어야 한다"** 또는 **"이 변주는 정당하다"** 라고 판단하는 본인만의 신호가 있다면 무엇인가요?
8. [최근 사례] 최근 검토 중 파트원 변주를 **통과시킨 사례 1개**, **되돌리거나 다시 잡아준 사례 1개** 떠올려 주세요. 각각 어떤 판단이었나요?
9. [신뢰성 확인] 파트원이 가져온 컴포넌트가 **최신/개발 완료 상태인지** 본인이 따로 확인하시나요? 한다면 어떤 방식으로 하시나요?
   - (참고: 실무자 인터뷰에서 "기존 화면이 업데이트 안 됐을 수 있어 코드 파일 재확인" 패턴이 다수 관찰됨)
10. [V1/V2·플랫폼 혼재] 파트원이 "iOS에는 V2가 없어요" 같은 이슈를 가져오면 보통 어떻게 결정하시나요?
11. [기획자 압박 코칭] 파트원이 "기획자가 detach해서 바꾸자고 한다"라는 이슈를 가져오면 어떻게 코칭하시나요?

---

### 4부. 코칭·거버넌스: 누가 결정하는가? (Coaching & Governance)

> ℹ️ 목적: 파트원의 자기검열·도메인 라이브러리 거버넌스·일관성 충돌에서 파트장의 역할 위치를 확인합니다.

12. [자기검열 감지] 파트원이 **"이렇게 만들어도 될까요? DS팀에 혼나지 않을까요?"** 같이 물어본 적이 있나요?
   - 있으면: "그때 어떻게 반응하셨어요?"
   - 없으면: "만약 그런 말을 들으면 어떻게 하실 것 같으세요?"
13. [본인 영향] 파트장이라는 위치가 파트원에게 **"허락자"처럼 작동**한다고 느끼신 적이 있나요? 그때 어떤 장면이었나요?
14. [도메인 라이브러리 승인] 파트원이 "도메인 전용 컴포넌트/라이브러리를 만들고 싶다"고 제안하면, **승인 결정은 누가** 하나요?
   - 본인 단독 / DS팀과 합의 / 파트원 자율 / 별도 프로세스
15. [CDS 승격 판단] 도메인 라이브러리에서 만든 컴포넌트 중 **CDS로 올릴 만한 것**은 어떤 기준으로 식별하시나요?
16. [일관성 vs 변별성] 파트원이 **"다 비슷비슷해 보여서 변별성을 주고 싶다"** 고 하면 어떻게 답하시나요? 그 판단 기준이 도메인에 따라 달라지나요?
17. [온보딩] 신규 파트원이 들어왔을 때 **DS 학습 가이드**를 어떻게 제공하시나요? 자주 막히는 지점이 있다면?

---

### 5부. 협업·DS팀 관계: 어디에 서 있는가? (Collaboration)

> ℹ️ 목적: 파트장이 DS팀과 파트원 사이의 번역·완충 채널로 기능하는지 확인합니다.

18. [DS팀 채널] DS팀과 직접 소통하시는 빈도와 주요 주제는 무엇인가요? 본인이 **파트 대표로 의견을 전달**하는 경우가 많나요?
19. [제안 추진] 파트원이 가져온 신규 컴포넌트/개선 제안을 DS팀으로 올릴 때, 본인이 **필터링하거나 다듬어서** 전달하시나요? 아니면 파트원이 직접 가게 하시나요?
20. [개발자 협업] 디자인 핸드오프 이후 개발자와의 합의 자리에 **본인이 직접 개입**하는 경우는 어떤 주제일 때인가요?
21. [기획자 협업] 기획자가 DS를 모른 채 변경 요청할 때, **파트원을 대신해 본인이 직접 설득**에 들어가는 경우가 있나요? 어떤 상황이었나요?

---

### 마무리

22. DS팀에 **한 가지만 요청**할 수 있다면 무엇을 요청하시겠어요?
23. 파트원 검토 시 가장 자주 하시는 **DS 관련 코멘트** 한 문장은?

## 인터뷰어 운영 가이드

### 시작 전 멘트

> ℹ️ "오늘은 파트장님이 DS를 잘 쓰고 있는지 평가하는 자리가 아니에요. 본인 실무뿐 아니라 **파트원의 디자인을 검토하실 때** DS가 어디에서 기준으로 작동하고, 어디에서 막히는지 알고 싶습니다. 답변이 본인 시점인지 파트원 시점인지 헷갈리면 제가 되묻겠습니다."

### 핵심 원칙

| 원칙 | 운영 메모 |
|---|---|
| 평가가 아니라 관찰 | "맞다/틀리다"보다 실제 순서와 판단 기준을 듣는다. |
| "왜"보다 "어떻게" | "왜 그렇게 했나요?"보다 "그때 어떤 순서로 결정하셨어요?"라고 묻는다. |
| 사례 요청 | 추상적 답이 나오면 "최근 사례로 기억나는 게 있으세요?"라고 되묻는다. |
| 변주는 중립적으로 다룬다 | detach나 규격 외 작업을 문제 행동으로 몰지 않는다. |
| 협업 장면을 구체화한다 | 기획자/개발자/파트원과 어떤 문장·화면·수치로 대화했는지 묻는다. |
| **(추가) 시점 명확화** | 답변이 "본인 실무"인지 "파트원 검토"인지 모호하면 "그건 본인 시안이었나요, 파트원 시안이었나요?"로 되묻는다. |
| **(추가) 권력 비대칭은 우회로** | "혼난다", "허락"처럼 민감한 단어는 파트장에게 직접 묻지 않는다. "그런 분위기 느낀 적 있나요?" 정도로 우회한다. |

### 기록 체크포인트

- ☐ 본인 시작 패턴 (CDS 먼저 / 자유 시안 후 워싱 / 별도 라이브러리)
- ☐ 본인 최근 detach 사례와 UX 목적
- ☐ 파트원 시안 **검토 순서**와 최우선 점검 항목
- ☐ 변주 **통과 / 되돌림** 결정 신호
- ☐ 컴포넌트 신뢰성(개발 상태·V1/V2·플랫폼) 확인 행동
- ☐ 파트원 **자기검열 신호** 감지/대응 사례
- ☐ 파트원에게 한 **기획자 압박 코칭** 멘트
- ☐ 도메인 라이브러리 **승인·CDS 승격** 결정권 위치
- ☐ **일관성 vs 변별성** 충돌 시 판단 기준
- ☐ 신규 파트원 **DS 온보딩** 방식과 막히는 지점
- ☐ DS팀과의 **소통 빈도·주제·방향** (대표 전달 여부)
- ☐ 파트원 제안의 **필터/번역** 행동
- ☐ 개발자/기획자 **직접 개입** 주제와 빈도

### 실무자 인터뷰와의 교차 검증 포인트

- 실무자에서 발견된 **"혼날까봐"(단비·유솔·한샘)** → 파트장은 이 신호를 인지하는가? (Q12, Q13)
- 실무자에서 발견된 **개발 상태 가시성 부재** → 파트장 검토에서도 동일 부담을 지는가? (Q9, Q10)
- 실무자에서 발견된 **도메인 라이브러리 거버넌스 공백** → 파트장 시점에서 누가 채워야 한다고 보는가? (Q14, Q15)
- 실무자(한샘)의 **"일관성 vs 단조로움" 정체성 충돌** → 파트장은 어느 쪽으로 판단을 유도하는가? (Q16)
- 실무자(한나)의 **챗봇 = 사람한테 묻기 부담 대체** → 파트장 채널의 심리적 안전감 평가 (Q12, Q13, Q18)

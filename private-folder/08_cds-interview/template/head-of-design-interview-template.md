# [Head of Design] 디자인 의사결정 기준·CDS 개입·예외 운영 인터뷰

> **유의사항**
> 이번 인터뷰는 Head of Design이 CDS 컴포넌트를 얼마나 상세히 알고 있는지 확인하는 자리가 아니다.
> 13명 인터뷰에서 드러난 실무자·파트리드·팀리드의 판단 기준을 바탕으로, 디자인 의사결정에서 **어떤 기준이 먼저 작동하고, CDS는 어느 시점에 개입하며, 예외는 어떤 조건에서 허용되는지** 확인한다.

## Overview

| 항목 | 내용 |
|---|---|
| 목적 | 화면을 판단할 때 CDS가 언제 기준으로 들어오고, 언제 예외로 빠지며, 그 판단을 누가 책임져야 하는지 확인 |
| 대상 | Head of Design 1명 |
| 형식 | 대면 1:1, 45분 |
| 선행 | 실무자 8명 + 파트리드 3명 + 팀리드 2명, 총 13명 인터뷰 인사이트 기반 후속 |
| 활용 | 최종 PPT의 `현황 → 판단 흐름 → 리더들이 검토할 운영 쟁점` 구조에 반영 |

## 인터뷰 맥락

| 레이어 | 현재까지 확인된 신호 | Head of Design에게 확인할 것 |
|---|---|---|
| 실무자 8명 | 운영에서는 CDS를 쓰려 하지만, 선행·신규·웹·도메인 맥락에서 예외 판단이 많아짐 | 어떤 상황에서 CDS보다 화면 목적·도메인·개발 조건을 먼저 보는가 |
| 파트리드 3명 | 예외를 감정이 아니라 기획 의도, 기능 가능성, 정의 일치 여부로 번역함 | 이 판단 순서를 공통 기준으로 참고할 수 있는가 |
| 팀리드 2명 | 운영에서는 CDS-first가 강하지만, 선행 복귀 시점과 도메인 책임은 추가 판단 필요 | 업무 단계별로 CDS 개입 시점을 어떻게 나눌 것인가 |
| DS팀 | 컴포넌트 제작, 사용 기준 운영, 예외 상담 역할이 섞여 있음 | DS팀이 판단 기준을 제공할 범위와 리더십이 책임질 범위 |

## 데이터 기반 가설

> 아래 가설은 실장님에게 그대로 읽어주기보다, 질문 설계의 기준으로 사용한다.

### HD1. 화면 의사결정의 우선순위

**화면을 결정할 때 UX 목적, 기획 의도, 일정, 개발 가능성, 일관성 중 무엇이 먼저 작동하는지 조직 차원의 판단 순서가 필요하다.**

- 근거: 기존 인터뷰에서 CDS를 벗어나는 이유는 단순 취향보다 기능 한계, 기획 의도, 개발 제약, 도메인 톤, 선행 검증으로 설명됐다.
- 리드 보강: 파트리드는 먼저 기획 적합성과 기능 가능성을 보고, 그다음 코드 컴포넌트 사용 여부를 확인하는 흐름을 보였다.
- 확인 포인트: Head of Design은 화면 목적과 CDS 기준이 충돌할 때 어떤 판단 순서를 기대하는지 확인한다.

### HD2. CDS 개입 시점

**운영/기존 화면에서는 CDS가 초기에 개입하지만, 선행/신규 화면에서는 특정 체크포인트에서 개입하는 흐름이 더 현실적일 수 있다.**

- 근거: 운영 화면에서는 CDS-first가 강하지만, 선행에서는 자유 시안 후 CDS 워싱, 두 번째 시안 이후 고려, 구축 전 맵핑 같은 패턴이 나왔다.
- 팀리드 보강: 김선영은 선행에서는 CDS를 보지 않는다고 했고, 안정호는 너무 늦게 적용하면 기획 재싱크 비용이 생긴다고 봤다.
- 확인 포인트: phase별로 CDS가 처음부터 들어와야 하는지, 중간 점검으로 들어와야 하는지, 개발 전 확인으로 들어와야 하는지 확인한다.

### HD3. 예외 판단 기준

**CDS를 그대로 쓰기 어렵다고 판단되는 조건은 취향이 아니라 기능·정의·도메인·개발 제약으로 설명되어야 한다.**

- 근거: 기능을 담을 수 없음, 컴포넌트 정의와 실제 쓰임 불일치, 개발 미구현, 브랜딩/도메인 톤, 선행 검증이 반복됐다.
- 리드 보강: 팀리드들은 예외를 무조건 금지하지 않지만, UX 목적·일정·사용성·현재안 훼손 정도를 함께 본다.
- 확인 포인트: 어떤 조건이면 예외로 인정하고, 어떤 조건이면 CDS 안에서 다시 풀어야 하는지 확인한다.

### HD4. 예외 처리 방식

**허용된 예외는 승인에서 끝나지 않고, 기록·재검토·도메인 패턴화·CDS 흡수 중 어디로 갈지 정해져야 한다.**

- 근거: 현재는 예외 승인 기준, 기록 위치, 재사용 여부, CDS 승격 기준, 도메인 담당자 책임이 약하다는 신호가 반복됐다.
- 리스크: 예외가 기록되지 않으면 합리적 판단도 시간이 지나 화면 파편화로 남는다.
- 확인 포인트: 예외를 개인 판단으로 끝낼지, 조직이 추적 가능한 운영 자산으로 다룰지 확인한다.

### HD5. 책임 경계

**실무자, 파트리드, 팀리드, DS팀, Head of Design이 각각 어떤 판단을 맡아야 하는지 경계가 필요하다.**

- 근거: 파트리드와 팀리드는 완충 장치로 작동하지만, 최종 판단은 개인 경험과 도메인 이해에 의존하는 부분이 크다.
- 현재 리스크: 모든 예외가 DS팀으로 몰리면 병목이 되고, 모든 예외를 개인에게 맡기면 기준 차이가 커진다.
- 확인 포인트: 판단 기준 제공, 예외 승인, 기록 관리, 도메인 패턴 관리의 책임 분기를 확인한다.

### HD6. 조직 운영 우선순위

**다음 단계는 컴포넌트 추가보다 판단 기준, 사례, 개발 구조 맵핑, 기획자 합의가 우선일 수 있다.**

- 근거: 팀리드 인터뷰에서 컴포넌트 양적 확대보다 개발 구조 맵핑, 사용 목적/사례 축적, 기획자 온보딩이 더 중요하게 나왔다.
- 현재 신호: CDS는 디자이너 내부 도구로는 작동하지만, 기획자와 개발자까지 포함한 의사결정 언어로는 아직 약하다.
- 확인 포인트: 조직 차원에서 DS팀과 리더십이 어디에 먼저 투자해야 하는지 확인한다.

## 질문지

### 1부. 디자인 의사결정의 우선순위

1. 화면이나 제품 방향을 결정할 때, 실장님은 어떤 기준을 가장 먼저 보시나요?
   - UX 목적 / 기획 의도 / 브랜드·도메인 톤 / 일관성 / 개발 효율 / 일정 / 재사용 가능성
2. 화면 목적과 일관성이 충돌할 때, 어떤 기준으로 우선순위를 잡는 것이 적절하다고 보시나요?
3. 기획 의도는 분명하지만 CDS 컴포넌트로는 잘 담기지 않는 경우, 어떤 순서로 판단해야 한다고 보시나요?
4. 개발 효율은 높지만 화면 목적이나 전달력이 약해지는 경우, 어느 수준까지 CDS 안에서 조정해야 한다고 보시나요?
5. 이번 리포트에서는 "누가 CDS를 안 지켰나"보다, "어떤 상황에서 CDS를 기준으로 삼고 어떤 상황에서는 예외로 판단하는가"를 보려고 하는데요. 이 방향이 적절하다고 보시나요?

### 2부. CDS가 개입해야 하는 시점

6. 운영/기존 화면에서는 CDS가 어느 시점부터 기준으로 들어와야 한다고 보시나요?
   - 기획 초기 / 와이어 단계 / 첫 시안 / 디자인 확정 전 / 개발 핸드오프 전
7. 선행/신규 화면에서는 CDS를 처음부터 고려해야 할까요, 아니면 방향이 잡힌 뒤 체크포인트로 들어오는 것이 적절할까요?
8. 선행 결과물이 실제 구축으로 넘어갈 때, 재싱크 비용을 줄이기 위해 어느 시점부터 CDS 맵핑을 필수로 봐야 할까요?
   - 2차 시안 / 기획 합의 전 / 개발 핸드오프 전 / 구축 착수 전 / 운영 반복 확인 후
9. "운영은 초기에 CDS 개입, 선행은 복귀 시점 관리"라는 프레임이 디자인실 기준으로 적절한가요?

### 3부. CDS를 벗어나는 예외 판단 기준

10. 화면 목적상 CDS를 그대로 쓰기 어렵다고 판단되면, 어떤 조건을 확인한 뒤 예외로 인정해야 하나요?
    - 기능을 담을 수 없음 / 컴포넌트 정의와 쓰임이 다름 / 개발 미구현 / 도메인 톤이 중요함 / 일정 제약 / 사용성이 나빠짐
11. 반대로 "이 정도는 예외가 아니라 CDS 안에서 다시 풀어야 한다"고 보는 조건은 무엇인가요?
12. "CDS를 그대로 쓰기 어렵다"는 의견이 나왔을 때, 그 이유가 사용자 이해나 기능 전달에 꼭 필요한 것인지, 아니면 화면 차별화를 위한 것인지 어떤 기준으로 확인하면 좋을까요?
13. 같은 화면에서 일부는 CDS를 쓰고 일부는 커스텀해야 할 때, 어떤 기준으로 경계를 나누면 좋을까요?

### 4부. 예외 처리의 승인·기록·흡수

14. 예외가 필요하다고 판단됐을 때, 어디까지는 실무자/파트리드가 결정하고 어디서부터 팀리드나 DS팀 확인이 필요하다고 보시나요?
15. 예외를 인정할 때 최소한 남겨야 하는 정보는 무엇인가요?
    - 예외 사유 / 사용 화면 / 대체한 CDS 컴포넌트 / 개발 구현 방식 / 재사용 가능성 / 재검토 시점
16. 예외로 판단한 내용을 팀이나 도메인 안에서 다시 확인할 수 있게 한다면, 어디에 가볍게 남기는 것이 현실적이라고 보시나요?
    - 작업 파일 안 메모 / 도메인 라이브러리 / 팀 문서 / 리뷰 회의록 / 별도 기록 없이 작업 맥락 안에 표시
17. 같은 예외가 반복되면, 어느 시점부터 도메인 패턴으로 관리하거나 CDS 흡수 후보로 봐야 할까요?
18. 예외가 기록되지 않고 화면마다 쌓이는 상황은 어느 수준부터 디자인실 리스크로 보아야 할까요?

### 5부. 도메인 반복 패턴과 책임 경계

19. 도메인 안에서 CDS를 조합하거나 변형한 패턴이 반복될 때, 누가 가장 먼저 알아차리고 관리해야 한다고 보시나요?
    - 도메인 담당자 / 파트리드 / 팀리드 / DS팀 / 공동 관리
20. 반복되는 패턴이 한 도메인 안에서만 쓰이면 도메인 담당자가 관리하고, 여러 도메인에서 같이 쓰이면 DS팀과 함께 봐야 한다고 볼 수 있을까요? 그 경계를 나누는 기준은 무엇이어야 할까요?
21. 공통으로 꼭 유지해야 하는 기준과, 도메인별로 다르게 가져가도 되는 영역은 어디서 나뉜다고 보시나요?
22. 도메인 자율성을 인정하더라도 절대 흔들리면 안 되는 공통 기준은 무엇인가요?

### 6부. 기획·개발 조건이 의사결정에 미치는 영향

23. 기획 단계에서 CDS 기준을 고려하지 않은 요구가 들어오면, 후반에 어떤 문제가 가장 크게 생긴다고 보시나요?
    - 디테치 요청 증가 / 디자인 재작업 / 개발 일정 영향 / 플랫폼별 불일치 / 기획 재논의
24. 기획자가 CDS 컴포넌트 이름까지 알 필요는 없더라도, 기획 단계에서 미리 알아야 하는 기준은 무엇이라고 보시나요?
    - 말줄임 / 글줄 수 / 버튼·CTA 배치 / 입력값 제약 / 개발 비용 / 플랫폼별 제약
25. 디자이너가 "이 CDS 컴포넌트가 실제 iOS/Android/Web에 구현되어 있는지" 바로 알기 어렵다면, CDS에서 어떤 정보까지 보여주는 게 판단에 도움이 될까요?
    - 구현 여부 / 플랫폼별 차이 / V1·V2 상태 / 실제 사용 화면 / 개발 구조와의 연결 방식

### 7부. 리더십이 정해야 할 운영 원칙

26. 앞에서 이야기한 내용 중, 리더십 차원에서 가장 먼저 이야기해봐야 할 쟁점은 무엇이라고 보시나요?
    - CDS가 들어와야 하는 시점 / 예외를 인정하는 기준 / 도메인 반복 패턴 관리 / 기획·개발과 맞춰야 할 기준
27. DS팀이 앞으로 더 집중해야 하는 일은 무엇이라고 보시나요?
    - 컴포넌트 추가 / 판단 기준 정리 / Usage·사례 축적 / 개발 구조 맵핑 / 도메인 패턴 관리 기준 / 기획자 합의 자료 / 예외 기록 체계

## 인터뷰어 운영 가이드

### 시작 전 멘트

> "오늘은 실장님이 CDS 컴포넌트를 얼마나 상세히 알고 계신지 확인하는 자리가 아닙니다. 앞선 13명 인터뷰에서 운영 화면에서는 CDS가 기본값으로 쓰이지만, 선행·도메인 특화·개발 구조 불확실성 앞에서는 예외 판단이 개인과 리드 경험에 많이 의존한다는 신호가 나왔습니다. 오늘은 'CDS를 지켜야 하나'보다, 디자인 의사결정 안에서 CDS가 언제 기준으로 들어오고 어떤 조건에서 예외로 빠져야 하는지 여쭤보고 싶습니다."

### 핵심 원칙

| 원칙 | 운영 메모 |
|---|---|
| CDS 평가보다 의사결정 기준 | "CDS를 어떻게 보시나요?"보다 "화면을 결정할 때 무엇을 먼저 보시나요?"로 묻는다. |
| 찬반보다 개입 시점 | "CDS를 써야 하나요?"보다 "어느 시점에 CDS가 기준으로 들어와야 하나요?"로 묻는다. |
| 예외 허용보다 예외 처리 | "깨도 되나요?"보다 "예외가 필요하면 어떤 조건을 확인하고 어디에 남겨야 하나요?"로 묻는다. |
| 선행/운영을 계속 분리 | 답변이 모호하면 "그건 선행 기준인가요, 운영 기준인가요?"라고 되묻는다. |
| 책임 분기를 명확히 | 실무자, 파트리드, 팀리드, DS팀, Head of Design이 각각 무엇을 판단하는지 나눈다. |
| 최종 리포트 문장으로 회수 | 답변 말미에 "이걸 리포트에는 어떤 의사결정 원칙으로 남기면 좋을까요?"라고 확인한다. |

### 기록 체크포인트

- ☐ 화면 의사결정에서 우선하는 기준
- ☐ 운영/기존 화면에서 CDS 개입 시점
- ☐ 선행/신규 화면에서 CDS 복귀 시점
- ☐ 예외 판단 기준
- ☐ 예외 처리 방식
- ☐ 예외 기록 위치와 반복 예외 흡수 기준
- ☐ 도메인 반복 패턴 책임 주체
- ☐ 파운데이션/컴포넌트/도메인 모듈의 책임 경계
- ☐ 기획자 합의가 필요한 정책
- ☐ 개발 구현 상태·구조 맵핑의 운영 포함 여부
- ☐ 실무자/파트리드/팀리드/DS팀/Head of Design 책임 분기
- ☐ 최종 리포트에 남길 리더 의사결정 질문

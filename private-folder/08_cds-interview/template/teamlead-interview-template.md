# [팀리드 그룹] 조직 판단·우선순위 시점 인터뷰

> **유의사항**
> 이번 인터뷰는 팀리드가 DS를 직접 얼마나 잘 쓰는지 확인하는 자리가 아니다.
> 11명 인터뷰에서 드러난 실무자·파트리드의 판단 기준을 바탕으로, 팀리드가 **어떤 기준으로 DS 적용 범위, 예외, 도메인 반복 패턴 관리, 선행/운영 우선순위**를 결정하는지 확인한다.

## Overview

| 항목 | 내용 |
|---|---|
| 목적 | DS(CDS)가 팀 단위의 품질 기준, 속도, 창의성, 도메인 자율성, 거버넌스 판단에서 어떻게 작동하는지 이해 |
| 대상 | 팀리드 2명 |
| 형식 | 대면 1:1, 45분 |
| 선행 | 실무자 8인 + 파트리드 3인 인터뷰(2026.05.06–05.21) 인사이트 기반 후속 |

## 대상 및 팀원 매핑

| 팀리드 | 팀원 | 팀 내 주요 신호 |
|---|---|---|
| 안정호 | 주선영, 이창규, 김단비, 이유솔, 손소의, 한샘 | 선행/웹/BDS 사각지대, 자유 시안 후 DS 워싱, 자기검열, 도메인 반복 패턴 관리, 일관성 vs 변별성 충돌 |
| 김선영 | 고유란, 류선영, 김진아, 이수연, 문한나 | 운영/기존 도메인 DS 준수, 파트리드 예외 판단, 신규/독립 라이브러리, 개발 상태 가시성, 사례집·LLM 코드 챗봇 수요 |

## 데이터 기반 가설

> 아래 가설은 팀리드에게 그대로 읽어주기보다, 질문 설계의 기준으로 사용한다.

### TH1. 선행 판단 기준

**선행 작업에서는 DS 준수보다 아이디어 전달력, 기획 싱크업, 디자인적 판단이 먼저 작동하고, DS는 후행 검증·정리 기준으로 밀린다.**

- 근거: 이창규는 선행 디자인을 먼저 만든 뒤 시스템에 맞춰 워싱한다고 했고, 한샘은 복잡한 과제에서 컴포넌트 사용을 생각하지 않고 자유롭게 시안 후 교체한다고 했다.
- 파트리드 보강: 주선영은 선행/BDS 맥락에서 자유 시안과 조합을 더 허용했고, 고유란은 운영은 최대한 지키지만 선행에서는 요구사항이 맞지 않으면 거의 깬다고 했다.
- 확인 포인트: 팀리드는 선행에서 DS를 어디까지 요구하는가? 창의성/전달력과 개발 효율이 충돌할 때 어느 기준을 먼저 보는가?

### TH2. 운영 판단 기준

**운영·기존 화면에서는 DS가 품질과 속도의 기본값으로 작동하지만, 최신 개발 상태와 플랫폼별 구현 여부가 불투명하면 팀 단위 신뢰가 흔들린다.**

- 근거: 이수연은 iOS에서 V2가 무시된 사례를 말했고, 문한나는 기존 화면 복사 후 코드 파일 재확인 패턴을 보였다.
- 파트리드 보강: 류선영은 운영/기존 도메인에서는 DS-first 기준이 명확했고, 고유란도 운영에서는 최대한 지킨다고 했다.
- 확인 포인트: 팀리드는 V1/V2, iOS/Android/Web 상태를 품질 리스크로 보고 있는가? 팀 차원에서 확인 루틴이 있는가?

### TH3. 도메인 반복 패턴 관리

**현재는 정식 도메인 라이브러리보다, CDS를 깨거나 조합해서 한 도메인 안에서 반복되는 패턴을 묶어 쓰는 초기 상태에 가깝다. 팀리드는 이 반복 패턴이 언제부터 관리 대상이 되는지 판단해야 한다.**

- 근거: 손소의·이유솔·김진아·김단비가 도메인 안에서 반복 패턴을 묶거나 조합해 쓰는 상황을 말했고, 주선영·고유란은 그 패턴을 누가 관리할지 불명확하다고 봤다.
- 핵심 발화: 고유란은 "파트리드는 못 볼 수 있지만 팀리드는 볼 수 있을 것"이라고 말해, 팀리드가 여러 도메인의 반복·중복 가능성을 조망하는 위치임을 시사했다.
- 확인 포인트: 팀리드는 깨서 반복 사용되는 패턴을 그냥 둘지, 도메인 안에서 관리할지, CDS팀과 정렬할지 어떤 기준으로 판단하는가?

### TH4. 일관성 vs 변별성 판단

**팀리드는 DS 일관성과 서비스별 변별성 사이의 최종 기준을 제공해야 하지만, 현재는 개인/파트 단위 판단으로 분산되어 있다.**

- 근거: 한샘은 "다 비슷비슷해 보인다"는 강한 변별성 욕구를 말했고, 김단비는 웹/스팟에서 기존 DS를 충분히 반영하지 못하는 소외감을 말했다.
- 반대 근거: 이유솔은 DS 준수를 오히려 효율과 품질로 받아들였고, 류선영은 창의성을 저해한다고 보지 않았다.
- 확인 포인트: 팀리드는 "다르게 보여야 하는 화면"과 "같아야 하는 화면"을 어떤 기준으로 나누는가?

### TH5. 심리적 안전감과 허락 구조

**실무자의 "혼날까봐", "소심함", "판단력이 없다"는 신호는 DS팀만의 문제가 아니라, 팀/파트 리더십의 허락 구조와도 연결될 수 있다.**

- 근거: 김단비는 "정호님한테 혼날 것 같다"고 말했고, 이유솔은 도메인 반복 패턴을 만들 때 "제 맘대로 만들면 안 될 것 같다"고 했다. 한샘은 판단력이 없어 자꾸 물어본다고 했다.
- 파트리드 보강: 류선영·주선영은 이를 감정보다 예외 판단 언어로 번역했고, 고유란은 리드임에도 "독단적으로 쓰기가 무섭다"고 말했다.
- 확인 포인트: 팀리드는 파트원/파트리드가 예외를 제안할 때 직접 판단 기준을 제공하는가, 아니면 DS팀/상위 의사결정자 확인 절차로 넘기는가?

### TH6. DS 투자 우선순위

**팀리드는 "DS를 더 확장할 것인가"보다 "어디까지 DS로 관리할 것인가"를 결정해야 하는 위치에 있다.**

- 근거: 고유란은 "쓰는 사람이 절반"인 상황에서 개발팀까지 엮어 시스템을 운영하는 것이 맞는지 질문했다.
- 측정 보강: LLM 코드 챗봇 실제 사용자는 1/11, AI 도구 통합은 아직 낮다. 반면 개발 상태 매트릭스, Usage 강화, 도메인 반복 패턴 관리 기준은 다수 신호로 반복됐다.
- 확인 포인트: 팀리드는 DS팀이 지금 더 투자해야 할 축을 컴포넌트 확대, 개발 상태 가시성, 도메인 거버넌스, 교육/온보딩 중 어디로 보는가?

## 질문지

### 1부. 도입 및 팀 구조 확인

1. 현재 담당하시는 팀/파트 구성과 각 파트의 주요 도메인을 간단히 소개해 주세요.
2. 팀 내에서 CDS를 가장 많이 쓰는 도메인과, 상대적으로 덜 쓰는 도메인은 어디라고 보시나요?
3. 최근 팀원/파트리드에게서 DS 관련해서 가장 자주 들은 질문이나 고민은 무엇인가요?

### 2부. 선행 vs 운영 판단

4. 선행 작업에서는 CDS를 어느 시점부터 고려하는 것이 적절하다고 보시나요?
   - 처음부터 적용 / 방향 잡은 뒤 정리 / 개발 합의 전 점검 / 케이스별
5. 선행 시안에서 컴포넌트를 쓰지 않고 자유롭게 그리는 것이 더 낫다고 판단되는 조건은 무엇인가요?
6. 기존 인터뷰에서는 선행에서 CDS를 유지하기 어려운 이유로 `초기 선행 단계의 DS 조립 비용`, `기획/아이디어 싱크 우선`, `컴포넌트 조합·레이아웃 불일치`, `웹/BDS처럼 코드 연결 약함`이 나왔습니다. 팀리드 입장에서 가장 큰 원인은 무엇이라고 보시나요?
   - 그 원인이 가장 크게 드러나는 시점은 언제인가요?
   - 그 상황에서도 최소한 지켜야 하는 CDS 기준이 있다면 무엇인가요?
   - 선행 단계에서 CDS를 유지하지 못해도, 어느 시점에는 다시 맞춰야 한다고 보시나요?
7. 반대로 운영·기존 화면에서는 어느 정도까지 CDS 준수를 요구하시나요?
8. "전달력 있는 화면"과 "개발 효율이 좋은 화면"이 충돌하면 어떤 기준으로 판단하시나요?
9. 팀원에게 "이건 선행이라 자유롭게 가도 된다" 또는 "이건 운영이라 DS를 지켜야 한다"고 말한 최근 사례가 있나요?

### 3부. 검토·승인 구조

10. 파트리드가 DS 예외를 판단할 때, 팀리드로서 어디까지 위임하고 어디서 직접 개입하시나요?
11. 파트리드/파트원이 "이건 detach해도 될까요?"라고 가져오면 어떤 정보를 먼저 보시나요?
    - 기획 의도 / UX 목적 / 개발 구현 여부 / 재사용 가능성 / 일정 / 도메인 특수성
12. 예외를 통과시키는 기준과 되돌리는 기준이 팀 안에서 공유되어 있나요?
13. V1/V2, iOS/Android/Web 개발 상태가 애매할 때 최종 결정은 누가 하나요?
14. 파트원이나 파트리드가 DS팀에 직접 질문하기 전, 팀리드가 먼저 방향을 잡아주는 경우가 있나요?

### 4부. 도메인 반복 패턴 관리

15. 현재는 정식 도메인 라이브러리라기보다, 각 도메인 안에서 CDS를 깨거나 조합한 반복 패턴을 쓰기 쉽게 묶어두는 정도로 보입니다. 이런 패턴이 생기기 시작할 때, 팀리드 입장에서는 어느 시점부터 관리 기준이 필요하다고 보시나요?
16. 팀 안에서 CDS를 깨서 쓰는 패턴이 반복되고 있는지, 혹은 따로 관리되고 있는지 파악하고 계신 영역이 있나요?
17. 만약 아직 관리되고 있지 않다면, 팀 차원에서 관리할 의향이나 필요성을 느끼시나요?
18. 관리가 필요하다면 어떤 방식이 현실적이라고 보시나요?
    - 담당자 파일 안에서만 관리 / 파트 내 공용 컴포넌트화 / 팀 차원 목록화 / DS팀과 정기 정렬 / CDS 승격 후보로 기록
19. 어떤 조건이 되면 "이건 도메인 안에서만 관리하면 된다"가 아니라 "CDS팀과 정렬하거나 CDS 승격을 검토해야 한다"고 보시나요?

### 5부. 심리적 안전감·코칭

20. 팀원들이 "이렇게 만들어도 될까요?", "DS팀에 혼나지 않을까요?" 같은 방식으로 불안을 표현한 적이 있나요?
21. 그런 불안을 줄이기 위해 팀리드가 해줄 수 있는 말이나 프로세스는 무엇이라고 보시나요?
22. 팀 안에서 "DS를 잘 쓰는 능력"은 무엇이라고 정의하시나요?
    - 최대한 준수 / 필요한 예외 판단 / 빠른 개발 협업 / 기획 설득 / 도메인 완성도
23. "DS를 깨는 것도 능력"이라는 인식이 팀 안에 있다고 느끼시나요? 있다면 어떤 방향으로 코칭하시나요?

### 6부. 협업·조직 우선순위

24. 기획자가 DS 범위 밖 변경을 요구할 때, 팀리드가 직접 개입하는 경우는 어떤 상황인가요?
25. 개발 쪽에서 CDS와 다르게 구현된 경우가 있다면, 나중에 어떤 문제가 제일 크게 돌아오나요? QA에서 다시 고치는 문제인지, 플랫폼별로 달라지는 문제인지, 업데이트가 안 따라가는 문제인지 궁금합니다.
    - 그런 일이 생겼을 때 팀리드가 직접 개입하는 기준이 있나요?
26. 팀 안에서 해결하기 어렵고 DS팀과 함께 맞춰야 한다고 느끼는 이슈가 있다면 어떤 것인가요?
    - 그런 이슈가 생겼을 때 팀리드가 먼저 정리하거나 기준을 잡아주는 편인가요?
27. 지금 DS팀이 가장 먼저 투자해야 한다고 보는 영역은 무엇인가요?
    - 컴포넌트 추가 / Usage·사례집 / 개발 상태 매트릭스 / 도메인 반복 패턴 관리 기준 / 온보딩 / LLM 코드 챗봇 확산
28. 팀 차원에서 CDS 도입 전후로 품질, 속도, 논의 방식이 달라졌다고 느끼시나요?

### 마무리

29. 팀리드 입장에서 "CDS는 여기까지 책임져야 한다"는 경계가 있다면 어디인가요?
30. 반대로 "이건 팀/도메인이 책임져야 한다"고 보는 영역은 무엇인가요?
31. 최종 리포트에 팀리드 판단 기준으로 꼭 남겨야 할 문장 하나를 꼽는다면 무엇인가요?

## 인터뷰어 운영 가이드

### 시작 전 멘트

> "오늘은 팀리드님이 DS를 얼마나 잘 알고 계신지 확인하는 자리가 아닙니다. 앞선 11명 인터뷰에서 선행/운영, 도메인 안에서 반복되는 변형 패턴, 예외 판단, 개발 상태 가시성 같은 이슈가 반복적으로 나왔고, 이걸 팀 차원에서는 어떤 기준으로 결정해야 하는지 여쭤보려 합니다. 정답보다 실제 판단 기준과 우선순위를 듣고 싶습니다."

### 핵심 원칙

| 원칙 | 운영 메모 |
|---|---|
| 개인 평가가 아니라 조직 판단 | "누가 잘못했다"보다 팀리드가 보는 기준·경계·우선순위를 듣는다. |
| 선행/운영을 계속 분리 | 답변이 모호하면 "그건 선행 작업 기준인가요, 운영 화면 기준인가요?"라고 되묻는다. |
| 위임과 개입을 구분 | 파트리드에게 맡기는 영역과 팀리드가 직접 보는 영역을 나눈다. |
| 도메인 반복 패턴은 관리 의향부터 묻기 | 라이브러리화 전제를 두지 말고 반복 패턴이 관리되고 있는지, 관리할 의향이 있는지, 어떤 방식이 현실적인지 확인한다. |
| 민감한 심리 신호는 우회 | "혼나나요?"보다 "예외 제안이 부담스러워지는 장면이 있나요?"로 묻는다. |
| 팀별 사례를 붙여 묻기 | 안정호님/김선영님 각각 팀원 발화를 짧게 요약해 확인한다. |

### 기록 체크포인트

- ☐ 선행 작업에서 DS를 고려하는 시점
- ☐ 운영/기존 화면에서 DS 준수 요구 수준
- ☐ 창의성/전달력 vs 개발 효율 충돌 시 판단 기준
- ☐ 파트리드에게 위임하는 영역과 팀리드가 직접 개입하는 영역
- ☐ detach/예외 통과 기준
- ☐ V1/V2·플랫폼 개발 상태 의사결정 루틴
- ☐ 도메인 반복 패턴의 관리 여부·관리 의향·CDS 정렬 기준
- ☐ 팀원이 느끼는 허락 구조/심리적 안전감에 대한 인식
- ☐ 기획자/개발자 이슈에 팀리드가 개입하는 조건
- ☐ DS팀 투자 우선순위

## 1. 문서 목적

조사는 Chord Design System(이하 CDS)를 누가 잘 지키고 누가 지키지 않았는지 평가하기 위한 것이 아니다.

목적은 Product Designer(이하 PD)들이 실제 업무 안에서 CDS를 어디까지 기준으로 삼고, 어느 순간부터 예외로 판단하는지 확인하는 데 있다. 인터뷰 발화에서는 개인 태도보다 업무 맥락, 개발 연결 상태, 도메인 특성, 예외 판단 기준의 차이가 반복해서 드러난다.

---

## 2. 글감 요약

CDS는 운영 화면에서 이미 기본값에 가깝게 작동하고 있다. 특히 CDS 구조와 연결된 운영/개선 업무에서는 속도, 일관성, 개발 커뮤니케이션 효용이 반복적으로 확인된다.

다만 CDS가 모든 화면을 같은 방식으로 흡수하는 단일 운영 체계로 작동하는 것은 아니다. 선행이 구축으로 넘어가는 시점, 도메인 특화 요구가 커지는 시점, 개발 구현 상태가 보이지 않는 시점, 기존 화면이 레거시화되는 시점에서는 CDS 기준이 다시 해석된다.

이번 조사에서 핵심 쟁점은 `CDS 준수 여부`가 아니라, CDS가 업무 흐름의 어느 전환점에서 기준으로 작동하고, 어떤 조건에서 예외로 판단되며, 그 예외가 어떻게 기록·재흡수되는가이다.

따라서 후속 논의는 교육이나 준수 독려보다 `예외 판단 기준`, `개발 구현 상태 가시화`, `도메인 패턴의 책임 범위`, `공통 CDS로의 재흡수 루프`를 정리하는 방향으로 이어지는 것이 적절하다.

숫자는 다음 기준으로 읽는다.

- `11명 기준`은 실무자 8명 + 파트리드 3명의 사용 방식 보조 분석에만 사용한다.
- `13명 기준`은 실무자, 파트리드, 팀리드까지의 사용자 인터뷰 정량 분석에 사용한다.
- `14명 전체`는 전체 조사 범위를 말할 때만 사용한다.
- HoD 문서는 사용자 인터뷰 정량 수치에 합산하지 않는다.
- 반복 신호는 `언급 인원`이다. 실제 사용률, 준수율, 만족도, 로그 데이터가 아니다.

### 4.2 CDS 사용 진입 방식

> 운영 화면에서는 CDS를 처음부터 조립하는 방식뿐 아니라, 기존 작업물을 출발점으로 삼고 이후 CDS 기준을 재확인하는 방식도 함께 나타난다.

| **사용 진입 방식** | **인원/비율** | **해석** |
| -------- | -------- | -------- |
| 강한 CDS-first | 3명, 약 27% | 빈 화면 또는 초기 구성 단계부터 CDS 컴포넌트를 먼저 가져와 조립한다. |
| 기존 화면/작업 파일 출발 후 CDS 재확인 | 2명, 약 18% | 운영 화면에서 기존 파일이나 화면을 참고하되, 최신 CDS나 코드 기준을 다시 확인한다. |
| 자유 시안 후 CDS 맞춤/디테치 | 4명, 약 36% | 선행/신규 화면에서 먼저 자유롭게 만들고, 이후 CDS와 맞추거나 일부 분리한다. |
| 별도 라이브러리 또는 도메인 패턴 출발 | 2명, 약 18% | CDS보다 도메인에서 반복되는 패턴을 먼저 출발점으로 삼는다. |

## 5. 인터뷰이의 발화 내 발견

### 5.1 CDS는 운영 업무에서 기본값처럼 작동한다

~~~panel type=info
- CDS는 운영 업무에서 효용을 만든다는 발화가 반복된다.
- 운영 효용은 이미 CDS와 연결된 화면에서 더 강하게 확인된다.
- 레거시나 개발 구조가 다른 화면에서는 별도 맵핑 비용이 언급된다.
~~~

운영/기존 화면에서는 CDS가 기본값처럼 작동한다. 기존 컴포넌트를 조합하거나 치환하는 작업에서는 속도와 일관성의 효용이 확인됐다. 토큰, 다크모드, 기본 상태값처럼 반복 품질을 매번 새로 판단하지 않아도 되는 점도 강점으로 언급됐다.

하지만 운영이라고 모두 같은 상태는 아니다. 이미 CDS 구조와 맞는 운영은 안정적이지만, 오래된 화면, V1/V2 혼재, 웹/외주/하이브리드 구조, 선행 결과물이 흡수되지 않은 화면에서는 운영 업무 안에서도 다시 예외 판단이 발생한다.

### 5.2 선행은 CDS 미적용 여부보다 복귀 시점이 쟁점으로 언급됐다

~~~panel type=info
- 선행은 무조건 CDS 밖이라는 뜻으로 발화되지 않는다.
- 초반 탐색과 확정/구축 전 확인 시점이 구분되어 언급된다.
- 핵심은 처음부터 썼는가보다 언제 돌아오는가에 가깝다.
~~~

선행/신규 화면에서는 초반부터 CDS를 강제하면 탐색 폭이 좁아질 수 있다는 맥락이 드러났다. HoD 답변에서도 초기 아이데이션 단계에서는 CDS에 갇히지 않는 쪽의 맥락이 제시됐다.

다만 방향성과 핵심 플로우가 정해지는 시점에 CDS를 다시 확인하는 흐름이 쟁점으로 언급됐다. 선행 결과가 확정되어 구축으로 넘어갈 때 CDS에 흡수되지 않으면, 운영 진입과 동시에 예외 또는 레거시 후보가 된다는 맥락이 있다.

### 5.3 예외는 취향보다 업무 조건과 연결되어 발생한다

> 예외 자체보다 예외가 왜 발생했는지, 일회성인지 반복 패턴인지, 도메인에 남는지 CDS로 재흡수될 수 있는지가 반복 쟁점으로 나타난다.

| **예외 사유** | **의미** |
| -------- | -------- |
| 기능 한계 | CDS 컴포넌트로 화면 목적이나 기능을 담기 어려움 |
| 정의 불일치 | 컴포넌트 이름은 같지만 실제 쓰임이 다름 |
| 개발 미구현 | Figma에는 있으나 코드 구현 상태가 다르거나 불명확함 |
| 도메인 톤 | 커머스, 음악, 공연, 콘텐츠 등 도메인 맥락이 강함 |
| 선행 검증 | 아직 반복성이나 운영 가능성이 확정되지 않음 |
| 레거시 구조 | 기존 화면과 과거 구현 히스토리 때문에 즉시 치환이 어려움 |
<!-- adf:table attrs='{"width":760}' -->

### 5.4 예외 기록은 HoD 문서에서 리더십 맥락으로 제시됐다

~~~panel type=info
- 예외는 금지 대상이라기보다 판단 맥락으로 발화된다.
- 예외 기록의 형식은 사용자 인터뷰 정량 수치가 아니라 HoD 문서의 리더십 맥락이다.
- 사용자 인터뷰에서는 예외 사유와 영향 범위를 확인하기 어렵다는 발화가 함께 나타난다.
~~~

HoD 문서에는 예외를 인정할 때 남길 수 있는 정보로 다음 세 가지가 제시됐다.

| **왜?** | 기존 CDS로 대응할 수 없었던 기획적/비즈니스적 배경 |
| -------- | -------- |
| **무엇을?** | 어떤 컴포넌트를 어떻게 변형하거나 커스텀했는지 |
| **언제까지?** | 이번 릴리즈 한정인지, 정식 패턴으로 유지할 것인지 |
<!-- adf:table attrs='{"width":760}' -->

이 내용은 사용자 인터뷰 정량에서 나온 수치가 아니라, HoD 문서에 정리된 리더십 맥락이다.  
사용자 인터뷰에서는 예외 판단의 불안, 디테치 확인의 어려움, 도메인 패턴의 산발성이 함께 확인됐다.

### 5.5 도메인 반복 패턴은 이미 일부 생기고 있다

~~~panel type=info
HoD 답변 기준으로 보면, 한 도메인 안에서만 쓰이는 패턴과 여러 도메인으로 전이될 수 있는 패턴을 구분하는 맥락이 제시된다.
~~~

인터뷰와 HoD 문서에서 함께 확인되는 질문은 다음과 같다.

| **Q1.** | 특정 도메인 안에서만 반복되는가? |
| -------- | -------- |
| **Q2.** | 여러 도메인으로 전이될 가능성이 있는가? |
| **Q3.** | 공통 접근성, 다국어, 다크모드, 기본 인터랙션과 연결되는가? |
| **Q4.** | 코드 구조상 공통화가 가능한가? |
| **Q5.** | 도메인 담당자가 관리하는 편이 더 현실적인가? |

### 5.7 개발 구현 상태가 보이지 않으면 CDS 판단이 흔들린다

~~~panel type=info
- CDS는 Figma 컴포넌트만으로 작동한다고 보기 어렵다.
- 실제 개발 구현 상태가 보이는지 여부가 디자이너 판단과 연결된다.
- 개발 연결 상태는 CDS 신뢰와 함께 반복적으로 언급된다.
~~~

디자이너가 Figma에서 CDS 컴포넌트를 확인하더라도, 실제 iOS/Android/Web에 구현되어 있는지 알기 어려우면 다시 개발자 확인이나 기존 화면 복사로 돌아가게 된다는 발화에서 드러났다.

HoD 답변에서도 플랫폼별 Production Ready 상태나 코드 컴포넌트 싱크 상태가 보이면 판단에 도움이 된다는 맥락이 제시됐다.

### 5.8 교육보다 판단 기준과 사례가 언급되는 맥락이 드러났다

HoD 답변에서는 교육과 워크샵을 더 진행하는 방향보다 판단 기준, 사례, 운영 프로세스 정립을 언급하는 맥락이 드러났다.

사용자 인터뷰에서도 단순히 “모른다”보다 “어디까지 바꿔도 되는지”, “이 케이스가 CDS에 맞는지”, “개발에서 실제로 구현되어 있는지”, “다른 사례가 있는지”를 확인하려는 발화가 되풀이된다.

### 7.2 보조 설문에서 확인되는 신호

- 공통 컴포넌트/규칙의 작업 효율, 개발 협업 효율, 일관된 디자인 역량 중요성은 높은 동의가 확인됐다.
- 시안 시작 시 공통 컴포넌트를 먼저 확인하는지에 대해서는 4~5점과 3점이 나뉜다.
- 예외 상황에서 변형 수준을 스스로 판단할 수 있는지에 대해서는 평균이 상대적으로 낮고, 1~2점 응답도 확인됐다.
- 창의적 판단이 규칙보다 우선한다는 문항은 3점 응답이 많고 높은 동의는 제한적이다.
- 별첨 문항에서는 인터뷰 분위기와 솔직한 발화 가능성에 대해 높은 동의가 확인됐다.

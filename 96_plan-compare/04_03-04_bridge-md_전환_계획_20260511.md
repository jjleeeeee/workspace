# 03/04 Bridge Description: 단일 `*.bridge.md` 전환 계획

## Context

`03_wf-figma-to-description/draft-descriptions`에는 Figma Description용
`*.description.yaml` 원본 draft가 있고, `04_wf-figma-to-react-components/src/figma`
에는 React 구현 과정에서 만든 `*.source.md` 근거 문서와 visual registry가 있다.

처음에는 `bridge-descriptions/*.bridge.yaml`을 별도 중간 계약서로 만들었지만,
외부 `source_note.path` 링크에 의존하면 링크 파일이 없을 때 bridge 단독 해석력이
떨어진다. Google `DESIGN.md` 방식처럼 한 파일 안에 YAML front matter와 Markdown
body를 같이 두면, 기계 검증 데이터와 사람이 읽는 판단 근거를 같은 문맥에 보관할 수
있다.

## 핵심 결정 요약

| 항목 | 결정 |
|---|---|
| 표준 산출물 | `bridge-descriptions/<component>.bridge.md` |
| 기계 판독 레이어 | Markdown YAML front matter |
| 사람 판독 레이어 | Markdown body |
| 기존 `*.bridge.yaml` | 같은 basename의 `*.bridge.md`로 변환 |
| Figma draft | `draft-descriptions/*.description.yaml` 유지, Figma Description 원본 draft로만 사용 |
| React source note | bridge body에 digest를 남기고, 원본 `*.source.md` path는 검증 가능한 근거 링크로 유지 |
| missing source note | `required: true`인 path가 없으면 validator fail |
| missing visual registry | `registry_path`가 없거나 entry가 다르면 validator fail |

## 파일 구조

## 추출 가능성

### Figma 링크만으로 가능한 것

- component name, file key, node id, component set key
- axes, variants, props
- 대표 node와 live read evidence
- Figma에서 확인 가능한 layout, tokens, assets, text behavior
- `draft-descriptions/<component>.description.yaml`
- `bridge-descriptions/<component>.bridge.md`의 `component_contract`
- Markdown body의 `Figma Evidence`

### Figma 링크만으로 불가능하거나 불완전한 것

- React prop naming, Storybook story id, CSS selector
- 04 visual registry id와 baseline path
- CSS/asset export path
- 구현 과정에서 발견한 known gaps
- 기존 `*.source.md`에만 남아 있는 React 구현 판단

따라서 전체 bridge를 자동 생성하려면 입력 소스는 두 종류가 필요하다.

1. Figma link/live read
2. local repo의 `draft-descriptions/*.description.yaml`과
   `04_wf-figma-to-react-components/src/figma/*.source.md`

## 구현 단계

### Phase 1 — Validator front matter 지원

`tools/validate-component-description.mjs`의 bridge mode가 `*.bridge.md`를 읽으면
맨 앞 YAML front matter만 파싱해 기존 bridge schema 검증에 사용한다.

- front matter가 없으면 fail
- front matter가 닫히지 않으면 fail
- 기존 `description` mode는 변경하지 않는다
- `*.bridge.yaml` 지원은 마이그레이션 기간에만 허용하거나, 계획 실행 시점에 제거 여부를 결정한다

### Phase 2 — required path fail-hard

React binding에서 다음 path는 `required: true`일 때 warning이 아니라 fail이다.

- `platform_bindings.react.source_note.path`
- `platform_bindings.react.visual_validation.registry_path`

visual registry가 존재하면 `registry_id` entry를 읽어 다음 필드 일치를 검증한다.

- `component`
- `story_id`
- `selector`
- `comparison_scope`
- `is_parity_gate`
- `expected_size`
- baseline path

### Phase 3 — 기존 6개 bridge 변환

현재 생성된 bridge 파일 6개를 같은 basename의 `*.bridge.md`로 변환한다.

- `text-button`
- `text-fields`
- `list-item-native`
- `avatar`
- `checkbox`
- `radio`

YAML 내용은 front matter로 옮기고, Markdown body에는 다음 섹션을 공통으로 둔다.

- `Figma Evidence`
- `Source Note Digest`
- `Visual Validation`
- `Implementation Notes`
- `Known Gaps`

### Phase 4 — 문서 갱신

03 워크플로우 문서에서 표준 산출물을 `*.bridge.md`로 바꾼다.

- `README.md`
- `docs/AI_RULES.md`
- `workflow/description-yaml-schema.md`
- `history/2026-05-11-source-note-bridge-yaml.md`

## Assumptions

- `*.bridge.md`가 bridge 표준 산출물이다.
- YAML front matter가 machine-readable SoT이고 Markdown body는 evidence/rationale이다.
- `draft-descriptions/*.description.yaml`은 Figma Description용 원본 draft로 유지한다.
- `bridge-descriptions`는 Figma write 대상이 아니라 구현 검증/생성 bridge로만 사용한다.
- 04는 산출물 위치가 아니라 React 검증 근거로만 사용한다.

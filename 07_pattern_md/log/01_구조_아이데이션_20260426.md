## 배경

`99_archive/03_component_md`에서 컴포넌트 단위 MD를 만든 연장선으로, `04_pattern_md`에서 피그마 화면(페이지) 단위 MD 구조를 탐색함.

목적: AI 에이전트가 화면을 읽고 구현/검토할 수 있는 spec 문서 포맷 정의.

---

## 실험한 두 가지 방식

### v1 — 전체 페이지 + 인라인
파일: `screens/experiments/checkout_v1_fullpage_inline.md`

- 한 파일에 화면 전체를 기술. 컴포넌트 props도 인라인으로 직접 명시.
- 장점: 한 파일로 완결. AI가 컨텍스트 스위칭 없이 전체 맥락을 읽을 수 있음.
- 단점: 파일이 길어짐. 비슷한 구조의 섹션(주문자, 배송주소)도 반복 작성.

### v2 — 섹션 단위 + 참조
파일: `screens/experiments/checkout_v2_section_ref.md`

- 인덱스 파일 + 섹션별 파일 분리. 컴포넌트는 `99_archive/03_component_md` 참조.
- 장점: 섹션 재사용 가능. 변경 범위 명확 (섹션 파일 하나만 수정).
- 단점: AI가 전체 화면을 이해하려면 파일 여러 개를 열어야 함 → 컨텍스트 단절 위험.

## 핵심 발견: 피그마 데이터 읽기 비용

### 문제
Framelink로 전체 화면 노드를 읽으면 ~128k chars → 토큰 소비 과다.
운영 단계에서 매번 전체를 읽는 건 비현실적.

### 해결 방향
**figma-console JS 실행**이 훨씬 효율적.

| 방식 | 데이터량 | 특징 |
| --- | --- | --- |
| Framelink 전체 읽기 | ~128k chars | fills, styles, layout 전부 포함 |
| figma-console JS (레이어 트리) | ~3k chars | 요청한 것만 반환, 형태 자유롭게 가공 가능 |
| Framelink 특정 nodeId | 섹션 1/N 수준 | 구조는 고정, 특정 섹션만 가능 |

### 운영 워크플로우 (가설)
```
피그마 변경 발생
    │
    ▼
figma-console로 레이어 트리 읽기 (저비용)
    │
    ├─ 구조 변화 없음 → 해당 노드 ID만 Framelink로 핀포인트 읽기
    └─ 구조 변화 있음 → 해당 섹션 노드만 JS로 상세 읽기
```

MD가 진실의 소스. 피그마는 변경 감지 + 필요한 부분만 재추출 용도.

## 방향성 결론 (잠정)

- **v2(섹션 분리) 방식**이 유지보수에 더 유리 — 섹션 하나가 바뀌면 그 파일만 업데이트.
- 단, 섹션 파일이 너무 잘게 쪼개지면 AI 컨텍스트 부담 발생 → 인덱스 파일에서 인라인 요약도 병기하는 하이브리드 방식 탐색 필요.
- **공통 영역(GNB, Tab Bar, Bottom CTA)은 참조**, **화면 고유 본문은 인라인** 절충안을 v3로 실험해볼 것.

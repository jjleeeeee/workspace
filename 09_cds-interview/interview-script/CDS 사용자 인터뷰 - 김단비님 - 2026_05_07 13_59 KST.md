### 인터뷰 개요

- 디자인 시스템(DS) 사용 평가가 아닌 실무 활용 방식과 어려움 파악을 위한 인터뷰
- 인터뷰이: @(WEV_프로덕트디자인팀) 이창규/Lee changkyu
- 2팀 소속이지만 1팀 경험 보유, CDS 머치 제작 등 다양한 DS 활용 경험 보유

### 디자인 시스템 활용 경험

- **프로젝트에서 활용한 컴포넌트**: 리스트, 패치, 박스, 라디오, 아이콘, 아바타, 아바타 그룹 등 전반적으로 활용
- **댓글 모듈 제안**: 랩 3.0 작업 시 댓글 모듈이 없어 공통 컴포넌트로 제안, 이후 DS에 패턴으로 추가됨
- **최근 프로젝트**: 계정보호 작업을 DS 활용하여 하루 만에 완료 - 텍스트, 버튼, 다크모드, AOS/iOS 시스템 모두 기존 컴포넌트로 해결

### 디자인 워크플로우

**기획서 분석 및 리서치 단계**  

- 기획서 파악 후 목적과 해결해야 할 문제 정리
- UI Vault, Mobbin 등에서 레퍼런스 리서치
- 직접 앱 캡처 및 유사 서비스 체험
- 해당 화면의 디자인 요소뿐 아니라 비슷한 기능을 가진 다양한 서비스의 교집합 파악
- 예시: 아티스트 알림 설정의 경우, 구독 서비스들의 설정 관리 방식 리서치

**시안 작업 프로세스**  

- 레퍼런스의 공통분모를 정리하여 위버스에 적용 가능한 요소 파악
- 1차 시안 작업 후 자체 검토
- 내부 논의 → 디토화(iteration) → 재논의
- 기획 전달 → 피드백 반영 → 최종 내부 논의
- 가이드 작업 후 개발 전달

### DS 활용 방식의 차이

**기존 화면 개선 프로젝트** 

- 무조건 시스템을 활용해서 시안 작업
- 컴포넌트를 찾아서 적용하는 방식

**신규 화면/선행 디자인**  

- 먼저 자유롭게 디자인 후 시스템 컴포넌트로 대체
- 이유: 와이어프레임 단계부터 시스템에 맞추려면 시간이 오래 걸림
- 기획 스펙에 맞는 컴포넌트를 찾거나 우회 표현 방식을 찾는 데 시간 소요
- 먼저 디자인하고 시스템으로 "깎아내는" 방식이 더 효율적

### 스팟(Spot) 프로젝트 사례

**초기 접근**  

- 리스트 아이템에 16:9 썸네일, 3-4줄 텍스트 등 기존 스펙을 넘어서는 요구사항
- 먼저 자유롭게 시안 작업 후 DS 적용 가능 여부 검토
- 일부는 DS로 소화 가능하도록 조정 (예: 4줄 정보를 2줄 + 별도 행으로 분리)

**DS 적용 및 조정**   

- 3줄 케이스가 필요했으나 리스트 아이템에 3줄 옵션 없음
- 여백 조정으로 해결 시도하다가 시스템이 깨질 것 같아 다른 방식 모색
- 커뮤니티 정보를 위로 빼서 리스트 아이템 타이틀에 배치
- 하단은 리스트 아이템 2개 조합으로 구성
- 디바이더로 구분하여 시스템을 깨지 않고 조합해서 사용

**접근성 및 새 컴포넌트 고려** 

- 시스템을 많이 깨야 하는 경우 접근성도 새로 챙겨야 할 수 있음
- 담당자 교체로 해당 프로젝트는 아이디어 제공까지만 참여
- 비주얼 요소는 최대한 DS 활용, 정보가 많아지는 부분은 조정 필요

### 컴포넌트 사용 방식

**새 컴포넌트 제작 시**  

- CDS가 아닌 작업 중인 파일에서 컴포넌트의 원본 요소를 복사해서 사용
- 이유: CDS로 이동하는 것이 번거롭고, 메인 컴포넌트 찾기에서 다른 페이지로 이동하는 경우가 많았음
- 빈 화면이나 엉뚱한 곳으로 앵커링되어 원하는 컴포넌트를 찾기 어려운 경험

**최신 버전 반영** 

- 시안 작업 시에는 현재 파일의 컴포넌트 사용
- 가이드 작업 시에는 CDS에서 최신 버전 확인 후 업데이트
- 새로 고침 아이콘이 있으면 CDS에서 최신 컴포넌트 가져와서 데이터 최신화

### 복잡한 컴포넌트 사용

**학습 및 적용**  

- 타이틀 헤더, 패널 등 복잡한 컴포넌트는 어렵지만 쓰려고 노력
- 패널을 보고 읽으며 눌러보고 만지면서 학습
- 하이드 시키는 것이 쉽지만 법칙에 위배되므로 프로퍼티 조정 시도
- 개념이 어렵지만 최대한 읽어서 사용하려고 노력
- 쓰다 보면 익숙해짐

**어려움**  

- 가끔 필드나 옵션 적용이 제대로 안 되는 경우 발생
- 여러 옵션이 한 곳에 몰려 나타나는 경우 혼란스러움
- 직접 선택 때문에 어려움 발생

### 디자인 시스템의 장단점

**장점**   

- 빠른 화면 구성 가능
- 통일된 유저 경험 제공
- 기존 화면 수정 작업의 경우 매우 효율적 (예: 계정보호 작업 하루 만에 완료)
- 다크모드, AOS/iOS 대응 등이 자동으로 지원됨

**단점**   

- 실무자들의 통일된 이해도를 동일한 시간대에 확보하기 어려움
- 누군가는 잘 알고 있고, 누군가는 몰라서 안 쓰거나, 더 좋은 것을 만들 수 있다고 생각해서 시스템을 깰 수 있음
- 선행 디자인에서는 제약으로 느껴질 수 있음
- 스팟 프로젝트에서 시스템을 깨는 것이 용인되는 분위기가 형성되어 혼란스러움
- 시스템이 있음에도 다른 방식이 더 낫다고 생각하는 경우 갈등 발생

**전반적 평가**   

- 현재는 지원으로 느낌 (제약이 아님)
- 단, 이번 대화는 대부분 선행 작업에 집중해서 제약처럼 느껴졌을 수 있음
- 단점은 어쩔 수 없는 부분이며, 디자인 시스템을 계속 알려야 함

### 향후 개선 니즈

**기존 컴포넌트 확장** 

- 신규 컴포넌트보다는 기존 컴포넌트에 기능 추가 니즈가 많을 것으로 예상
- 예시:
    - 리스트 아이템에 3줄 옵션 추가
    - 바이콘 왼쪽 서브텍스트 옵션 추가
    - 썸네일 16:9 비율 옵션 추가
- 디자인 시스템이 특정 시점 기준으로 완성되었기 때문에 이후 프로젝트에서는 맞추기 어려운 경우 발생

### 기타 사항

- **사용 툴**: 피그마만 주로 사용, 과거에는 프로토파이, 히프트 사용
- **인터랙션 고려**: 선행 디자인 시에도 스크롤, 전환 등 인터랙션을 상세히 고려하며 작업
- **개선 방안**: 설명을 읽고, 예시 화면을 보고, 하이드 시키지 말고 프로퍼티로 조정하는 것이 중요

### 인터뷰 목적 및 배경

- DS 사용 평가나 사용법 점검이 아닌, 실제 업무에서 DS가 어디서 도움이 되고 어디서 막히는지 파악하기 위한 인터뷰
- 상황과 맥락에 집중하여 진행

### 참여자 배경

- 인터뷰 대상: @(WEV_프로덕트디자인2팀) 문한나/Hanna moon (최근 2팀에서 1팀으로 이동)
- 주요 담당 프로젝트: 커뮤니티, 백스테이지, 미디어 라이브
- 인터뷰 중점 프로젝트: 어카운트 (Account) - 1.0과 2.0에서 전면 개편이 필요했던 프로젝트

### 디자인 프로세스

- 초기에는 기획서 기반으로 디자인 논의 후 제안하는 방식으로 진행했으나, 과정이 길어지는 문제 발생
- 개선된 프로세스: 큰 틀을 기획과 디자인이 함께 논의 → 구체적 기획서 작성 → 디자인 제작 → 사용성 피드백 → 개발 공유
- 법적 이슈가 많은 어카운트 특성상 보안팀 검토 과정이 필요했음
- 리소스 부족으로 일부 개선사항은 추후 DS 업데이트로 계획

### 컴포넌트 사용 및 탐색

- 처음에는 기존 화면(아티스트 탭)을 복사하여 작업했으나, 이후 업데이트된 버전이 있음을 발견
- 이후 DS 문서에서 직접 컴포넌트를 가져오는 방식으로 변경
- 새 화면 제작 시에는 DS 확인, 기존 화면이 있고 업데이트 격차가 적으면 복사, 동시 작업 시에는 화면 공유
- 컴포넌트 찾기 자체는 어렵지 않았으나, 배지(badge) 같은 경우 명칭에 대한 인식 차이 발생
- 배지를 칩과 유사한 형태로 생각했으나, DS에서는 닷(dot) 형태로 정의되어 있어 혼란

### 커뮤니케이션

- 개발자와 소통 시: 화면을 컴포넌트로 구성한 후 소통하므로 어려움 없음
- 기획자와 소통 시: "이런 컴포넌트가 있어요?" "이것도 커스텀인가요?" 등의 질문이 많음
- DS 가이드가 개발과의 약속으로서 논리적 근거 역할을 하여 대체로 도움이 됨
- 기획자들이 DS 기준을 대부분 수긍함

### 컴포넌트 커스터마이징

- 개인적으로는 디테치하여 비슷한 모양으로 만들려고 시도
- 시안 컨펌이나 반려 시에는 타사 참고하거나 처음부터 새로 제작
- 디테치 결정 기준: 재전송 같은 기능이 텍스트필드에 추가될 때 화면이 지나치게 길어지는 경우
- 프로토타입으로 직접 테스트하며 "이거 좀 긴데?"라고 느껴지면 커스텀 제안
- 최종 결정: A안/B안을 만들어 장단점 정리 후 상위에 올려 결정

### 접근성 고려

- 규모가 있는 커스터마이징의 경우 기획이나 개발에 먼저 요구사항 확인
- 자잘한 경우는 개발 공유 시 상황 설명과 함께 전달
- 인터랙션, 로딩 등에 대한 질문을 개발로부터 받기도 함

### 리스트 아이템 케이스

- 어카운트, 샵, 멤버십 등에서 위계가 다른 리스트 아이템 사용
- 공통화를 위해 해당 케이스를 DS 담당자에게 전달 제안

### DS 장점과 단점

- 장점: 개발과의 약속으로서 논리적 근거 제공, 커뮤니케이션 도구로 활용
- 단점: 때로는 제약사항으로 다가옴 - 예를 들어 리스트 아이템을 가로 배치만 가능한데 세로가 더 나을 때 혼란
- 시각적으로 돋보여야 하는 화면에서 DS 준수와 UX 개선 사이의 균형 고민

### 개선 제안사항

- **실제 사용 사례집 필요**: 리스트 아이템을 "이렇게 썼어요"라는 실제 사례가 있으면 팀 내 시안 공유 시에도 근거로 활용 가능
- **챗봇 활용**: 편하게 질문하면 답변을 잘 찾아주지만 가끔 오해석 있음
- 챗봇이 더 고도화되어 사례를 알려주면 더 좋을 것

### 개발 관련 이슈

- **버전 불일치 문제**: 기존 화면은 V1, 새 화면은 V3인데 한번에 업데이트할 리소스가 없을 때 어디에 맞춰야 할지 고민
- **개발 완료 상태 불명확**: DS에는 있지만 개발이 완료되지 않은 컴포넌트 존재 (예: 툴팁)
- **플랫폼별 차이**: 텍스트필드가 안드로이드는 개발 완료, iOS는 미완료 등 플랫폼별 개발 상태가 다름
- 컴포넌트의 개발 완료 여부와 DS와의 동기화 상태가 명확히 표시되면 좋겠음

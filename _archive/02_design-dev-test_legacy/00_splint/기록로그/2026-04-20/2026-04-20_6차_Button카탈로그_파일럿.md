핵심 교훈:
- **Framelink는 "누락 없는 사실 기반"** (전체 트리·layout·styles) 이지만 토큰 아이덴티티가 지워진 상태로 내려옴.
- **figma-console analyze는 state machine은 정확** 하지만 이번 컴포넌트에서 **fillToken 자동 해상도 실패(576/576 null)** — `get_component_for_development` + `get_variables` 2단 보강이 필요.
- 따라서 **파이프라인 규칙**: `analyze_component_set → (fillToken=null이면 → get_component_for_development → VariableID 수집 → get_variables batch 해상도)`.

### 6.B Schema v1 초안 ajv 수용성
- `schemas/components.v1.json` 작성 (Draft 2020-12). 토큰 스키마와 동일 패턴: `Source`/`Component`/`Prop`/`Slot`/`State`/`TokenBinding`/`AnatomyNode`/`A11y`/`Diagnostic` `$defs`.
- 초기 FAIL: `component:text-button` 같은 id가 `^component:[a-z0-9]+(\\.[a-z0-9_-]+)*$` 패턴 첫 세그먼트에 하이픈 불허 → **패턴을 `[a-z0-9_-]+`로 완화** (토큰 스키마의 `token:system.color.xxx` 같은 dot-segmented 관례는 컴포넌트엔 과함).

## 실패 및 원인

- **Framelink 응답이 1.48MB** 로 Claude 한 번에 수용 불가 → 자동으로 tool-results 파일로 저장되고 `jq`/`grep`으로 핀포인트 조회. 원인: 576 variant raw tree + globalVars.styles 전체. 해결: YAML 파싱 + axis 추출 + globalVars 섹션만 선별 조회.
- **`figma_analyze_component_set`의 signature.fillToken이 576개 variant 전부 null** — 원인 미확정. 가능한 요인: (a) 컴포넌트 셋 루트에서 diff 계산 시 boundVariables를 노드별로 잘 못 읽음, (b) 이 파일의 fill이 여러 레이어를 교차 참조해 analyze의 heuristic과 불일치, (c) MCP 버전 제약. 우회: `get_component_for_development` + `get_variables` 2단 해상도.

## 해결 방법

- 대용량 응답은 파일 저장 후 `jq '.[0].text' → /tmp/tb.yaml` 변환, axis 추출은 `awk`로 cross-join 파싱.
- fillToken null 이슈는 대표 variant 1건에 대해 `get_component_for_development` 호출 → 노드 boundVariables 수집 → `get_variables` 1회 batch 필터로 44개 토큰 이름을 한번에 해상도.
